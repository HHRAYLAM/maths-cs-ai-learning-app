# 缩放和部署

*向数百万用户服务大模型需要跨多个 GPU 分布推理、在需要之前预测 token、缓存共享上下文，并选择正确的框架。本文件涵盖推理时的并行性、推测解码、前缀缓存、推理框架、成本优化和监控。*

- 单个 H100 GPU 服务 70B 模型可以在交互延迟下处理约 100 个并发用户。服务 1000 万用户需要 10 万 GPU — 成本约每年 30 亿美元云计算。每个百分点的效率改进节省数千万美元。这就是为什么推理优化不是学术性的：它直接决定 AI 产品的经济性。

## 推理时的模型并行

- 当模型太大无法适应单个 GPU 时，必须跨多个 GPU 分割。来自训练（第 6 章）的并行策略在推理时应用，具有不同的权衡。

### 张量并行

- **张量并行**（Megatron 风格，第 6 章）跨 GPU 分割单个权重矩阵。对于线性层 $Y = XW$，权重矩阵 $W$ 跨 $N$ 个 GPU 按列分割。每个 GPU 计算部分结果，然后 all-reduce 聚合：

$$W = [W_1 | W_2 | \cdots | W_N], \quad Y_i = X W_i, \quad Y = \text{concat}(Y_1, \ldots, Y_N)$$

- 在推理时，张量并行是放不下单个 GPU 的模型的默认值。FP16 的 70B 模型需要 140 GB — 用张量并行分割到 2 个 80 GB GPU。

- **延迟影响**：张量并行每层添加一个 all-reduce 通信步骤。在 NVLink（900 GB/s）上，这每层添加约 0.1 ms。在 PCIe（32 GB/s）上，这每层添加约 3 ms。对于 2 个 GPU 上的 70B 模型（80 层）：NVLink 添加总共约 8 ms，PCIe 添加约 240 ms。这就是为什么 NVLink 对多 GPU 推理 enormously 重要。

### 流水线并行

- **流水线并行**将不同层分配给不同 GPU。GPU 1 处理层 0-39，GPU 2 处理层 40-79。Token 按顺序流经管道。

- 在推理时，流水线并行比张量并行有更高延迟（每个 token 必须遍历整个管道）但更低通信开销（仅激活在 GPU 之间传递，无 all-reduce）。当 GPU 通过慢互连连接时（不同节点，无 NVLink）更受青睐。

### 序列并行

- 对于非常长的序列，即使模型能放下，KV-cache 本身可能不适合单个 GPU。**序列并行**跨 GPU 分片 KV-cache：每个 GPU 存储序列缓存键和值的一部分。

- 注意力期间，每个 GPU 在其缓存段上计算部分注意力分数，然后约简组合结果。这在长上下文推理（128K+ token）中使用，其中 KV-cache 超过单 GPU 内存。

## 推测解码

- **推测解码**是最有影响力的 LLM 推理优化之一。洞察：解码慢是因为一次生成一个 token，每个 token 需要大模型的完整前向传递。但小模型可以更快生成候选 token，大模型可以**并行验证**多个候选。

![推测解码：快速草稿模型生成 5 个候选 token，目标模型在一次传递中验证所有，接受的 token 保留，拒绝的重新采样](../images/speculative_decoding.svg)

- **算法**：
    1. **草稿模型**（小、快 — 例如 1B 参数）自回归生成 $k$ 个候选 token。
    2. **目标模型**（大、准确 — 例如 70B）在整个草稿序列上运行单个前向传递，计算每个候选 token 的概率。
    3. 每个候选被**接受**如果目标模型同意（它对该 token 的概率足够高）。拒绝的候选从目标模型的分布重新采样。
    4. 平均每次验证步骤接受多个 token，给出与接受率成比例的加速。

$$\text{Speedup} \approx \frac{k \times \text{acceptance\_rate}}{\text{cost\_ratio}} \approx 2\text{-}3\times$$

- **为什么有效且无质量损失**：拒绝采样方案保证输出分布与目标模型完全匹配。推测解码是无损的 — 输出在统计上与仅运行目标模型相同，只是更快。

- **变体**：
    - **Medusa**（Cai 等人，2024）：不是单独的草稿模型，在目标模型上添加多个轻量级"头"同时预测多个未来 token。无需单独模型。
    - **EAGLE**（Li 等人，2024）：训练轻量级草稿头，使用目标模型的隐藏状态预测未来 token。比独立草稿模型更高的接受率。
    - **自推测解码**：目标模型本身使用提前退出生成草稿（仅为草稿运行前几层，然后用完整模型验证）。
    - **并行解码**：并行生成多个延续（候选树）并一次性验证整个树。更高吞吐量但对分支 KV-cache 使用更多内存。

## 前缀缓存

- 许多请求共享公共前缀：系统提示、few-shot 示例或常见查询模式。**前缀缓存**存储这些前缀的 KV-cache 并在请求之间复用。

- **系统提示缓存**：如果每个请求以相同的 2000 token 系统提示开始，这 2000 token 的 KV-cache 计算一次并在所有请求之间共享。对于 80 层的 70B 模型，这每请求节省约 200 MB。

- **基数树缓存**（SGLang）：在基数树（trie）中组织缓存前缀。新请求到达时，找到最长缓存前缀匹配并从那里开始生成，跳过匹配前缀的计算。

- **影响**：对于具有长共享前缀的应用（带系统提示的聊天机器人、带常见检索段落的 RAG），前缀缓存将 TTFT 降低 50-90% 并节省成比例的 GPU 计算。

## KV-Cache 驱逐

- 除了量化 KV-cache（文件 01）和使用 GQA/MLA 减少其大小（文件 02），**KV-cache 驱逐**策略选择性移除未来不太可能被关注的缓存 token。

- **H2O**（Heavy-Hitter Oracle，Zhang 等人，2023）观察注意力分数遵循幂律：一小部分 token（"heavy hitters"）接收大多数注意力，而大多数接收可忽略的注意力。H2O 保持：

    1. **最近 token**（最后 $w$ 个 token 的滑动窗口，像 StreamingLLM）。
    2. **Heavy-hitter token**（跨所有过去解码步骤按累积注意力分数的 top-$k$ token）。

- 既不是最近也不是 heavy-hitter 的 token 被驱逐。这维护固定大小 KV-cache，同时保持实际影响生成的 token。H2O 实现接近完整 KV-cache 的质量，仅需 20% 内存。

- **Scissorhands**（Liu 等人，2023）采用类似方法但使用更复杂的重要性指标：在**当前**步骤接收高注意力的 token 被保持，而 $T$ 步未被关注的 token 被驱逐。这适应生成期间变化的注意模式。

- **动态驱逐 + StreamingLLM**：结合注意力槽（永久保持前几个 token）与动态驱逐（保持最近 + heavy-hitter token）。这是对非常长生成的最内存高效方法，实现无限长度生成，质量下降有界。

- 所有驱逐方法的关键洞察：LLM 注意力**实践中稀疏** — 即使架构在所有缓存 token 上计算注意力，实际注意力权重集中在小子集。驱逐其余的对输出质量影响最小。

## 推理框架

- LLM 服务生态系统已收敛到几个主导框架：

| 框架 | 优势 | 最适合 |
|-----------|-----------|----------|
| **vLLM** | PagedAttention、连续批处理、高吞吐量 | 通用 LLM 服务、最高吞吐量 |
| **TensorRT-LLM** | NVIDIA 优化内核、FP8、飞行中批处理 | NVIDIA GPU 上最大性能 |
| **SGLang** | 前缀缓存（RadixAttention）、快速结构化生成 | 具有共享前缀的应用、约束输出 |
| **llama.cpp** | CPU/Metal/CUDA/Vulkan、GGUF 量化、便携 | 消费级硬件、设备上推理 |
| **TGI**（HuggingFace） | 简单 API、易于部署、模型枢纽集成 | 快速部署、HuggingFace 生态系统 |
| **Ollama** | 一键模型下载和服务 | 个人使用、本地开发 |
| **ExLlamaV2** | 极端量化优化（EXL2 格式） | 内存受限 GPU 推理 |

- **vLLM** 是生产 LLM 服务的默认选择。它支持连续批处理、PagedAttention、张量并行、推测解码、LoRA 服务，以及大多数开源模型。

- **TensorRT-LLM** 在 NVIDIA 硬件上实现最高原始性能（相同 GPU 上比 vLLM 快 10-30%）但较不灵活且更难定制。

- **SGLang** 在应用具有结构化输出（JSON、特定格式的代码）或共享前缀时表现出色，归功于其基数注意力缓存和约束解码引擎。

## 成本优化

- 在大规模下，推理成本主导 ML 预算。降低成本的策略：

- **正确大小 GPU 选择**：不是每个模型都需要 H100。量化的 7B 模型在 A10G（约$1/小时）上运行良好，而非 H100（约$8/小时）。将 GPU 匹配到工作负载。

- **Spot 实例**：云提供商以 60-90% 折扣提供未使用 GPU 容量（AWS Spot、GCP Preemptible）。Spot 实例可能被中断，所以它们适用于批处理推理而非延迟关键服务。与抢占处理结合（保存状态、在新实例上恢复），spot 实例也可以服务交互流量。

- **自动缩放**：根据流量缩放 GPU 数量。高峰时间扩展，夜间缩小。Kubernetes HPA（水平 Pod 自动缩放器）或云原生自动缩放（AWS SageMaker、GCP Vertex AI）处理这个。

- **批处理 + 利用率**：30% 和 90% GPU 利用率之间的差异是每 token 成本的 3 倍。连续批处理、智能调度和 PagedAttention 都增加利用率。

- **量化**：INT4 对比 FP16 是 4 倍更少内存 → 适应更小 GPU → 2-4 倍更低成本。另外，更多请求适应同一批次 → 更高吞吐量 → 更低每 token 成本。

- **每 token 成本基准**（近似，2026）：

| 设置 | 每 1M token 成本 |
|-------|-------------------|
| GPT-4o API | $2.50 |
| Claude 3.5 Sonnet API | $3.00 |
| Llama-70B on H100 (vLLM, FP16) | $0.50 |
| Llama-70B on H100 (TRT-LLM, INT8) | $0.25 |
| Llama-8B on A10G (vLLM, INT4) | $0.05 |
| Llama-3B on-device (llama.cpp) | $0（硬件摊销） |

## 监控

- 生产推理需要持续监控以在用户受影响前捕捉退化：

- **延迟监控**：在 p50、p95 和 p99 跟踪 TTFT 和 TPOT。为 p99 超过 SLO 设置警报。p99 飙升通常表明：KV-cache 内存压力（颠簸）、长运行请求独占批次，或 GPU 热节流。

- **吞吐量监控**：跟踪每 GPU 每秒 token。下降表明：批次效率降低（许多短请求 → 低批次利用率）、序列长度增加（每请求更多 KV-cache 内存），或硬件问题（GPU 在 ECC 错误纠正模式，运行更慢）。

- **GPU 利用率**：跟踪 SM 占用率、内存利用率和内存带宽。低 SM 占用率 + 高内存利用率 = 内存绑定（需要更多带宽或量化）。高 SM 占用率 + 低内存利用率 = 计算绑定（需要更多 FLOPS 或更小模型）。

- **模型质量监控**：跟踪每请求指标（响应长度、保留集上的困惑度、用户反馈信号）。模型质量可能因以下退化：数据漂移（传入请求的分布变化）、KV-cache 量化误差在长对话中累积，或服务管道中的 bug。

- **成本监控**：跟踪每 token 每模型每 GPU 类型的成本。如果成本增加而无吞吐量增加，调查效率回归（新模型版本具有更高内存使用、次优批次配置，或未充分利用 GPU）。

- **工具**：Prometheus + Grafana（第 15 章）用于基础设施指标，vLLM/TRT-LLM 的内置指标端点，以及模型级指标的自定义日志。

## 编码任务（使用 CoLab 或 notebook）

1. 模拟推测解码。使用快速"草稿"函数和慢速"目标"函数，并测量一次生成和验证多个 token 的加速。
```python
import random
import time

def target_model(tokens):
    """Slow but accurate model. Returns probability of each candidate token."""
    time.sleep(0.01)  # simulate 10ms per forward pass
    # For simulation: accept tokens that are even numbers
    return [0.9 if t % 2 == 0 else 0.1 for t in tokens]

def draft_model():
    """Fast but approximate model. Generates one candidate token."""
    time.sleep(0.001)  # simulate 1ms per token
    return random.randint(0, 9)

def standard_decoding(n_tokens):
    """Generate one token at a time with the target model."""
    tokens = []
    for _ in range(n_tokens):
        time.sleep(0.01)  # target model generates 1 token
        tokens.append(random.randint(0, 9))
    return tokens

def speculative_decoding(n_tokens, k=4):
    """Generate k draft tokens, verify with target, accept/reject."""
    tokens = []
    total_target_calls = 0

    while len(tokens) < n_tokens:
        # Draft: generate k candidates quickly
        candidates = [draft_model() for _ in range(k)]

        # Verify: one target model call for all k candidates
        probs = target_model(candidates)
        total_target_calls += 1

        # Accept tokens until one is rejected
        for i, (tok, prob) in enumerate(zip(candidates, probs)):
            if random.random() < prob:
                tokens.append(tok)
                if len(tokens) >= n_tokens:
                    break
            else:
                # Resample from target distribution
                tokens.append(tok + 1)  # simplified resampling
                break

    return tokens, total_target_calls

n = 50

start = time.time()
_ = standard_decoding(n)
standard_time = time.time() - start

start = time.time()
_, target_calls = speculative_decoding(n, k=5)
spec_time = time.time() - start

print(f"Standard:    {standard_time:.2f}s ({n} target calls)")
print(f"Speculative: {spec_time:.2f}s ({target_calls} target calls)")
print(f"Speedup:     {standard_time / spec_time:.1f}x")
```

2. 估计应用于 LLM 服务部署的不同优化策略的成本节省。
```python
def serving_cost_analysis(
    model_name, params_B, precision_bits,
    gpu_name, gpu_mem_gb, gpu_cost_per_hr,
    target_throughput_tps,
):
    """Estimate serving cost for an LLM deployment."""
    model_size_gb = params_B * 1e9 * precision_bits / 8 / 1e9
    gpus_for_model = max(1, int((model_size_gb * 1.2) / gpu_mem_gb + 0.99))  # 1.2x for KV-cache

    # Rough throughput estimate (memory-bandwidth limited)
    tokens_per_gpu = 500 / (params_B * precision_bits / 16)  # normalised to 500 tok/s for 7B FP16
    total_throughput = tokens_per_gpu * gpus_for_model

    replicas = max(1, int(target_throughput_tps / total_throughput + 0.99))
    total_gpus = gpus_for_model * replicas
    cost_per_hr = total_gpus * gpu_cost_per_hr
    cost_per_1M_tokens = cost_per_hr / (total_throughput * replicas * 3600 / 1e6)

    print(f"{model_name} @ {precision_bits}-bit on {gpu_name}:")
    print(f"  Model size: {model_size_gb:.0f} GB → {gpus_for_model} GPU(s)/replica")
    print(f"  Throughput: {total_throughput:.0f} tok/s/replica")
    print(f"  Replicas for {target_throughput_tps} tok/s: {replicas}")
    print(f"  Total GPUs: {total_gpus}")
    print(f"  Cost: ${cost_per_hr:.0f}/hr, ${cost_per_1M_tokens:.2f}/1M tokens")
    print()

print("=== Cost Comparison ===\n")

# Baseline: FP16 on H100
serving_cost_analysis("Llama-70B", 70, 16, "H100", 80, 8.0, 1000)

# Quantised: INT8 on H100
serving_cost_analysis("Llama-70B", 70, 8, "H100", 80, 8.0, 1000)

# Quantised: INT4 on A100
serving_cost_analysis("Llama-70B", 70, 4, "A100", 80, 4.0, 1000)

# Smaller model: 8B on A10G
serving_cost_analysis("Llama-8B", 8, 4, "A10G", 24, 1.0, 1000)
```
