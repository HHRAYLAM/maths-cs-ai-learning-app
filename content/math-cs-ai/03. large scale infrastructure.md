# 大规模基础设施

*构建服务数百万用户的系统需要的不仅仅是单个服务器。本文件涵盖可扩展性模式、分布式系统基础、微服务、数据管道、数据库扩展、搜索和向量系统、可观测性、可靠性工程和 CI/CD*

- 服务每秒 1 个请求的模型可以在笔记本电脑上运行。服务每秒 100,000 个请求，99.9% 可用性需要分布式系统、自动故障转移和精心设计的数据管道。本文件涵盖弥合这一差距的模式。

## 可扩展性

- 垂直扩展（scale up）：获取更大的机器。更多 CPU、更多 RAM、更大的 GPU。简单但有硬性限制（可用的最大机器）和单点故障。

- 水平扩展（scale out）：添加更多机器。每个处理一部分流量。没有单机限制，但需要：负载均衡（文件 01）、数据分区和处理分布式状态。

- 无状态服务默认是水平可扩展的。在负载均衡器后添加更多实例。在启动时加载权重并独立处理请求的模型推理服务器是无状态的——任何实例可以处理任何请求。

- 有状态服务（数据库、KV 缓存、特征存储）更难扩展。状态必须跨机器分区（分片，文件 01）并复制以实现容错。

- 可扩展性方程：对于有 $n$ 台服务器的系统：
    - 理想：吞吐量线性扩展（$n$ 台服务器 → $n\times$ 吞吐量）。
    - 实际：来自协调、负载均衡和数据传输的开销意味着吞吐量亚线性扩展。阿姆达尔定律（第 13 章）适用：串行部分（共享状态、协调）限制加速。

## 分布式系统

- 分布式系统是一组协调提供服务的机器。基本挑战：

- 网络分区：机器并不总是能通信。网线被切断、交换机故障、数据中心断电。系统必须处理部分故障。

- 时钟偏移：机器有不同的时钟。"事件 A 在机器 1 上发生在 10:00:01"和"事件 B 在机器 2 上发生在 10:00:01"并不意味着它们同时发生。逻辑时钟（Lamport 时间戳、向量时钟）在不依赖物理时钟的情况下建立顺序。

- 共识：多台机器如何就一个值达成一致（例如，谁是领导者）？Raft 是标准共识算法。一群节点选举领导者。领导者处理所有写入。如果领导者失败，剩余节点选举新领导者。需要大多数（5 个节点中的 3 个）才能操作，所以它能容忍 $\lfloor(n-1)/2\rfloor$ 个故障。

- 分布式锁：确保只有一台机器执行关键操作。Redlock（基于 Redis）跨多个 Redis 实例获取锁。如果大多数实例授予锁，则获取锁。用于：防止重复模型部署、确保只有一个训练作业写入检查点。

## 微服务

![微服务 ML 架构：API 网关路由到特征、模型和日志服务，每个都有自己的数据库，通过消息队列连接](../images/microservices_architecture.svg)

- 微服务将系统分解为小的、独立部署的服务。每个服务拥有一个域：

```
┌─────────────┐  ┌──────────────┐  ┌─────────────┐
│ API 网关    │→ │ 特征服务     │→ │ 特征数据库  │
└─────────────┘  └──────────────┘  └─────────────┘
       │
       ├────────→ ┌──────────────┐  ┌─────────────┐
       │          │ 模型服务     │→ │ 模型存储    │
       │          └──────────────┘  └─────────────┘
       │
       └────────→ ┌──────────────┐  ┌─────────────┐
                  │ 日志服务     │→ │ 日志存储    │
                  └──────────────┘  └─────────────┘
```

- 优势：独立部署（更新模型服务而不触及特征服务）、独立扩展（根据请求负载扩展模型服务器，根据特征存储读取速率扩展特征服务器）、技术自由（模型服务用 Python，特征服务用 Go）。

- 劣势：网络开销（每个服务调用是网络往返）、复杂性（调试跨越多个服务）、数据一致性（没有跨服务事务）。

- 服务发现：API 网关如何找到模型服务？选项：基于 DNS（每个服务注册 DNS 名称）、K8s 服务（内置）或服务注册表（Consul、Eureka）。

- Saga 模式：对于跨越多个服务的操作（创建用户 + 分配资源 + 发送欢迎电子邮件），使用 saga：一系列本地事务，如果任何步骤失败则有补偿操作。

## 数据管道

- ML 系统消耗大量数据。数据管道移动、转换和服务这些数据：

### 批处理

- 定期（每小时、每天）处理大量数据。

- MapReduce：原始批处理范式。Map（独立转换每个记录）→ Shuffle（按键分组）→ Reduce（每组聚合）。概念简单但实现冗长。

- Apache Spark：现代批处理引擎。内存处理（对于迭代算法比 MapReduce 快 100 倍）。支持 SQL、DataFrame 和 ML 管道。大规模特征工程的标准。

- 示例：为推荐系统计算用户特征。输入：过去 30 天的 10 亿用户活动事件。输出：1 亿用户特征向量。每天作为 Spark 作业运行，输出到特征存储。

### 流处理

- 实时处理到达的数据（亚秒级延迟）。

- Apache Flink：领先的流处理引擎。精确一次处理、事件时间处理（按事件发生时间处理，而不是到达时间）、窗口（滚动、滑动、会话窗口）。

- Kafka Streams：轻量级流处理，内置于 Kafka。适合更简单的转换（过滤、聚合），无需部署单独的集群。

- 示例：实时欺诈检测。每笔信用卡交易是 Kafka 事件。Flink 作业计算运行统计信息（交易频率、位置变化）并在 100ms 内标记异常。

### Lambda 架构

- 结合批处理和流处理。批处理层提供准确、全面的结果（但有延迟）。速度层提供近似、实时结果。服务层合并两者。

- 在实践中，许多团队现在使用Kappa 架构：仅流处理，将流视为真实来源。流是可重放的（Kafka 保留事件），所以批处理可以通过重放流来模拟。

## ML 训练基础设施

- 训练前沿模型（1000 亿 + 参数）是大规模基础设施问题：数千个 GPU 运行数月，消耗兆瓦电力，生成 PB 数据，成本数千万美元。基础设施决定训练成功还是失败。

### GPU 集群

- 训练集群是通过高速网络连接的 GPU 服务器集合。关键组件：

![GPU 集群：每个节点有 8 个 GPU 通过 NVLink 连接，节点通过 InfiniBand 以胖树拓扑连接，从 64 扩展到 16,000+ GPU](../images/gpu_cluster_topology.svg)

- GPU 服务器（节点）：每个服务器有 4-8 个 GPU。典型配置：8 × H100 GPU、2 × AMD EPYC CPU、2 TB RAM、30 TB NVMe SSD。节点内的 GPU 通过NVLink连接（H100 上每个 GPU 900 GB/s），比 PCIe 快 30 倍。

- 集群大小：小型训练集群有 64-256 个 GPU（8-32 个节点）。前沿模型训练集群有 4,000-32,000 个 GPU（500-4000 个节点）。Meta 的 Llama 3 使用 16,384 个 H100 GPU。Google 在 TPU pod 上训练，有 8,000+ 芯片。

- 粗略估计：训练 70B 模型需要约 200 万美元计算成本。训练 400B+ 前沿模型需要约 5000 万 -1 亿美元。集群硬件本身成本约 5-10 亿美元（H100 价格 3 万美元×16,000 GPU = 4.8 亿美元）。

### 网络拓扑

- GPU 节点之间的网络是最关键的基础设施组件。如果 GPU 不能快速交换梯度，它们会空闲等待通信完成。

- InfiniBand 是 GPU 集群网络的标准。NVIDIA 的Quantum-2 InfiniBand每端口提供 400 Gb/s。每个节点通常有 8 个 InfiniBand 端口（每个 GPU 一个），给每个节点 400 GB/s 总分割带宽。

- RDMA（远程直接内存访问）：InfiniBand 支持 RDMA，直接在不同节点的 GPU 内存之间传输数据，无需涉及 CPU。这将延迟从约 100μs（TCP）降低到约 1μs，对于高效的梯度 all-reduce（第 6 章）至关重要。

- 网络拓扑很重要：胖树（Clos 网络）提供完整分割带宽（任何 GPU 可以全速与任何其他 GPU 通信）。更便宜的拓扑（ Rail 优化、3D 环面）提供较少带宽但成本更低。拓扑必须匹配并行策略：
    - 数据并行：跨所有 GPU all-reduce → 需要高分割带宽（胖树）。
    - 张量并行：节点内通信 → NVLink 处理这个（不需要网络）。
    - 流水线并行：相邻流水线阶段之间的通信 → 只需要特定节点对之间的带宽（Rail 优化即可）。

- 以太网替代方案：RoCE v2（融合以太网上的 RDMA）在标准以太网基础设施上提供 RDMA。比 InfiniBand 便宜但延迟更高、更多拥塞。Google 为一些 TPU pod 网络使用 RoCE。超以太网联盟正在为 AI 工作负载开发无损以太网。

### 训练存储

- 训练需要三个存储层：

    - 数据集存储：训练语料库（1-100 TB 文本，或 PB 级多模态数据）。存储在分布式文件系统或对象存储中。必须支持高吞吐量顺序读取（数据加载器大批量读取数据）。Lustre和GPFS是常见的 HPC 文件系统；云替代方案包括FSx for Lustre（AWS）和Filestore（GCP）。

    - 检查点存储：定期保存训练状态（模型权重 + 优化器状态 + 调度器状态）。对于混合精度的 70B 模型，带 Adam 优化器：每个检查点约 560 GB（70B × 4 字节×2 用于优化器）。3 个月运行每小时保存 = 约 2000 个检查点 = 1.1 PB。实践中，只保留最新 N 个检查点，旧的删除。必须足够快，检查点不会显著减慢训练。

    - 日志和指标：实验跟踪数据（损失曲线、学习计划、梯度范数）。相对较小但必须实时写入。W&B、MLflow 或 TensorBoard 处理这个。

- 存储瓶颈：16,000-GPU 集群加载训练批次需要持续读取约 100 GB/s 数据。如果文件系统不能维持这个吞吐量，GPU 会空闲等待数据。数据管道优化（预取、缓存、使用 WebDataset 或 Mosaic Streaming 的格式优化）至关重要。

### 作业调度

- GPU 集群服务于多个团队和项目。作业调度器分配 GPU 给训练作业：

- SLURM：标准 HPC 作业调度器。用户提交作业，指定 GPU 数量、内存和时间限制。SLURM 分配资源并管理队列。支持基于优先级的调度、抢占和团队之间的公平份额分配。

- 带 GPU 调度的 Kubernetes（第 18 章文件 02）：云原生方法。K8s GPU 设备插件将 GPU 暴露为可调度资源。Volcano和Run:ai添加 ML 特定调度功能：帮组调度（一次性为作业分配所有 GPU，而不是一次一个）、优先级队列和 GPU 时间共享。

- 调度挑战：
    - 碎片化：有 1000 个 GPU 的集群可能有 200 个空闲，但分布在 50 个节点上（每个节点 4 个空闲）。需要 128 个连续 GPU 的作业无法运行，即使总共有足够的 GPU。碎片整理（迁移作业以整合空闲 GPU）或拓扑感知调度（分配连接良好的 GPU）解决这个问题。
    - 优先级和抢占：紧急实验应该抢占低优先级作业。但抢占已运行 2 天的训练作业会浪费计算。调度器必须平衡优先级与效率。
    - 公平份额：团队应该随时间获得其分配的计算份额，即使团队提交的作业超过其份额。

### 容错

- 在数千个 GPU 运行数月的规模上，硬件故障不是异常——它们是常规的。16,000-GPU 集群的平均故障间隔时间以小时计，而不是月。

- 常见故障：GPU 内存错误（ECC 可纠正和不可纠正）、NVLink 故障（节点内 GPU 到 GPU 通信）、InfiniBand 链路故障（节点到节点通信）、节点崩溃（内核恐慌、电源故障）、存储故障（磁盘或控制器故障）。

- 检查点是主要防御。每 N 步保存完整训练状态（模型、优化器、数据加载器位置）。故障时：识别故障节点，替换或移除它，从最新检查点重新启动训练。故障成本是最后一次检查和故障之间的计算。

- 检查点频率权衡：频繁检查点（每 10 分钟）在故障时浪费较少计算但减慢训练（检查点 560 GB 需要时间）。不频繁检查点（每 2 小时）更快但在故障时浪费多达 2 小时计算。大多数团队每 20-60 分钟检查点。

- 弹性训练：现代框架（PyTorch Elastic、DeepSpeed）支持调整训练运行大小，无需重新启动。如果 500 个节点中有 2 个故障，训练继续用 498 个节点。故障节点被替换，训练在它们恢复上线时自动合并它们。

- 健康监控：持续监控所有 GPU（温度、内存错误、计算吞吐量）、网络链路（丢包、延迟）和存储（吞吐量、错误率）。异常自动告警。一些集群运行定期 GPU 健康检查（短计算测试），主动识别降级硬件，在故障之前。

- 大规模：训练 Meta 的 Llama 3（16,384 个 H100，54 天）经历了约 466 次作业中断。有效训练时间仅为挂钟时间的 90%——10% 损失于故障和恢复。实现 90%（而不是 50% 或 70%）的基础设施是区分可以训练前沿模型的组织与不能训练的组织的关键。

### 成本和效率

- 训练基础设施成本以 GPU 小时为主：

| 组件 | 总成本百分比 |
|-----------|----------------|
| GPU 计算 | 70-80% |
| 网络 (InfiniBand) | 10-15% |
| 存储 | 5-10% |
| 冷却和电力 | 5-10% |

- GPU 利用率（模型 FLOPS 利用率，MFU）测量 GPU 理论峰值性能的实际使用比例。H100 峰值是 989 TFLOPS（FP8）。实现 40-50% MFU 是好的；50-60% 是优秀的。差距是由于：通信开销（all-reduce、流水线气泡）、内存带宽限制、检查点和数据加载期间的空闲时间。

- 提高 MFU：重叠计算和通信（第 6 章）、使用高效注意力（Flash Attention，第 16 章）、优化数据加载（防止 GPU 饥饿）、减少检查点开销（异步检查点、先检查点到快速 NVMe、然后后台复制到持久存储）。

- 自建 vs 购买：小规模（<256 GPU），云更便宜（无前期成本，按小时付费）。大规模（>1000 GPU，持续使用 6+ 个月），拥有硬件更便宜（3 年 TCO 约低 2-3 倍）。大多数 AI 公司使用混合：拥有集群用于持续训练，云用于突发容量和实验。

## 数据库扩展

- 读副本：将读查询路由到主数据库的副本。主数据库处理写入，副本处理读取。由于大多数工作负载是读重（95%+ 读取），这随副本数量线性扩展读吞吐量。

- 分区（分片，来自文件 01）：跨多个数据库拆分数据。每个分区独立，实现并行读取和写入。挑战是跨分区查询（连接来自不同分片的数据）。

- 连接池：数据库有有限的连接容量。连接池（PostgreSQL 的 PgBouncer）跨请求重用连接，防止数百个服务实例尝试连接时的连接耗尽。

## 搜索和向量系统

### 文本搜索

- 倒排索引：文本搜索的基础。对于每个词，存储包含该词的文档列表。查询交叉每个查询词的列表。Elasticsearch 是标准：分布式、实时、支持全文搜索、聚合和地理空间查询。

- BM25：标准文本检索评分函数。根据词频、逆文档频率和文档长度归一化对文档评分。简单但有效——对于关键词重的查询仍然与神经方法竞争。

### 向量搜索

- 向量数据库存储嵌入（高维向量）并支持快速近似最近邻（ANN）搜索。给定查询嵌入，找到 $k$ 个最相似的存储嵌入。

- FAISS（Facebook AI Similarity Search）：一个库（不是数据库）用于 ANN 搜索。支持多种索引类型：
    - Flat：精确搜索，$O(n)$。用于小数据集或作为基准。
    - IVF（倒排文件）：将向量分区为簇，只搜索最近的簇。每查询 $O(n/k)$。
    - HNSW（分层可导航小世界）：基于图。构建分层图，从粗到细导航。非常快和准确，大多数应用的默认选择。
    - 乘积量化（PQ）：将向量压缩为紧凑代码，实现内存高效搜索。用准确性换取内存。

- 托管向量数据库：Pinecone、Weaviate、Milvus、Qdrant。它们处理 FAISS 不处理的扩展、复制和实时更新。

- 对于 RAG（检索增强生成）：用户查询 → 用文本编码器嵌入 → 在向量数据库中搜索相关文档 → 将检索到的文档添加到 LLM 提示。检索质量直接决定 LLM 响应的质量。

## 可观测性

- 可观测性是从系统外部输出了解系统内部发生什么的能力。三大支柱：

### 日志

- 结构化日志（JSON）是可搜索和可解析的。非结构化日志（"ERROR: 出错了"）不是。始终记录：时间戳、服务名称、请求 ID（用于跨服务跟踪）、严重性和相关上下文。

- ELK 栈（Elasticsearch、Logstash、Kibana）：标准日志管道。Logstash 收集和转换日志，Elasticsearch 索引它们，Kibana 可视化和搜索。

### 指标

- 指标是随时间的数值测量：请求率、错误率、延迟百分位数、GPU 利用率、队列深度。Prometheus 从服务抓取指标；Grafana 在仪表板中可视化它们，带告警。

- RED 方法用于服务：Rate（请求/秒）、Errors（错误率）、Duration（延迟）。监控每个服务的这些。

- USE 方法用于资源：Utilisation（使用中百分比）、Saturation（队列深度）、Errors。监控每个资源（CPU、GPU、内存、磁盘、网络）的这些。

### 跟踪

- 分布式跟踪跟随单个请求跨越多个服务。用户请求命中 API 网关 → 特征服务 → 模型服务 → 后处理。跟踪记录每个跳的时间，显示延迟花费在哪里。

- OpenTelemetry：跟踪、指标和日志的开放标准。用一次插桩代码，导出到任何后端（Jaeger、Zipkin、Datadog）。

## 可靠性

- SLO（服务水平目标）：目标准确性。"99.9% 的请求在<200ms 内完成。"这给出具体的错误预算：0.1% 的请求（每月约 43 分钟）可以慢或失败。

- SLI（服务水平指标）：测量。"过去 5 分钟的第 99 百分位延迟。"

- SLA（服务水平协议）：带后果的合同承诺。"如果可用性低于 99.95%，客户获得积分。"

- 错误预算：如果你的 SLO 是 99.9%，你在 99.99%，你有预算进行风险变更（部署新模型、迁移数据库）。如果你在 99.85%，冻结所有变更，专注于可靠性。错误预算将可靠性从抽象目标转化为可测量资源。

- 混沌工程：故意注入故障（杀死服务器、添加网络延迟、损坏数据）测试系统是否正确处理它们。Netflix 的 Chaos Monkey 随机终止生产实例。如果系统保持运行，它有弹性。如果它崩溃，你在用户发现之前找到了 bug。

## CI/CD

- 持续集成：自动构建和测试每个代码变更。每次推送触发：lint、类型检查、单元测试、集成测试。如果任何失败，变更被拒绝。这在 bug 到达生产之前捕获它们。

- 持续部署：自动部署通过 CI 的变更。部署策略：

    - 蓝绿：运行两个相同环境（蓝=当前，绿=新）。瞬间从蓝切换到绿。如果绿失败，切换回蓝（瞬间回滚）。

    - 金丝雀：路由一小部分流量（1-5%）到新版本。监控错误。如果指标看起来好，逐渐增加流量。这限制坏部署的影响范围。

    - 功能标志：部署新代码但隐藏在标志后。为部分用户启用标志（内部测试者，然后测试用户，然后每个人）。解耦部署（代码上线）和发布（用户看到功能）。

- 对于 ML：CI/CD 包括模型特定步骤。模型变更触发：单元测试（形状测试、梯度检查）、在保留集上评估（准确性不能回退）、影子部署（新模型与旧模型一起运行，比较输出）和逐渐推出（金丝雀从 1% → 100%）。
