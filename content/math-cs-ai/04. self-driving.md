# 自动驾驶汽车

*自动驾驶汽车是商业化最先进的自主系统，将感知、预测、规划和控制集成到单一车辆中。本章涵盖自动驾驶技术栈、高精地图、运动预测、规划、端到端驾驶、仿真、安全标准和自主化等级。*

- 自动驾驶汽车可以说是在大规模上尝试的最难机器人问题。与在受控环境中运行的工厂机器人不同，自动驾驶汽车必须应对开放世界：不可预测的人类驾驶员、乱穿马路的行人、一夜之间出现的施工区域，以及每分钟都在变化的天气。

- 风险也特别高。自动驾驶汽车在高速公路上行驶，周围是脆弱的道路使用者。对于安全关键故障，错误容忍度接近于零。

## 自动驾驶技术栈

- 经典的自动驾驶架构是模块化流水线，包含四个阶段，每个阶段馈送到下一个：

$$\text{感知} \to \text{预测} \to \text{规划} \to \text{控制}$$

![自动驾驶技术栈：传感器馈送感知，馈送预测，规划，最后是控制](../images/autonomous_driving_stack.svg)

- 感知（本章文件 1 涵盖）将原始传感器数据处理为结构化场景表示：检测到的物体具有 3D 位置、速度和类别标签；车道标记；交通灯；可行驶表面边界。

- 预测预测其他智能体（车辆、行人、骑自行车者）未来将如何移动。给定场景的当前状态，预测模块输出每个智能体在时间范围内（通常是未来 3-8 秒）的轨迹。

- 规划决定自车应该做什么：遵循哪条路径、何时变道、何时让行、何时加速或刹车。它获取预测的场景并为自车生成安全、舒适且朝着目的地前进的轨迹。

- 控制将规划的轨迹转换为执行器命令：转向角、油门和刹车。这是最低级别，将抽象轨迹转换为物理运动。

- 模块化设计具有清晰的工程优势：每个模块可以独立开发、测试和改进。但它也有弱点：错误会向下游传播（漏检对规划器不可见），并且信息在每个接口处丢失（规划器看到边界框，而不是产生它们的丰富传感器数据）。

## 高精地图

- 高精（HD）地图是详细的、厘米级精度的数字地图，编码道路结构：车道边界、车道连通性（哪个车道在交叉口连接到哪个车道）、交通标志位置、限速、人行横道位置和路面高程。

- HD 地图为驾驶任务提供强先验。感知模块不需要每帧从头开始发现车道边界；它只需要在地图内定位车辆并验证实况与存储的结构匹配。这大大简化了规划。

- 构建 HD 地图需要配备高端 LiDAR、摄像头和 RTK-GPS 的专用测量车辆。随着道路变化，地图必须维护和更新。这很昂贵，不容易扩展到地球上的每条道路。

- 无地图驾驶（也称为"在线建图"）旨在消除对预建 HD 地图的依赖。车辆改为从传感器实时构建局部地图。像 MapTR 和 MapTRv2 这样的模型使用 Transformer 架构直接从相机图像预测矢量化地图元素（车道中心线、道路边界、人行横道），将折线输出为有序点序列。

- 无地图方法用地图精度换取可扩展性：汽车可以行驶的任何道路都可以建图。但它要求感知系统足够稳健，能够实时检测所有相关道路结构，包括在复杂交叉口、高速公路匝道和施工区域。

- 在实践中，许多系统使用混合方法：来自现有地图提供商的轻量级地图（具有粗略道路拓扑），由车辆的传感器实时丰富。

## 运动预测

- 预测其他道路使用者将去哪里是自动驾驶中最难的子问题之一。人类是不可预测的，意图是隐藏的，可能未来的空间迅速分支。

- 预测模型的输入是场景上下文：所有检测到的智能体在过去的位置和速度（通常是 1-2 秒历史），加上静态上下文（车道几何、交通信号、道路边界）。

- 输出是每个智能体的一组预测轨迹，通常覆盖未来 3-8 秒。由于未来不确定，好的预测模型输出多个可能的轨迹及相关概率，而不是单一的点估计。

- 轨迹预测作为回归问题：预测每个智能体在离散未来时间步的未来 $(x, y)$ 坐标。损失通常是 $K$ 个预测轨迹的最小平均位移误差（minADE）：

$$\text{minADE}_K = \min_{k \in \{1, \ldots, K\}} \frac{1}{T} \sum_{t=1}^{T} \| \hat{\mathbf{p}}_t^{(k)} - \mathbf{p}_t \|_2$$

- 这是一个"K 选最佳"指标：如果模型的 $K$ 个预测中有任何一个接近真实值，模型就得分。这鼓励多样化、多模态预测。

- 社会力模型将行人行为建模为动力系统，其中每个人经历吸引力（朝向目标）和排斥力（远离其他行人和障碍物）。人 $i$ 的加速度是：

$$\mathbf{a}_i = \frac{\mathbf{v}_i^{\text{desired}} - \mathbf{v}_i}{\tau} + \sum_{j \neq i} \mathbf{f}_{ij}^{\text{repulsive}} + \sum_{\text{walls}} \mathbf{f}_{\text{wall}}$$

- 这是一个与本 chapter 文件 2 中的机器人动力学方程相似的微分方程组。该模型很优雅，但依赖于手工调整的力参数，并且在复杂多智能体交互方面表现不佳。

- 图神经网络（GNN） 用于预测，将场景建模为图：每个智能体是一个节点，边表示空间关系（接近度、车道共享）。节点之间的消息传递捕捉交互："这辆车正在给那个行人让行"或"这两辆车正在合并到同一车道"。

- 现代预测架构（如 MTR、QCNet）使用基于 Transformer 的模型，联合推理智能体历史、地图上下文和智能体 - 智能体交互。智能体通过交叉注意力关注相关地图特征（它们当前所在的车道、前方交叉口）和其他智能体（前方的车、人行横道的行人）。输出是自回归生成的或通过混合模型生成的一组轨迹假设。

- 目标条件预测 首先预测智能体可能去哪里（一组候选目标点，如车道端点或交叉口出口），然后预测到达每个目标的轨迹。这将问题分解为"去哪里"（离散的、可管理的）和"如何去"（给定目标的连续路径），使多模态预测问题更容易处理。

## 规划

- 给定预测的场景，规划器必须为自车生成轨迹。这是一个约束优化问题：找到安全、舒适、高效且合法的轨迹。

- 基于规则的规划器 将驾驶行为编码为 if-then 规则集："如果人行横道上有行人，让行"、"如果与前车的间隙小于 2 秒，不要变道"、"如果接近红灯，减速在停车线停止"。这些规则可解释且可审核，但对于复杂场景变得笨拙（数千条规则，许多边缘情况、规则之间的交互）。

- 基于优化的规划器 将驾驶 formul 化为轨迹优化。自车轨迹被参数化（例如作为未来时间步的 $(x, y, \theta, v)$ 状态序列），并最小化目标函数：

$$\min_{\boldsymbol{\xi}} \underbrace{w_1 \cdot J_{\text{progress}}(\boldsymbol{\xi})}_{\text{到达目的地}} + \underbrace{w_2 \cdot J_{\text{comfort}}(\boldsymbol{\xi})}_{\text{平稳行驶}} + \underbrace{w_3 \cdot J_{\text{safety}}(\boldsymbol{\xi})}_{\text{避免碰撞}}$$

$$\text{约束：运动学约束、限速、车道边界}$$

- 进度项惩罚与期望路线的偏差。舒适项惩罚高横向加速度、加加速度（加速度的导数）和突然转向，因为乘客能感觉到这些。安全项惩罚与其他智能体的接近程度，使用预测轨迹评估碰撞风险。

- 这是约束优化（第 3 章）：在不等式约束下最小化成本函数。权重 $w_1, w_2, w_3$ 权衡竞争目标（激进驾驶更快但不太舒适且不太安全）。

- 基于学习的规划器 使用在人类驾驶数据上训练的神经网络生成轨迹。模型观察场景并直接输出规划轨迹，从专家人类驾驶示例中隐式学习复杂的权衡。

- 优点是 holistic 捕捉人类驾驶行为，包括微妙、难以形式化的方面：合并时多激进、何时在交叉口轻轻向前移动、给骑自行车者多少空间。缺点与模仿学习中的分布偏移问题相同（文件 2）：模型在训练数据中未充分代表的情况下可能表现不可预测。

## 端到端驾驶

- 端到端驾驶 完全移除模块化边界。单个神经网络获取原始传感器输入（相机图像、LiDAR 点云）并直接输出驾驶命令（转向、油门、刹车）或规划的轨迹。没有单独的感知、预测或规划模块。

- 吸引力在于整个系统为最终任务（安全驾驶）联合优化，因此在模块边界处不会丢失信息。感知模块学习提取规划器需要的确切特征，而不是可能未捕获任务相关细节的通用物体检测。

- UniAD（统一自动驾驶）是里程碑式的端到端架构。它通过 BEV 编码器处理多相机图像，然后应用级联基于 Transformer 的模块：跟踪、在线建图、运动预测、占据预测和规划。虽然它有内部模块，但它们都是可微的并联合端到端训练，规划损失通过整个网络反向传播。

- UniAD 中的规划模块通过关注预测的 BEV 特征、预测的智能体轨迹和预测的占据来生成未来自车辆航点。这是多元链式法则（第 3 章）的实际应用：梯度从规划损失一直流回到图像编码器，告诉感知特征如何对规划更有用。

- 最近的端到端方法使用 VLA 风格架构（本章文件 3）。像 DriveVLM 这样的模型采用相机图像和导航指令（或路线），并使用 VLM 主干产生驾驶动作。这直接将大规模预训练（视觉理解、推理）的优势带入驾驶技术栈。

- 端到端驾驶的张力是可解释性。模块化系统可以报告"我检测到 (x, y) 处有行人并预测他们将穿过"——故障模式是可诊断的。端到端系统是产生转向角的黑盒。当它失败时，诊断为什么很困难，这对安全认证是严重问题。

## 驾驶世界模型

- 世界模型 学习根据当前状态和自车动作预测驾驶场景的未来状态：$p(s_{t+1} \mid s_t, a_t)$（如第 10 章介绍）。在驾驶中，这意味着生成真实的未来帧或 BEV 布局："如果我加速并向左转向，3 秒后场景将看起来像这样"。

- 世界模型为自动驾驶提供两种强大能力：

    - 基于想象的规划：不是承诺一个动作并查看发生什么，规划器可以通过世界模型展开多个候选轨迹，评估每个轨迹的安全性和舒适性，然后选择最佳。这是应用于驾驶的基于模型的 RL（本章文件 2 涵盖）。

    - 学习仿真：在真实驾驶数据上训练的世界模型实际上是数据驱动的仿真器。它生成真实的场景（包括罕见的边缘情况），无需手动构建手工仿真器。关键是，它捕捉真实驾驶的统计模式：其他驾驶员实际如何行为、光线如何变化、雨如何影响能见度。

- GAIA-1（Wayve）是用于驾驶的生成世界模型。给定过去相机帧和自车动作序列，它自回归生成未来视频帧。它使用以动作输入为条件的视频扩散架构。模型学习生成合理的未来：遵守交通规则的车辆、在人行道上行走的行人、正确转换的交通灯，所有这些都从训练数据中涌现，而非编程规则。

- DriveDreamer 和 GenAD 采用类似方法，但在 BEV 空间而非像素空间操作。预测未来 BEV 布局比生成完整视频帧更紧凑（类似于 DreamerV3 在机器人学中在潜在空间而非像素空间预测，如文件 2 讨论）。BEV 世界模型预测所有智能体将在哪里、道路结构将是什么样子以及哪里存在自由空间，规划器直接使用这个。

- 神经闭环仿真 使用世界模型替换手工构建的仿真器进行测试。给定真实驾驶日志作为起点，世界模型生成如果自车采取不同动作会发生什么。这实现反事实评估："如果我晚 0.5 秒刹车会怎样？"而无需实际重现场景。

- 与 JEPA 框架（第 10 章）的联系在这里很自然。驾驶世界模型不需要预测像素完美的未来（每个像素的确切 RGB 值）。它们需要预测对规划重要的方面：智能体在哪里、它们移动多快、哪里存在自由空间。嵌入空间预测（JEPA 风格）捕捉这些语义上有意义的属性，而不会在无关视觉细节（如精确云纹理）上浪费容量。

- 主要挑战是长视野保真度。世界模型随时间累积误差：第 2 帧的小错误会偏移所有后续帧。对于驾驶，3 秒预测视野对战术决策（我现在应该合并吗？）很有用，但 30 秒视野（路线规划等战略决策所需）仍然不可靠。当前的工作通过重新锚定（定期用真实观察重置模型）和不确定性估计（标记预测何时变得不可靠）来缓解这个问题。

## 仿真

- 通过在真实道路上驾驶来测试自动驾驶汽车是必要但不充分的。危险场景（接近碰撞、边缘情况）很罕见，因此通过行驶里程测试效率低下。汽车需要行驶数亿英里才能在统计上证明安全性，这是不可行的。

- 仿真 提供无限、可控和安全的测试。在现实世界中罕见的场景（孩子跑到路上、轮胎爆裂、突然障碍物）可以在仿真中测试数百万次。

- CARLA 是建立在 Unreal Engine 上的开源驾驶仿真器。它提供逼真的城市环境、动态天气、交通智能体和传感器仿真（相机、LiDAR、雷达）。研究人员使用 CARLA 训练基于 RL 的驾驶智能体并评估感知算法。

- nuPlan（Motional）是闭环规划基准。与开环评估（重放记录的数据并将规划器的输出与人类驾驶员的实际轨迹比较）不同，闭环评估让规划器的决策影响仿真：如果规划器决定变道，仿真相应演变。这测试反应行为，而不仅仅是轨迹相似性。

![开环重放日志无交互；闭环让模型的动作改变仿真状态](../images/open_vs_closed_loop.svg)

- 开环和闭环评估之间的区别至关重要：

    - 开环：重放记录的场景，计算模型的输出与人类驾驶员动作的相似程度。这很容易设置但具有误导性：总是预测"直行"的模型在高速公路上可能误差很低，但在第一个转弯处会撞车。

    - 闭环：模型的动作改变仿真状态，仿真响应演变。这测试模型从自身错误中恢复和应对动态情况的能力。它昂贵得多，但意义深远。

- 场景生成 创建压力测试系统的测试用例。对抗场景（车辆突然刹车、隐藏在停放车辆后面的行人）通过优化生成自驾驶系统表现最糟的情况。这与 ML 中的对抗训练（第 6 章）相关：找到最大化损失的输入。

## 安全

- 自动驾驶中的安全由工程标准管理，而不仅仅是 ML 指标。

- ISO 26262（功能安全）是安全关键电子系统的汽车标准。它定义汽车安全完整性等级（ASIL） 从 A（最低）到 D（最高），基于潜在危害的严重性、暴露度和可控性。自动驾驶系统的感知和规划组件通常是 ASIL-D，这是最高级别，需要广泛验证、冗余和故障安全设计。

- SOTIF（预期功能安全，ISO 21448）解决不同类别的危害：不是硬件故障（ISO 26262 涵盖），而是系统按设计运行但仍产生不安全结果的情况。将白色卡车误分类为天空的感知模型（真实事件）是 SOTIF 问题：硬件工作正常，但算法的局限性导致危害。

- 运行设计域（ODD） 定义自动驾驶系统设计为运行的条件：特定地理区域、道路类型（仅高速公路、城市、两者）、天气条件（无大雪）、速度范围和时间。在 ODD 之外运行是不允许的：如果系统不能处理雪，它就不应在雪中驾驶。

- 故障安全与故障运行设计：
    - 故障安全：检测到故障时，系统过渡到安全状态（例如靠边停车）。这是最低要求。
    - 故障运行：系统使用冗余组件在故障情况下继续安全运行。具有冗余转向、制动和计算的自动驾驶汽车可以在单个组件故障后幸存并仍然驾驶到安全位置。

- 冗余 是根本。关键感知传感器被复制：多个相机覆盖重叠视场、LiDAR 和雷达提供独立深度测量、运行相同软件的双计算平台。如果任何单个组件故障，其他组件提供足够信息以安全驾驶。

## 自主化等级

![SAE 自主化等级从 L0（无自动化）到 L5（全自动化），显示责任从哪里从人类转移到系统](../images/sae_autonomy_levels.svg)

- SAE J3016 标准定义六个驾驶自动化等级，从 0（无自动化）到 5（全自动化）：

    - Level 0（无自动化）：人类做一切。系统可以提供警告（车道偏离警报）但不控制车辆。

    - Level 1（驾驶辅助）：系统控制转向或速度，但不同时控制两者。自适应巡航控制（维持速度和跟随距离）或车道保持辅助（保持车辆在车道中央）是 Level 1。

    - Level 2（部分自动化）：系统同时控制转向和速度，但人类必须始终监控并准备接管。Tesla Autopilot、GM Super Cruise 和大多数当前"自动驾驶"功能是 Level 2。人类仍然是负责的驾驶员。

    - Level 3（条件自动化）：系统驾驶并监控环境，但仅在特定条件下（ODD）。人类可以脱离但必须在系统请求时准备接管（通常有 10 秒以上缓冲时间）。Mercedes Drive Pilot（某些高速公路、低于 60 km/h）是首个认证的 Level 3 系统。

    - Level 4（高度自动化）：系统在 ODD 内驾驶并处理所有情况，无需人类干预。如果遇到 ODD 之外的情况，它可以安全停止。Waymo 的 robotaxi 服务在特定地理区域以 Level 4 运营。

    - Level 5（完全自动化）：系统在任何人类可以驾驶的地方、任何条件下驾驶。不需要方向盘或踏板。这还不存在。

- 关键区别是谁对安全负责。在 Level 0-2，人类负责。在 Level 3-5，系统负责（在其 ODD 内）。这具有深远的法律、保险和伦理影响。

- 当前行业状态是 Level 2（广泛部署）、Level 3（开始部署）和 Level 4（有限地理部署）的混合。Level 5 仍然是长期研究目标。

import jax
import jax.numpy as jnp
import matplotlib.pyplot as plt

# Trajectory: N waypoints, each (x, y)
N = 20

goal = jnp.array([10.0, 0.0])
obstacle = jnp.array([5.0, 0.0])

    wp = jnp.concatenate([start[None], waypoints, goal[None]], axis=0)

    accel = wp[2:] - 2 * wp[1:-1] + wp[:-2]
    smooth_cost = jnp.sum(accel  2)

    collision_cost = jnp.sum(jnp.maximum(0, obs_radius + 0.5 - dists)  2)

import jax.numpy as jnp
import matplotlib.pyplot as plt

# Ground truth: vehicle turning right
dt = 0.1
T = 40  # 4 seconds
v = 10.0  # m/s
omega = 0.3  # rad/s (turning rate)

# True trajectory (constant turn rate)
t = jnp.arange(T) * dt
theta = omega * t
gt_x = (v / omega) * jnp.sin(theta)
gt_y = (v / omega) * (1 - jnp.cos(theta))

# Constant velocity prediction from t=0
# Assumes the car continues straight at its current heading
obs_steps = 10  # observe first 1 second
vx0 = v * jnp.cos(theta[obs_steps - 1])
vy0 = v * jnp.sin(theta[obs_steps - 1])
pred_t = jnp.arange(T - obs_steps) * dt
pred_x = gt_x[obs_steps - 1] + vx0 * pred_t
pred_y = gt_y[obs_steps - 1] + vy0 * pred_t

plt.figure(figsize=(8, 6))
plt.plot(gt_x[:obs_steps], gt_y[:obs_steps], "ko-", label="Observed")
plt.plot(gt_x[obs_steps:], gt_y[obs_steps:], "g-", linewidth=2, label="True future")
plt.plot(pred_x, pred_y, "r--", linewidth=2, label="Constant velocity prediction")

3. 实现简单基于规则的规划器，根据检测到的障碍物在车道保持和停止之间决策。
```python
import jax.numpy as jnp

def rule_based_planner(ego_speed, obstacles, speed_limit=13.9):
    """
    Simple rule-based planner.
    ego_speed: current speed (m/s)
    obstacles: list of (distance, speed) tuples for vehicles ahead
    speed_limit: max allowed speed (m/s), default ~50 km/h

    Returns: (target_speed, action_label)
    """
    min_following_distance = 2.0 * ego_speed  # 2-second rule
    emergency_distance = 5.0  # metres

    if not obstacles:
        return speed_limit, "cruise"

    # Find closest obstacle ahead
    closest_dist, closest_speed = min(obstacles, key=lambda o: o[0])

    if closest_dist < emergency_distance:
        return 0.0, "EMERGENCY STOP"
    elif closest_dist < min_following_distance:
        # Match speed of vehicle ahead
        target = min(closest_speed, speed_limit)
        return target, "following"
    else:
        return speed_limit, "cruise"

# Test scenarios
scenarios = [
    (13.9, [], "Empty road"),
    (13.9, [(30.0, 10.0)], "Slower car ahead"),
    (13.9, [(3.0, 0.0)], "Stopped car very close"),
    (13.9, [(50.0, 13.9)], "Car ahead at same speed"),
]

for speed, obs, desc in scenarios:
    target, action = rule_based_planner(speed, obs)
    print(f"{desc:30s}  →  {action:15s} target_speed={target:.1f} m/s ({target*3.6:.0f} km/h)")
```
