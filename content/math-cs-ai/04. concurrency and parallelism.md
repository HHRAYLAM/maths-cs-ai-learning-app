# 并发和并行

*并发和并行是程序同时做多件事的方式。本文件涵盖并发与并行的区别、同步原语、经典并发问题、死锁、无锁数据结构、并行编程模型、异步编程和缩放定律，这些是支撑多线程服务器、分布式训练和每个现代应用的概念。*

- 单个 CPU 核心一次执行一条指令。但现代系统有 8、64 甚至数千个核心（GPU）。即使在单个核心上，我们想处理多个任务：下载文件时渲染 UI 时处理用户输入。**并发**和**并行**是管理多个活动的两种策略。

## 并发与并行

![并发在一个核心上交错任务；并行在多个核心上同时运行任务](../images/concurrency_vs_parallelism.svg)

- **并发**是关于*管理*多个任务。任务通过交错取得进展：任务 A 运行一会儿，然后任务 B，然后回到 A。在单个核心上，并发创造同时执行的假象。任务不是真正同时；它们轮流。

- **并行**是关于*执行*多个任务同时。有$n$个核心，$n$个任务可以真正同时运行。并行需要多个硬件执行单元。

- 类比：并发是一个厨师交替切菜和搅拌锅。并行是两个厨师，每个同时做一个任务。系统可以并发但不并行（一个核心，交错任务）、并行但不并发（多个核心运行独立程序无交互）或两者兼有（多个核心运行交错的交互任务）。

- 在 ML 中，并发出现在数据加载中（重叠数据预处理与 GPU 计算），而并行出现在分布式训练中（多个 GPU 同时计算梯度，第 6 章）。

## 同步原语

- 当多个线程共享数据时，**同步**防止竞争条件。当结果取决于线程执行的不可预测顺序时发生竞争条件。

- 考虑两个线程都递增共享计数器：`counter += 1`。这实际上是三个运算：（1）读计数器，（2）加 1，（3）写计数器。如果两个线程读相同值（比如 5），都加 1，都写 6，计数器最终是 6 而不是正确 7。丢失一个递增。

- **互斥锁**（mutual exclusion lock）确保一次只有一个线程访问临界区。线程在进入临界区前**获取**锁，在之后**释放**锁。任何其他线程尝试获取已持有锁时阻塞直到释放。

```
lock.acquire()
counter += 1      # 一次只有一个线程在这里
lock.release()
```

- 互斥锁正确但引入**争用**：如果许多线程竞争同一锁，它们花时间等待而不是计算。这限制可扩展性。极端情况，所有线程想要同一锁，序列化整个程序。

- **信号量**泛化互斥锁。计数信号量维护计数器：`wait()` 递减计数器（如果会变负则阻塞），`signal()` 递增它。初始化为 1 的信号量行为像互斥锁。初始化为$n$ 的信号量允许最多$n$ 个线程同时进入临界区（用于数据库连接等资源池）。

- **条件变量**让线程等待直到特定条件满足。线程释放锁，在条件变量上等待，当另一个线程发信号条件时被唤醒。这避免忙等待（在循环中重复检查条件，浪费 CPU）。

- **监视器**将互斥锁与条件变量和共享数据捆绑成单个抽象。Java 的 `synchronized` 关键字和 Python 的 `threading.Condition` 实现类似监视器语义。

- **读写锁**区分读取者（可以共享访问，因为读取不修改数据）和写入者（需要独占访问）。多个读取者可以同时持有锁，但写入者阻塞所有读取者和其他写入者。当读取远多于写入时这是最优的（例如，缓存模型服务预测）。

## 经典并发问题

- **生产者 - 消费者**（有界缓冲区）：生产者生成项目并放入固定大小缓冲区；消费者移除项目。挑战：缓冲区满时生产者必须等待，空时消费者必须等待，两者必须避免损坏缓冲区。

- 解决方案使用两个信号量（一个计数空槽，一个计数满槽）加上缓冲区本身的互斥锁。这是大多数消息队列、日志系统和数据管道背后的模式。

- **读者 - 写者**：多个读者可以同时读取，但写入者需要独占访问。挑战是公平性：如果读者不断到达，写入者可能饿死（永远得不到访问）。解决方案优先读者或写入者，或公平交替。

- **哲学家进餐**：五个哲学家坐在桌子周围，中间有五个叉子。每个需要两个叉子吃饭。如果五个同时拿起左叉，没人能拿起右叉，所有人都饿死（死锁）。解决方案包括：原子地拿起两个叉子、引入不对称（一个哲学家先拿右叉）或使用服务员（信号量限制用餐者为 4）。

## 死锁

- **死锁**发生在一组线程每个都在等待集合中另一个线程持有的资源，创建依赖循环时。没人能继续。

![死锁：线程 A 持有锁 1 并想要锁 2，线程 B 持有锁 2 并想要锁 1 — 循环等待](../images/deadlock_cycle.svg)

- 死锁的四个**必要条件**（必须同时成立）：

    1. **互斥**：资源只能被一个线程持有。
    2. **持有并等待**：线程持有一个资源同时等待另一个。
    3. **不可抢占**：资源不能强制从线程拿走。
    4. **循环等待**：等待 - for 图中存在循环。

- **死锁预防**打破四个条件之一：
    - 消除循环等待：对资源强加全序。所有线程以相同顺序获取资源。如果每个线程总是先获取锁 A 再锁 B，循环不可能。
    - 消除持有并等待：要求线程一次性请求所有资源（原子地）。

- **死锁避免**动态决定是否授予资源请求可能导致死锁。**银行家算法**维护每个线程的最大可能需求，只授予使系统处于"安全状态"的请求（所有线程最终能完成的状态）。算法是$O(n^2 m)$ 每请求（$n$线程，$m$ 资源类型），对大多数真实系统太昂贵。

- **死锁检测**让死锁发生，然后检测它们（通过在等待 -for 图中找循环）并恢复（通过杀死线程或回滚事务）。

- 在实践中，大多数系统对常见情况使用预防（资源排序），对罕见情况使用检测。数据库系统是经典示例：它们检测事务间死锁并中止一个打破循环。

## 无锁和免等待数据结构

- 锁引入争用、优先级反转和死锁风险。**无锁**数据结构完全避免锁，使用硬件提供的**原子运算**。

- 关键原子运算是**比较并交换（CAS）**：原子地检查内存位置是否有期望值，如果是，用新值替换它。伪代码：

```
CAS(address, expected, new_value):
    if *address == expected:
        *address = new_value
        return true
    else:
        return false
```

- CAS 作为单个硬件指令实现，所以即使无锁也是原子的。无锁算法在重试循环中使用 CAS：读当前值，计算新值，尝试 CAS。如果另一个线程同时修改值，CAS 失败，线程重试。

- **无锁**：至少一个线程在有限步数内取得进展（不可能死锁，但单个线程可能在争用下无限重试）。

- **免等待**：每个线程在有界步数内取得进展（最强保证，但最难实现）。

- 无锁栈、队列和哈希表广泛用于高性能系统。Java 的 `ConcurrentHashMap` 和 Go 的原子运算构建在 CAS 上。

## 并行编程模型

- **共享内存**并行：所有线程访问相同内存空间。同步是程序员的责任。**OpenMP** 提供编译器指令并行化循环：

```c
#pragma omp parallel for
for (int i = 0; i < n; i++) {
    result[i] = compute(data[i]);
}
```

- 编译器在可用核心间分割循环迭代。OpenMP 对数据并行工作负载（对许多数据点相同运算）有效，广泛用于科学计算。

- **消息传递**并行：每个进程有自己的内存。通信通过发送和接收消息发生。**MPI**（Message Passing Interface）是跨节点分布式计算的标准：

```c
MPI_Send(data, count, MPI_FLOAT, dest, tag, MPI_COMM_WORLD);
MPI_Recv(data, count, MPI_FLOAT, src, tag, MPI_COMM_WORLD, &status);
```

- MPI 扩展到数千节点因为无共享状态同步。分布式深度学习（第 6 章）使用集体运算如 `MPI_AllReduce`（环 all-reduce）跨 GPU 同步梯度。

- **GPU 并行**遵循**SIMT**（单指令多线程）模型：数千线程在不同数据上执行相同指令。这对矩阵运算（第 2 章）完美，其中相同乘加应用于每个元素。我们将在后面章节详细涵盖 GPU 编程。

## 异步和事件驱动编程

- 不是所有并发需要线程。**异步**编程使用**事件循环**用单个线程处理许多 I/O 绑定任务。

- 事件循环维护任务队列。当任务需要等待 I/O（网络响应、文件读取）时，它注册回调并交出控制。事件循环选择下一个就绪任务。当 I/O 完成，回调入队并最终执行。等待期间无线程阻塞。

- **协程**是可以挂起和恢复的函数。`async/await` 语法（Python、JavaScript、Rust）使协程看起来像常规顺序代码：

```python
async def fetch_data(url):
    response = await http_get(url)  # 这里挂起，事件循环运行其他任务
    return process(response)         # 响应到达时恢复
```

- `await` 关键字挂起协程并将控制返回事件循环。当等待操作完成，协程在离开处恢复。这是协作式多任务：协程自愿交出，与 OS 强制切换线程的抢占式多任务不同。

- 异步对**I/O 绑定**工作负载理想，带许多并发连接（Web 服务器处理数千客户端）。它不适合**CPU 绑定**工作（单线程事件循环不能利用多个核心）。对于 CPU 绑定工作，使用线程或进程。

- Python 的**全局解释器锁（GIL）** 防止线程真正并行：一次只有一个线程可以执行 Python 字节码。这就是为什么 Python 对 CPU 并行使用多进程（独立进程，每个有自己的解释器），对 I/O 并发使用异步。GIL 正在 Python 3.13+ 中移除（无线程 Python），这将启用真正多线程并行。

## 缩放定律

- **阿姆达尔定律**描述并行化程序的理论上加速。如果程序分数$p$ 可并行化，其余$1 - p$ 是串行：

$$\text{Speedup}(n) = \frac{1}{(1-p) + \frac{p}{n}}$$

![阿姆达尔定律：串行部分限制最大加速 — 10% 串行意味着最大 10 倍，无论多少核心](../images/amdahl_serial_bottleneck.svg)

- 其中$n$ 是处理器数量。当$n \to \infty$，最大加速接近$\frac{1}{1-p}$。如果 95% 程序可并行，最大加速是$\frac{1}{0.05} = 20\times$，无论添加多少核心。串行部分是瓶颈。

- 这对 ML 有深远影响：如果数据加载占用 10% 训练时间且串行，添加更多 GPU 最多加速训练 10 倍。10% 串行瓶颈限制一切（这就是为什么高效数据管道和重叠计算与 I/O 重要，第 6 章）。

- **古斯塔夫定律**提供更乐观视角。不是固定问题大小添加处理器，它固定总时间并问可以做多少更多工作。如果并行部分随问题大小缩放：

$$\text{Speedup}(n) = 1 - p + p \cdot n$$

- 这在$n$中线性。论点：有更多处理器，我们解决更大问题，不是更快解决相同问题。在 ML 中，这对应于有更多 GPU 时增加批量大小（弱缩放）而不是保持批量大小固定（强缩放）。

## 编码任务（使用 CoLab 或 notebook）

1. 演示竞争条件。两个线程递增共享计数器无同步并观察丢失更新。
```python
import threading

counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1  # 非原子：读、加、写

threads = [threading.Thread(target=increment, args=(100000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Expected: {4 * 100000}")
print(f"Actual:   {counter}")
print(f"Lost updates: {4 * 100000 - counter}")
```

2. 用锁修复竞争条件并测量开销。
```python
import threading
import time

lock = threading.Lock()
counter = 0

def increment_locked(n):
    global counter
    for _ in range(n):
        with lock:
            counter += 1

start = time.time()
threads = [threading.Thread(target=increment_locked, args=(100000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
elapsed = time.time() - start

print(f"Counter: {counter} (correct: {4 * 100000})")
print(f"Time with lock: {elapsed:.3f}s")
```

3. 可视化阿姆达尔定律。绘制不同并行分数的加速与处理器数量。
```python
import jax.numpy as jnp
import matplotlib.pyplot as plt

n_procs = jnp.arange(1, 65)

for p, color in [(0.5, "#e74c3c"), (0.9, "#f39c12"), (0.95, "#27ae60"), (0.99, "#3498db")]:
    speedup = 1 / ((1 - p) + p / n_procs)
    plt.plot(n_procs, speedup, color=color, linewidth=2, label=f"p={p}")
    # 最大加速线
    plt.axhline(1 / (1 - p), color=color, linestyle="--", alpha=0.3)

plt.xlabel("Number of processors")
plt.ylabel("Speedup")
plt.title("Amdahl's Law: Serial Fraction Limits Speedup")
plt.legend()
plt.grid(True)
plt.show()
```
