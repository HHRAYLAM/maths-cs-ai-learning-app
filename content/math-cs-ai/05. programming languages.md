# 编程语言

*编程语言是人类意图与机器执行之间的接口。本文件涵盖语言范式、类型系统、内存管理策略、编译流水线、解释执行与 JIT 编译、关键语言特性、领域特定语言以及设计权衡*

- 每一款软件、每一个机器学习模型、每一个操作系统都是用编程语言编写的。但是有数百种语言，每种语言都有不同的优势。为什么？因为语言设计涉及根本性的权衡：性能 vs 安全性、表达力 vs 简洁性、控制力 vs 抽象性。理解这些权衡有助于你为具体任务选择合适的工具，并理解你所面临的各种约束。

## 语言范式

- **范式（paradigm）** 是一种编程风格：一套指导你组织代码和思考问题的原则集合。

- **命令式（Imperative）** 编程将计算描述为一系列改变状态的命令。"将 x 设为 5。给 x 加 3。如果 x > 7，打印它。"C、Python 和 Java 的核心都是命令式的。其思维模型是一台拥有内存的机器，你逐步修改这些内存。

- **面向对象（OOP）** 编程围绕 **对象（objects）** 组织代码：数据（属性）和行为（方法）的捆绑体。对象通过相互发送消息来交互。关键概念包括 **封装（encapsulation）**（通过公共接口隐藏内部状态）、**继承（inheritance）**（通过扩展现有类来创建新类）和 **多态（polymorphism）**（通过共享接口统一对待不同类型）。Java、C++ 和 Python 都支持 OOP。

- **函数式编程（FP）** 将计算视为数学函数的求值。核心原则：**不可变性（immutability）**（数据一旦创建就不会改变）、**纯函数（pure functions）**（输出仅取决于输入，无副作用）和 **一等函数（first-class functions）**（函数是可以作为参数传递、从其他函数返回并存储在变量中的值）。Haskell 是纯函数式的。Python、JavaScript 和 Scala 支持函数式风格。

- 纯函数易于推理、测试和并行化（没有共享的可变状态意味着没有竞态条件）。这就是为什么函数式思想越来越多地用于分布式系统和数据管道。JAX（本书中通篇使用）是函数式的：`jax.grad` 之所以有效，是因为 JAX 函数是纯函数。

- **逻辑编程（Logic programming）** 描述 *什么* 应该为真，而非 *如何* 计算它。你陈述事实和规则，运行时环境负责寻找解决方案。Prolog 是经典示例：给定"苏格拉底是人"和"所有人都会死"，引擎推导出"苏格拉底会死"。逻辑编程用于 AI 知识库和类型检查。

- 大多数现代语言都是 **多范式（multi-paradigm）** 的：Python 支持命令式、OOP 和函数式风格。Rust 支持命令式和函数式。范式是工具，而非信仰。

## 类型系统

- **类型（type）** 对值进行分类，并确定哪些运算是有效的。整数 3 和字符串 "3" 是不同的类型：你可以对整数做加法，但不能对字符串做加法（好吧，你可以拼接字符串，但那是另一种运算）。

- **静态类型（Static typing）**：类型在 **编译时（compile time）** 检查，即程序运行之前。类型错误会被及早发现。C、Java、Rust 和 Go 是静态类型的。你必须声明类型（或者让编译器推断）：

```rust
let x: i32 = 5;     // Rust：x 是 32 位整数
let y: f64 = 3.14;  // y 是 64 位浮点数
// let z = x + y;    // 编译错误：不能相加 i32 和 f64
```

- **动态类型（Dynamic typing）**：类型在 **运行时（runtime）** 检查，即实际操作执行时。更灵活，但类型错误只在代码运行时才暴露。Python、JavaScript 和 Ruby 是动态类型的：

```python
x = 5       # x 是 int（暂时）
x = "hello" # 现在 x 是字符串 -- 没有错误
```

- **强类型（Strong typing）**：语言阻止隐式类型转换。Python 是强类型的：`"3" + 5` 会抛出 TypeError。**弱类型（Weak typing）**：语言静默地转换类型。JavaScript 是弱类型的：`"3" + 5` 得到 `"35"`（数字被强制转换为字符串）。C 是弱类型的：你可以将指针转换为整数。

- **类型推断（Type inference）** 让编译器无需显式注解即可推导类型：

```rust
let x = 5;        // 编译器推断为：i32
let y = x + 3.0;  // 编译错误：混合类型，即使有推断
```

- **泛型（Generics）**（参数化多态）让你编写适用于任何类型的代码：

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {
    let mut max = &list[0];
    for item in &list[1..] {
        if item > max { max = item; }
    }
    max
}
// 适用于整数、浮点数、字符串 -- 任何支持比较的类型
```

- 对于机器学习：Python 的动态类型使实验快速，但会隐藏 bug。生产环境的 ML 系统越来越多地使用类型注解（`def train(model: nn.Module, lr: float) -> float`）和静态分析工具（mypy）在部署前捕获错误。PyTorch 和 JAX 使用 Python 以获得灵活性；TensorRT 和 ONNX Runtime 使用 C++ 以获得性能。

## 内存管理

- 每个程序都会分配和释放内存。如何管理这一点是语言设计中最关键的决策之一。

![内存布局：栈从高地址向下增长，堆从低地址向上增长，代码和数据位于底部](../images/stack_vs_heap.svg)

- **栈（stack）** 存储局部变量和函数调用帧。分配是平凡的（移动栈指针），释放是自动的（函数返回时弹出帧）。栈访问很快，因为它始终在缓存中。但栈的大小是固定的（通常为 1-8 MB），且仅支持 LIFO（后进先出）分配。

- **堆（heap）** 存储动态分配的数据（对象、数组、编译时大小未知的字符串）。堆分配较慢（需要找到空闲块），并且需要显式或自动释放。堆可以增长到填满可用内存。

- **手动内存管理（Manual memory management）**（C、C++）：程序员显式分配（`malloc`）和释放（`free`）堆内存。具有最大的控制力和性能，但极易出错：
    - **悬空引用（Use-after-free）**：访问已释放的内存。导致崩溃或安全漏洞。
    - **重复释放（Double free）**：释放同一块内存两次。破坏分配器的内部数据结构。
    - **内存泄漏（Memory leak）**：分配内存但从不释放。程序缓慢消耗所有可用 RAM。

- **垃圾回收（Garbage collection, GC）**：运行时自动检测并释放不再可达的内存。程序员从不调用 `free`。

    - **追踪式 GC（Tracing GC）**（Java、Go、Python 的循环回收器）：定期从"根"（栈变量、全局变量）遍历所有可达对象，并释放不可达对象。简单但会导致 **GC 暂停（GC pauses）**：收集器运行时程序会停止。现代收集器（Go 的并发 GC、Java 的 ZGC）将暂停时间最小化到亚毫秒级。

    - **引用计数（Reference counting）**（Python 的主要机制、Swift、Objective-C）：每个对象跟踪有多少引用指向它。当计数降至 0 时，对象立即被释放。没有暂停，但无法处理 **循环引用（cycles）**（A 引用 B，B 引用 A，两者计数都 > 0 但都不可达）。Python 使用独立的循环检测器来处理这个问题。

- **所有权（Ownership）**（Rust）：编译器在编译时强制执行内存安全规则，零运行时开销。

    - 每个值恰好有一个 **所有者（owner）**。当所有者超出作用域时，值被丢弃（释放）。
    - 值可以被 **借用（borrowed）**（引用），但编译器强制：要么一个可变引用，要么任意数量的不可变引用，两者不能同时存在。
    - 这在编译时就防止了悬空引用、重复释放、数据竞争和悬垂指针。无需 GC，无运行时成本。

- **借用检查器（borrow checker）** 是 Rust 的杀手级特性，也是其最陡峭的学习曲线。它保证内存安全和线程安全而无需垃圾回收，这就是为什么 Rust 越来越多地用于性能关键型系统（操作系统内核、游戏引擎、ML 推理运行时如 Candle 和 Burn）。

## 编译流水线

- **编译器（compiler）** 在程序运行之前将源代码翻译成机器码（或其他目标语言）。流水线包含多个阶段：

![编译流水线：源代码 → 词法分析器 → 语法分析器 → 语义分析 → 优化器 → 代码生成 → 机器码](../images/compilation_pipeline.svg)

1. **词法分析（Lexing/tokenisation）**：将源文本转换为令牌流。`x = 3 + y` 变成 `[IDENT("x"), EQUALS, INT(3), PLUS, IDENT("y")]`。词法分析器移除空白和注释。

2. **语法分析（Parsing）**：从令牌流构建 **抽象语法树（AST）**。AST 表示程序的层次结构。`3 + y * 2` 被解析为 `Add(3, Mul(y, 2))`（乘法优先级更高）。语法分析器检查语法：不匹配的括号和缺失的分号在这里被捕获。

3. **语义分析（Semantic analysis）**：检查类型、解析变量名、验证函数调用是否正确。这是静态类型检查发生的地方。输出的是带类型标注的 AST。

4. **优化（Optimisation）**：在不改变程序行为的前提下转换程序以使其运行更快。常见优化：
    - **常量折叠（Constant folding）**：在编译时计算 `3 + 5`，将其替换为 `8`。
    - **死代码消除（Dead code elimination）**：移除永远不会执行的代码。
    - **循环展开（Loop unrolling）**：用重复的内联代码替换循环，以减少分支开销。
    - **内联（Inlining）**：用函数体替换函数调用，消除调用开销。

5. **代码生成（Code generation）**：将优化后的表示转换为目标机器码（x86、ARM）或中间表示。

- **LLVM** 是主流的编译器基础设施。它提供一种通用的中间表示（LLVM IR），许多语言都编译到此。LLVM 的优化器在此 IR 上工作，其后端为多种目标生成机器码。Clang（C/C++）、Rust、Swift、Julia 以及许多其他语言都使用 LLVM。这意味着 LLVM 优化器的改进会同时惠及所有这些语言。

## 解释执行与 JIT 编译

- **解释器（interpreter）** 逐行（或逐语句）执行程序，不生成机器码。这使得启动快速、开发具有交互性，但执行较慢（每次运行时都要重新分析每一行）。

- 大多数被解释的语言实际上会编译成 **字节码（bytecode）**：一种比源代码更简单但非机器特定的中间表示。字节码在 **虚拟机（VM）** 上运行。

    - **CPython**（标准 Python 实现）将 Python 源代码编译为字节码（`.pyc` 文件），由 CPython VM 执行。VM 逐条指令解释字节码。这就是为什么 Python 在计算密集型代码上比 C 慢约 100 倍。

    - **JVM**（Java 虚拟机）：Java 编译为 JVM 字节码（`.class` 文件）。JVM 最初解释字节码，然后将频繁执行的代码路径（"热点"）**JIT 编译** 为原生机器码。这就是为什么 Java 启动比 C 慢（解释开销），但对于长期运行的程序（JIT 优化的热点路径）速度接近 C。

- **JIT（Just-In-Time）编译** 在运行时将代码编译为机器码，利用仅在运行时可用的信息。JIT 可以基于实际运行时数据进行优化：如果一个函数总是用整数参数调用，JIT 会生成专门化的仅整数机器码，跳过类型检查。

- **PyPy** 是一种带有 JIT 编译器的替代 Python 实现。通过将热点循环 JIT 编译为机器码，它运行大多数 Python 代码比 CPython 快 5-10 倍。然而，它与 C 扩展模块（NumPy、PyTorch）的兼容性有限，这限制了它在 ML 中的使用。

- 从解释到编译的光谱不是二元的：
    - 纯解释：Bash shell 脚本。
    - 字节码解释：CPython。
    - 字节码 + JIT：JVM、.NET CLR、LuaJIT、PyPy。
    - 提前（AOT）编译：C、C++、Rust、Go。
    - AOT + 运行时代码生成：JAX 的 `jax.jit` 在第一次调用时将 Python 函数编译为优化的 XLA 代码，然后缓存编译版本。

## 关键语言特性

- **闭包（Closures）**：一个从其外围作用域捕获变量的函数。该函数"关闭"了它被定义时的环境：

```python
def make_adder(n):
    def add(x):
        return x + n  # n 从外围作用域捕获
    return add

add5 = make_adder(5)
print(add5(3))  # 8
```

- 闭包是回调、装饰器和偏应用背后的机制。它们是函数式编程的基础。

- **模式匹配（Pattern matching）**：一种强大的控制流机制，可解构数据并根据其形状进行分支：

```rust
match value {
    Some(x) if x > 0 => println!("正数：{}", x),
    Some(0)           => println!("零"),
    Some(x)           => println!("负数：{}", x),
    None              => println!("无"),
}
```

- 模式匹配比 if-else 链更具表达力：它检查数据的结构（是 Some 还是 None？是否包含匹配条件的值？），而不仅仅是相等性。Python 在 3.10 中添加了结构化模式匹配（`match`/`case`）。

- **代数数据类型（ADTs）**：可以是几种变体之一的类型，每种变体携带不同的数据。`Result` 类型要么是 `Ok(value)`，要么是 `Err(error)`。`Tree` 要么是 `Leaf(value)`，要么是 `Node(left, right)`。ADTs 与模式匹配结合使用，可以穷尽地处理所有情况，消除整类 bug（空指针异常、未处理的错误码）。

- **特征和接口（Traits and interfaces）**：定义类型必须实现的一组方法，而不指定如何实现。这实现了多态：一个接受"任何实现 Display 特征的东西"的函数适用于整数、字符串和自定义类型。Rust 使用特征，Java 使用接口，Go 使用隐式接口，Python 使用鸭子类型（"如果它走起来像鸭子..."）。

## 领域特定语言

- **领域特定语言（DSL）** 是为特定问题领域设计的，在该领域内牺牲通用性以换取表达力。

- **SQL**：关系型数据库的语言。`SELECT name FROM users WHERE age > 30` 比等价的命令式循环易读且可优化得多。数据库引擎优化查询执行计划，自动选择连接策略和索引使用。

- **正则表达式（Regular expressions）**：用于文本模式匹配的微型语言。`\d{3}-\d{4}` 匹配像 "555-1234" 这样的电话号码。正则引擎将模式编译为有限自动机以进行高效匹配。

- **着色器语言（Shader languages）**（GLSL、HLSL、Metal Shading Language）：在 GPU 核心上运行的程序，用于计算像素颜色、顶点位置或计算操作。着色器是大规模并行的：每个调用独立处理一个像素或一个元素。这与 CUDA 用于 ML 计算的执行模型相同。

- 在 ML 中，像 PyTorch 和 JAX 这样的框架本质上是嵌入在 Python 中的张量计算 DSL。它们提供领域特定的抽象（张量、自动微分、设备放置），同时利用 Python 的生态系统。

## 语言设计权衡

- 没有语言在所有方面都是最好的。设计就是关于选择进行哪些权衡：

- **性能 vs 安全性**：C 提供原始速度和硬件控制，但允许你破坏内存。Rust 提供相当的速度，并具有编译时内存安全性。Java 提供内存安全性，但有垃圾回收开销。Python 提供最大的安全性和表达力，但执行速度慢 100 倍。

- **表达力 vs 简洁性**：Haskell 的类型系统可以表达非常精确的约束，但学习曲线陡峭。Go 刻意省略泛型（直到最近）、继承和异常以保持简洁。Python 的"应该有一种明显的方式来做"的哲学使语言易于学习。

- **控制力 vs 抽象性**：C/C++ 让你控制内存布局、缓存行为和硬件交互。Python 隐藏了所有这些。对于 ML 训练（GPU 计算占主导），Python 的开销可以忽略不计。对于 ML 推理（每微秒都很重要），可能需要 C++ 或 Rust。

- **编译速度 vs 运行时速度**：Go 在几秒内编译完成（简单的类型系统，最少的优化）。Rust 编译需要几分钟（复杂的类型系统，激进的优化）。权衡在于开发者迭代速度与部署后的性能。

- ML 生态系统反映了这些权衡：Python 用于实验和训练（表达力获胜），C++/CUDA 用于内核和推理（性能获胜），Rust 用于基础设施和安全关键系统（安全性获胜）。

```python
def make_multiplier(factor):
    """返回一个乘以 factor 的函数。"""
    def multiply(x):
        return x * factor
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

print(f"double(5) = {double(5)}")  # 10
print(f"triple(5) = {triple(5)}")  # 15

# 闭包通过引用捕获，而非通过值
def make_counter():
    count = [0]  # 可变容器以允许修改
    def increment():
        count[0] += 1
        return count[0]
    return increment

counter = make_counter()
print(f"counter() = {counter()}")  # 1
print(f"counter() = {counter()}")  # 2
print(f"counter() = {counter()}")  # 3
```

2. 比较动态类型与静态类型行为。展示 Python 的动态类型如何提供灵活性但可能隐藏 bug。
```python
def add(a, b):
    return a + b

# 适用于不同类型 -- 灵活！
print(add(3, 5))           # 8 (int + int)
print(add("hello ", "world"))  # "hello world" (str + str)
print(add([1, 2], [3, 4]))    # [1, 2, 3, 4] (list + list)

```

3. 对于计算密集型任务，测量解释型 Python 与编译/JIT 方法之间的性能差异。
```python
import time
import jax
import jax.numpy as jnp

n = 1_000_000

# 纯 Python 循环（解释型）
start = time.time()
total = 0.0
for i in range(n):
    total += i * i
python_time = time.time() - start

# JAX（通过 XLA 编译）
@jax.jit
def sum_squares_jax(n):
    return jnp.sum(jnp.arange(n, dtype=jnp.float32) ** 2)

_ = sum_squares_jax(10)  # 预热 JIT
start = time.time()
result = sum_squares_jax(n)
jax_time = time.time() - start

print(f"Python 循环：{python_time:.4f}s")
print(f"JAX (JIT):   {jax_time:.6f}s")
print(f"加速比：     {python_time / jax_time:.0f}x")
```
