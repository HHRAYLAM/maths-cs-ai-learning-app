# 服务与批处理

*服务 LLM 给数千并发用户需要的不仅仅是加载模型和运行推理。本文件涵盖预填充 - 解码分离、连续批处理、PagedAttention 和 vLLM、调度策略、分离式服务、多模型和 LoRA 服务，以及重要的指标*

- 单个 LLM 推理请求很简单：输入标记，生成输出标记。但服务 LLM 给 10,000 并发用户，低延迟高吞吐量是系统问题。朴素方法（一次处理一个请求）浪费 90%+ GPU 容量。智能批处理和调度可以增加吞吐量 10-50 倍，无需添加硬件。

## 预填充 vs 解码：两个非常不同的阶段

- LLM 推理有两个截然不同的阶段，根本不同的计算特征：

- 预填充（提示处理）：同时处理所有输入标记。这是单个大矩阵乘法：$O(\text{prompt\_length} \times d_{\text{model}}^2)$。提示可以并行处理（所有标记已知）。预填充是计算绑定：GPU 的 ALU 是瓶颈。

- 解码（标记生成）：一次生成一个标记，自回归。每个新标记需要通过 KV 缓存关注所有先前的标记。解码是内存带宽绑定：GPU 大部分时间从内存加载模型权重和 KV 缓存，而不是计算。每个解码步骤只产生一个标记，但必须加载整个模型（FP16 的 70B 模型约 140 GB）。

- 含义：

| | 预填充 | 解码 |
|--|---------|--------|
| 处理的标记 | 一次全部（并行） | 一次一个（顺序） |
| 瓶颈 | 计算 (FLOPS) | 内存带宽 |
| 算术强度 | 高 | 非常低 |
| GPU 利用率 | 高 (50-80%) | 低 (1-10%) 无批处理 |
| 延迟指标 | 首标记时间（TTFT） | 每输出标记时间（TPOT） |

- TTFT 对用户体验重要（响应开始流式传输前多久）。TPOT 决定感知生成速度。用户容忍更高 TTFT（1-5 秒），但期望快 TPOT（对话应用每标记 30-100 毫秒）。

## 静态批处理（朴素）

- 最简单的批处理：收集 $B$ 个请求，将它们填充到相同长度，作为单个批处理处理它们。

- 问题 1：请求有不同的提示长度和生成不同数量的输出标记。短请求早完成，但必须等待批处理中最长请求，然后才能开始下一批。GPU 空闲等待剩余长请求的生成。

- 问题 2：填充浪费计算。如果最长提示是 2000 标记，最短是 50，批处理填充到 2000。GPU 为短请求处理 1950 个填充标记——纯浪费。

![静态批处理浪费 GPU 插槽等待最长请求；连续批处理立即填充空闲插槽](../images/static_vs_continuous_batching.svg)

## 连续批处理

- 连续批处理（也称为迭代级批处理）通过在单个解码步骤的粒度操作，而不是整个请求，解决这两个问题。

- 在每个解码步骤：
    1. 所有飞行中的请求并行生成一个标记（作为批处理）。
    2. 完成的请求（生成 EOS 标记）立即移除出批处理。
    3. 来自队列的新请求立即插入到空闲插槽。

- 批处理大小每步动态变化。GPU 从不空闲等待落后者，没有浪费填充（每个请求只使用它需要的插槽）。

- 影响：连续批处理通常比静态批处理增加吞吐量 2-10 倍，无模型质量变化或显著延迟增加。

## PagedAttention 和 vLLM

- KV 缓存创造内存管理噩梦。每个请求有 KV 缓存，随每个生成的标记增长。不同请求处于不同阶段（不同缓存大小）。为每个请求分配连续内存浪费空间（你必须为最大可能长度分配，即使请求只生成几个标记）。

![PagedAttention 将虚拟 KV 缓存页面映射到非连续物理 GPU 内存，消除碎片，实现按需分配](../images/paged_attention.svg)

- PagedAttention（Kwon 等人，2023）将操作系统的虚拟内存概念（第 13 章）应用于 KV 缓存。缓存划分为固定大小页面（标记位置块）。页面按需分配，在物理 GPU 内存中可以非连续。

- 好处：
    - 无碎片：页面统一大小，所以请求之间没有"空洞"浪费内存。
    - 惰性分配：内存仅在标记实际生成时分配，不为最大长度预分配。
    - 写时复制：共享公共前缀的请求（如系统提示）共享相同的 KV 缓存页面。只有当请求分歧时，页面才复制。

- vLLM 是围绕 PagedAttention 构建的推理引擎。它实现比静态分配服务（如 HuggingFace 的 text-generation-inference，无分页注意力）高 2-4 倍吞吐量，通过实际上消除 KV 缓存内存浪费。

## 调度策略

- 当多个请求等待且 GPU 只能处理有限批处理时，调度决定服务哪些请求：

- 先来先服务（FCFS）：按到达顺序处理请求。简单但不公平：用户提交 10K 标记生成阻塞后面所有用户。

- 短作业优先（SJF）：处理将最快完成的请求。最小化平均延迟但惩罚长运行请求（它们可能饿死）。实践中，估计输出长度未知，所以 SJF 使用启发式（提示长度、用户历史）。

- 抢占：如果高优先级请求到达，暂停低优先级进行中的请求（将其 KV 缓存交换到 CPU 内存或 SSD），服务高优先级请求，然后恢复暂停的请求。vLLM 支持这个。

- 基于优先级：为用户或请求类型分配优先级。实时交互式查询获得比批处理作业更高优先级。与抢占结合，这确保高优先级流量的延迟 SLO。

- 标记预算：限制活动批处理中的总标记数。这防止几个长请求垄断 GPU 内存，饿死新请求。

## 分离式服务

- 预填充和解码有相反的计算特征。在同一个 GPU 上运行两者意味着 GPU 在计算绑定（预填充）和内存带宽绑定（解码）之间交替，从不充分利用任一资源。

- 分离式服务分离它们：
    - 预填充节点：针对计算优化的 GPU（高 FLOPS，可能较少内存）。处理所有传入提示。
    - 解码节点：针对内存带宽优化的 GPU（大 KV 缓存容量，高内存带宽）。处理所有标记生成。

- 预填充节点计算初始 KV 缓存，发送给解码节点（通过 NVLink 或网络）。解码节点使用接收的缓存生成标记。

- 这是Mooncake（Moonshot AI）的架构，几个 LLM 服务团队正在探索。好处：每个 GPU 类型匹配其工作负载特征，提高整体利用率。

## 多模型和 LoRA 服务

- 在生产中，你经常服务多个模型（不同大小用于不同层级，不同微调变体用于不同任务）。

- 模型复用：在同一个 GPU 上加载多个模型，路由请求到适当的模型。GPU 内存共享：40 GB GPU 可能同时容纳 13B 模型（26 GB）和 7B 模型（14 GB）。

- LoRA 服务：不是部署单独的微调模型，而是部署一个基础模型，带多个LoRA 适配器（第 6 章）。每个适配器添加<1% 参数。推理时请求路由到适当的适配器。

- S-LoRA（Sheng 等人，2023）：从单个基础模型服务数千个 LoRA 适配器。适配器存储在 CPU 上，按需分页到 GPU 内存。基础模型的 KV 缓存和权重共享；只有小的 LoRA 矩阵每个请求不同。

- Punica（Chen 等人，2023）：跨不同 LoRA 适配器批处理请求，使用自定义 CUDA 内核，在同一个批处理内为不同请求应用不同 LoRA 矩阵。这避免每个请求切换适配器的开销。

## 约束和引导生成

- 许多应用需要 LLM 以特定格式产生输出：有效 JSON、SQL 查询、特定语言的代码，或遵循模式的响应。约束生成保证输出符合语法或模式。

- 语法约束解码：在每个解码步骤，掩码违反语法的标记。如果输出到目前为止是 `{"name": "Alice", "age":`，语法要求下一个是整数，掩码除数字外的所有标记。LLM 的概率分布在有效标记上重新归一化。

- Outlines（Willard & Louf，2023）：将 JSON 模式或正则表达式编译为有限状态机（FSM）。在每个解码步骤，FSM 确定哪些标记是有效延续。无效标记获得概率 0。这保证 100% 模式合规，零重试。

- SGLang 原生集成约束生成：你在 Python 中指定输出结构，引擎高效处理标记掩码和缓存。这与 RadixAttention（前缀缓存）结合，所以结构化输出重用缓存前缀。

- 为什么重要：没有约束生成，你自由生成并解析输出，失败时重试。复杂 JSON 模式的重试率 10-30% 常见，浪费计算。约束生成完全消除重试。

## 请求路由

- 不是每个查询都需要最大模型。请求路由根据估计难度将查询定向到不同模型：

- 级联：先尝试小模型。如果小模型的置信度低于阈值（例如，顶部标记的 softmax 概率<0.8），升级到更大模型。简单查询（80%+ 流量）由小模型便宜服务；只有难查询使用昂贵模型。

- 学习路由：训练轻量级分类器（或使用小模型的困惑度）预测查询需要哪个模型层级。路由"2+2 是多少？"到 3B 模型，"解释量子纠缠的数学基础"到 70B 模型。

- 影响：如果 80% 查询可以由成本低 10 倍的模型处理，每查询平均成本下降约 70%。这是多模型部署中影响最大的成本优化之一。

- 设备 + 云混合路由：Cactus（[github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)）在设备级实现请求路由。它通过自定义 ARM SIMD 内核在设备上运行小模型（手机、笔记本电脑、可穿戴设备），当本地模型的置信度低或查询超过设备能力时，自动路由到云模型。应用对两条路径使用 OpenAI 兼容 API——路由是透明的。这是基础设施级的级联：第一层免费（设备上），第二层花钱（云 API）。对于简单查询为主的应用（助手问答、自动完成、转录），设备处理覆盖 70-90% 流量，零边际成本。

## 推理指标

- 正确的指标取决于用例：

| 指标 | 测量什么 | 目标（对话） | 目标（批处理） |
|--------|-----------------|------------------------|-----------------|
| TTFT | 首标记时间 | <1 秒 | 不重要 |
| TPOT | 每输出标记时间 | <100 毫秒 | 不重要 |
| 吞吐量 | 标记/秒（总计） | 不重要 | 最大化 |
| p99 延迟 | 最慢 1% 请求 | <5 秒 | <30 秒 |
| 每标记成本 | $/100 万标记 | 最小化 | 最小化 |
| SLO 合规 | 满足延迟目标的请求% | >99% | >95% |

- TTFT vs TPOT 权衡：激进批处理增加吞吐量（总标记/秒更多）但增加 TPOT（每个标记花更长时间，因为 GPU 处理更多请求）。调度策略必须平衡吞吐量（收入）与延迟（用户体验）。

- 每标记成本是生产的最终指标。它结合硬件成本（GPU 租赁）、吞吐量（标记/秒）和利用率。50% GPU 利用率的系统每标记成本是 100% 的 2 倍。这就是为什么批处理、调度和 PagedAttention 如此重要——它们提高利用率。

1. 模拟连续 vs 静态批处理，测量吞吐量差异。
```python
import random
import time

def simulate_static_batching(requests, batch_size=8):
    """固定批处理处理请求。等待所有完成。"""
    total_tokens = 0
    total_time = 0

    for i in range(0, len(requests), batch_size):
        batch = requests[i:i + batch_size]
        max_len = max(r['output_len'] for r in batch)
        # 批处理中的所有请求花最长时间
        batch_time = max_len * 0.01  # 每标记 10ms
        total_time += batch_time
        total_tokens += sum(r['output_len'] for r in batch)

    return total_tokens / total_time  # 每秒标记

def simulate_continuous_batching(requests, max_batch=8):
    """连续批处理。移除完成的，添加新的。"""
    total_tokens = 0
    total_time = 0
    active = []
    queue = list(requests)

    while active or queue:
        # 填充批处理
        while len(active) < max_batch and queue:
            active.append({'remaining': queue.pop(0)['output_len']})

        if not active:
            break

        # 一个解码步骤：所有活动请求生成 1 个标记
        for req in active:
            req['remaining'] -= 1
        total_tokens += len(active)
        total_time += 0.01  # 每步 10ms

        # 移除完成的请求
        active = [r for r in active if r['remaining'] > 0]

    return total_tokens / total_time

# 生成不同输出长度的请求
random.seed(42)
requests = [{'output_len': random.randint(10, 500)} for _ in range(100)]

static_tps = simulate_static_batching(requests)
continuous_tps = simulate_continuous_batching(requests)

print(f"静态批处理：     {static_tps:.0f} 标记/秒")
print(f"连续批处理： {continuous_tps:.0f} 标记/秒")
print(f"加速：{continuous_tps / static_tps:.1f}x")
```

2. 计算 PagedAttention 的 KV 缓存内存节省。比较预分配（最坏情况）vs 分页（实际使用）。
```python
def paged_vs_preallocated(n_requests, max_seq_len, avg_seq_len, page_size, kv_per_token_bytes):
    """比较内存使用：预分配 vs 分页 KV 缓存。"""
    # 预分配：每个请求获得 max_seq_len 插槽
    preallocated_gb = n_requests * max_seq_len * kv_per_token_bytes / 1e9

    # 分页：只分配使用的（带页面粒度）
    import math
    avg_pages = math.ceil(avg_seq_len / page_size)
    paged_gb = n_requests * avg_pages * page_size * kv_per_token_bytes / 1e9

    waste_preallocated = (max_seq_len - avg_seq_len) / max_seq_len
    waste_paged = (avg_pages * page_size - avg_seq_len) / (avg_pages * page_size)

    print(f"请求：{n_requests}, 最大序列：{max_seq_len}, 平均序列：{avg_seq_len}")
    print(f"  预分配：{preallocated_gb:.1f} GB (浪费：{waste_preallocated:.0%})")
    print(f"  分页：       {paged_gb:.1f} GB (浪费：{waste_paged:.0%})")
    print(f"  节省：      {preallocated_gb - paged_gb:.1f} GB ({preallocated_gb/paged_gb:.1f}x)")
    print()

# Llama-70B: 每层约 1.3 KB 每标记，80 层 = 约 100 KB 每标记总计
kv_bytes = 100_000

# 场景 1：短请求，大最大
paged_vs_preallocated(256, max_seq_len=4096, avg_seq_len=256, page_size=16, kv_per_token_bytes=kv_bytes)

# 场景 2：不同长度
paged_vs_preallocated(256, max_seq_len=8192, avg_seq_len=1024, page_size=16, kv_per_token_bytes=kv_bytes)

# 场景 3：长上下文
paged_vs_preallocated(64, max_seq_len=131072, avg_seq_len=16000, page_size=16, kv_per_token_bytes=kv_bytes)
```
