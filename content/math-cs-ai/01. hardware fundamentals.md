# 硬件基础

*在编写 SIMD 或 GPU 代码之前，你需要了解你正在编程的硬件。本文件涵盖为什么并行性取代了时钟速度、现代 CPU 如何执行指令、什么是 SIMD、用于推理性能的理论模型，以及芯片架构的前景*

- 几十年来，软件免费变快：购买具有更高时钟速度的新 CPU，你的程序无需更改一行代码就能运行得更快。那个时代在 2005 年左右结束。理解为什么结束，以及什么取代了它，对于任何想编写快速代码的人来说都是至关重要的。

## 免费性能的终结

- **摩尔定律**（1965）观察到芯片上的晶体管数量大约每两年翻一番。这持续了 60 年。更多的晶体管意味着更小的晶体管，意味着更高的时钟速度，意味着更快的程序。

- 但在 2005 年左右，时钟速度在约 4 GHz 时遇到了墙。问题是**功耗**。芯片消耗的功率近似为：

$$P \propto C \cdot V^2 \cdot f$$

- 其中 $C$ 是电容（与晶体管数量成正比），$V$ 是电压，$f$ 是时钟频率。要提高频率，你必须增加电压（以更快地切换晶体管）。但功率与 $V^2 \cdot f$ 成比例，所以频率的小幅增加会导致功率（和热量）的大幅增加。在 4 GHz 时，芯片已经达到 100+ 瓦特。达到 8 GHz 需要不切实际的冷却。

- 解决方案：不是让一个核心更快，而是在同一芯片上放置**多个核心**。4 核心芯片在 3 GHz 下使用与单核心在 4.5 GHz 下相似的功率，但可以做 4 倍的并行工作。这就是为什么每个现代 CPU 都有多个核心，以及为什么并行性（SIMD、多线程、GPU 计算）是获得更多性能的唯一途径。

- **对 ML 的影响**：在一个核心上需要 10 分钟的训练步骤不能通过购买更快的 CPU 来加速。它只能通过使用更多核心（数据并行，第 6 章）、更宽的 SIMD 单元（本章）或 GPU（数千个核心）来加速。

## 现代 CPU 如何执行指令

- 现代 CPU 核心远比第 13 章中的简单取指 - 译码 - 执行模型复杂。它使用几种技巧来每个周期执行更多指令：

- **超标量执行**：CPU 有多个执行单元（ALU、FPU、加载/存储单元），可以同时执行多条独立指令。如果现代核心不相互依赖，每个周期可以执行 4-6 条指令。

- **乱序执行**（OoO）：CPU 不按程序顺序执行指令。它在指令流中向前看，找到输入就绪的指令，并立即执行它们，不管它们的位置。这隐藏了延迟：当一条指令等待来自内存的数据（100+ 周期）时，CPU 执行其他就绪的指令。

- **分支预测**：条件分支（`if` 语句、循环条件）产生不确定性：CPU 在评估条件之前不知道走哪条路径。CPU 不是停滞，而是**预测**结果并推测性地沿预测路径执行。如果预测正确（现代预测器超过 95% 的时间），不会损失时间。如果错误，推测性工作被丢弃并执行正确路径（约 15 周期惩罚）。

- **推测执行**：分支预测的扩展。CPU 执行*可能*不需要的指令，赌它们会被需要。这填充了流水线并保持执行单元忙碌。

- 所有这些都是自动的——CPU 无需任何程序员干预就能做到这一点。但它们只对**指令级并行性**（ILP）有帮助：单一流中的独立指令。对于**数据级并行性**（对多个数据元素执行相同操作），我们需要 SIMD。

## SIMD：单指令多数据

- **SIMD** 是同时对多个数据元素应用一条指令的思想。不是添加两个数字，而是在单条指令中添加两个 4（或 8、16）个数字的向量。

- 没有 SIMD（标量）：

```cpp
// 逐个元素添加两个数组：4 条加法指令
for (int i = 0; i < 4; i++) {
    c[i] = a[i] + b[i];  // 每次迭代一次加法
}
```

- 使用 SIMD（向量化）：

```cpp
// 添加两个数组：1 条 SIMD 指令完成所有 4 次加法
#include <immintrin.h>  // x86 SIMD 内部函数

__m128 va = _mm_load_ps(a);    // 加载 4 个浮点到 128 位寄存器
__m128 vb = _mm_load_ps(b);    // 加载 4 个浮点到另一个寄存器
__m128 vc = _mm_add_ps(va, vb); // 同时添加所有 4 对
_mm_store_ps(c, vc);            // 存储 4 个结果
```

- SIMD 版本用 1/4 的指令完成相同的工作。这是理论上的 4 倍加速，通过每条指令处理 4 个浮点数而不是 1 个实现。

### 向量寄存器

- SIMD 指令在**向量寄存器**上操作：保存多个数据元素的宽寄存器。

| 寄存器宽度 | 浮点数 (32 位) | 双精度 (64 位) | 名称 |
|----------------|-----------------|-------------------|------|
| 128 位 | 4 | 2 | SSE (x86), NEON (ARM) |
| 256 位 | 8 | 4 | AVX/AVX2 (x86) |
| 512 位 | 16 | 8 | AVX-512 (x86) |
| 可变 (128-2048) | 变化 | 变化 | SVE/SVE2 (ARM) |

- 更宽的寄存器 = 更多并行性。512 位 AVX-512 指令一次处理 16 个浮点数，理论上比标量代码快 16 倍。在实践中，由于内存带宽限制（你计算的速度比向 CPU 提供数据的速度快），加速较低。

- 对于 ML：float32 值的矩阵乘法从 SIMD 中受益匪浅。内循环（两个向量的点积）直接映射到 SIMD 乘 - 累加指令。这就是为什么 BLAS 库（NumPy 和 PyTorch 调用的）如此 heavily 优化 SIMD。

## 理论模型

- 你如何知道你的代码是否快？**理论模型**提供了一个框架，通过表征两个硬件限制的性能：

1. **峰值计算**（FLOPS）：每秒最大浮点运算次数。对于 4 GHz CPU，具有 256 位 AVX（每条指令 8 个浮点数）和 2 个 FMA 单元：$4 \times 10^9 \times 8 \times 2 = 64$ GFLOPS。

2. **峰值内存带宽**（字节/秒）：数据从内存移动到 CPU 的速度。现代 CPU 可能有 50 GB/s 的内存带宽。

- 代码的**算术强度**是计算与内存访问的比率：

$$\text{算术强度} = \frac{\text{FLOPS}}{\text{传输字节数}}$$

- 如果算术强度低（每个加载字节的操作少），你的代码是**内存受限**的：它大部分时间都在等待数据。使计算更快（更宽的 SIMD、更高的时钟）不会有帮助。

- 如果算术强度高（每个字节许多操作），你的代码是**计算受限**的：它大部分时间都在计算。更快的内存不会有帮助。

- 理论模型：

$$\text{可达 FLOPS} = \min\left(\text{峰值 FLOPS}, \; \text{带宽} \times \text{算术强度}\right)$$

- **矩阵乘法**具有高算术强度：$O(n^3)$ 操作对 $O(n^2)$ 数据，所以强度 $\approx O(n)$。对于大矩阵，它是计算受限的。这就是为什么 GPU（高计算）主导矩阵重度 ML 工作负载。

- **逐元素操作**（ReLU、加法、乘法）具有低算术强度：每个加载元素 1 个操作。这些是内存受限的。使 GPU 更快没有帮助；你需要更快的内存（或将这些操作与计算密集型操作融合以避免单独的内存往返）。

- 理论模型解释了为什么**内核融合**如此重要：将矩阵乘法与偏置加法和 ReLU 组合到单个内核中，避免将中间结果写入内存并读回，将三个内存受限操作变成一个计算受限操作。

## 延迟与吞吐量

- **延迟**是完成一个操作的时间。**吞吐量**是单位时间内完成的操作数。

- 一个类比：公交车具有高延迟（在每站等待）但高吞吐量（一次载 50 人）。出租车具有低延迟（直接去你的目的地）但低吞吐量（载 1-4 人）。

- GPU 是公交车：每个操作的高延迟（每条指令需要许多周期完成）但巨大的吞吐量（数千个核心同时处理）。CPU 是出租车：低延迟（乱序执行、分支预测、深缓存最小化延迟）但有限的吞吐量（4-64 个核心）。

- 这就是为什么 GPU 更适合 ML 训练（吞吐量重要：处理数百万个示例），而 CPU 更适合 OS 任务（延迟重要：立即响应按键）。

- **流水线**将延迟转换为吞吐量。如果指令需要 5 个周期但流水线每个周期启动新指令，吞吐量是每个周期 1 条指令（即使每条需要 5 个周期完成）。这与第 13 章中的 CPU 流水线原理相同，但它适用于每个级别：SIMD 单元、内存控制器和 GPU 核心都是流水线的。

## 芯片架构前景

- 你编写代码的硬件决定了哪些 SIMD 指令可用：

### x86（Intel、AMD）

- 主导台式机、笔记本电脑和数据中心 CPU。SIMD：SSE（128 位）、AVX/AVX2（256 位）、AVX-512（512 位）。Intel AMX 为 AI 工作负载提供专用矩阵乘法单元。

- **优势**：最高单核性能、最宽 SIMD、成熟的软件生态系统（MKL、oneDNN）。
- **劣势**：高功耗、复杂指令集、昂贵。

### ARM

- 主导移动设备（每部智能手机），在服务器（AWS Graviton、Ampere Altra）和笔记本电脑（Apple M 系列）中增长。SIMD：NEON（128 位）、SVE/SVE2（可扩展，128-2048 位）。

- **优势**：出色的功耗效率（每瓦性能）、定制核心（Apple M4 在单核性能上与 Intel 相当，但功耗只有一小部分）。
- **劣势**：较窄 SIMD（NEON 只有 128 位，尽管 SVE 可以更宽）、HPC 软件生态系统较小。

### Apple Silicon（M1/M2/M3/M4）

- 基于 ARM，具有定制补充。包括**AMX**（Apple 矩阵扩展）——未文档化的矩阵乘法单元，Accelerate 框架用于 BLAS 操作。统一内存架构：CPU 和 GPU 共享相同的物理内存，消除 CPU↔GPU 复制瓶颈。

- **对于 ML**：Apple 的神 engine（16 核，专用 ML 加速器）和统一内存使 M 系列芯片对于本地 ML 推理和小规模训练令人惊讶地有能力。没有 CUDA， though：你必须使用 Metal（Apple 的 GPU API）或 MLX（Apple 的 ML 框架）。

### RISC-V

- 开源 ISA。无许可费（与 ARM 不同）。在嵌入式系统、IoT 和研究中增长。SIMD："V"（向量）扩展提供类似于 ARM SVE 的可扩展向量处理。

- **对于 ML**：尚未与 x86/ARM 竞争 ML 工作负载，但关注这个空间。几家 AI 加速器初创公司使用 RISC-V 核心。

### GPU（NVIDIA、AMD、Intel）

- 在文件 04-05 中深入涵盖。数千个简单核心为吞吐量优化。NVIDIA 与 CUDA 主导 ML；AMD 与 ROCm 竞争；Intel 与 Arc GPU 和 Gaudi 加速器进入。

### TPU（Google）

- 专门为 ML 设计的定制 ASIC。脉动阵列优化用于矩阵乘法。在文件 05 中涵盖。

## 散热和功耗限制

- 性能最终受功耗和冷却限制：

- **TDP**（热设计功耗）：芯片可以消耗的最大持续功率。笔记本电脑 CPU 可能有 15W TDP；服务器 CPU 250W；数据中心 GPU 700W（NVIDIA B200）。

- **暗硅**：在任何给定时刻，晶体管的相当大部分必须关闭以保持在热预算内。芯片理论上可以同时使用所有晶体管，但它会熔化。

- **功率效率**（FLOPS/瓦特）越来越成为重要的指标，而不是原始 FLOPS。这就是为什么：
    - ARM 正在接管数据中心（比 x86 更好的 FLOPS/瓦特）。
    - TPU 与 GPU 竞争，尽管峰值 FLOPS 较低（对 ML 工作负载好得多的 FLOPS/瓦特）。
    - 量化（INT8、FP8）不仅是关于内存：它还减少每个操作的功耗。

- 对于大规模 ML：训练前沿 LLM 消耗数兆瓦电力数月。电力成本可能超过硬件成本。功率效率直接影响 AI 研究的经济学。

## 实践：在 C++ 中测量性能

- 要推理性能，你需要测量它。这里是一个最小 C++ 基准测试设置：

```cpp
#include <iostream>
#include <chrono>
#include <vector>

// 标量加法
void add_scalar(const float* a, const float* b, float* c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}

int main() {
    const int N = 1 << 24;  // 约 1600 万元素
    std::vector<float> a(N, 1.0f), b(N, 2.0f), c(N);

    // 预热（填充缓存，触发频率缩放）
    add_scalar(a.data(), b.data(), c.data(), N);

    // 基准测试
    auto start = std::chrono::high_resolution_clock::now();

    for (int trial = 0; trial < 100; trial++) {
        add_scalar(a.data(), b.data(), c.data(), N);
    }

    auto end = std::chrono::high_resolution_clock::now();
    double elapsed = std::chrono::duration<double>(end - start).count();

    double total_bytes = 3.0 * N * sizeof(float) * 100;  // 读 a，读 b，写 c
    double bandwidth = total_bytes / elapsed / 1e9;        // GB/s

    std::cout << "时间：" << elapsed << " 秒\n";
    std::cout << "带宽：" << bandwidth << " GB/s\n";

    return 0;
}
```

```bash
# 使用优化编译
g++ -O3 -march=native -o bench bench.cpp
./bench
```

- **此代码中的关键 C++ 概念**：
    - `#include <vector>`：动态数组（`std::vector<float>`）——类似于 Python 的 `list`，但有类型且在内存中连续。
    - `a.data()`：返回底层数组的原始指针（`float*`）——SIMD 内部函数需要。
    - `std::chrono`：用于基准测试的高分辨率定时器。
    - `-O3`：最大编译器优化级别。编译器可能自动向量化你的循环（自动使用 SIMD）。`-march=native` 启用你的 CPU 支持的所有 SIMD 指令。

- **为什么预热**：第一次运行填充缓存并可能触发 CPU 频率缩放（turbo boost）。后续运行更具代表性。

- **为什么测量带宽**：对于内存受限操作（如逐元素加法），有意义的指标是带宽（GB/s），而不是 FLOPS。如果你测量的带宽接近硬件限制（DDR5 约 50 GB/s），你是内存受限的，SIMD 不会有多大帮助（瓶颈是内存，而不是计算）。

1. 计算常见 ML 操作的算术强度，并将它们分类为内存受限或计算受限。
```python
import jax.numpy as jnp

def arithmetic_intensity(flops, bytes_transferred):
    return flops / bytes_transferred

# 逐元素 ReLU：每个元素 1 次比较，读 + 写
n = 1024
relu_flops = n  # 每个元素 1 次操作
relu_bytes = 2 * n * 4  # 读输入 + 写输出 (float32)
print(f"ReLU: {arithmetic_intensity(relu_flops, relu_bytes):.2f} FLOPS/字节 → 内存受限")

# 矩阵乘法：2*n^3 次操作，读 2*n^2 + 写 n^2 个浮点数
matmul_flops = 2 * n**3
matmul_bytes = 3 * n**2 * 4  # 读 A + 读 B + 写 C
print(f"矩阵乘法 ({n}×{n}): {arithmetic_intensity(matmul_flops, matmul_bytes):.0f} FLOPS/字节 → 计算受限")

# 层归一化：约 5n 次操作（均值、方差、归一化），读 + 写
ln_flops = 5 * n
ln_bytes = 2 * n * 4
print(f"LayerNorm: {arithmetic_intensity(ln_flops, ln_bytes):.2f} FLOPS/字节 → 内存受限")

# 卷积 3x3: 2*9*C_in*C_out*H*W, 读核 + 特征图 + 写输出
C_in, C_out, H, W = 64, 128, 32, 32
conv_flops = 2 * 9 * C_in * C_out * H * W
conv_bytes = (9 * C_in * C_out + C_in * H * W + C_out * H * W) * 4
print(f"Conv3x3: {arithmetic_intensity(conv_flops, conv_bytes):.0f} FLOPS/字节 → 计算受限")
```

2. 演示为什么并行性重要。随着数据大小增长，比较顺序与并行（NumPy）执行。
```python
import numpy as np
import time

for n in [1000, 10000, 100000, 1000000, 10000000]:
    a = np.random.randn(n).astype(np.float32)
    b = np.random.randn(n).astype(np.float32)

    # "顺序"（Python 循环）
    start = time.time()
    c = [a[i] * b[i] for i in range(min(n, 100000))]  # 限制在 100K 用于合理性
    seq_time = time.time() - start
    if n > 100000:
        seq_time *= n / 100000  # 外推

    # "并行"（NumPy，内部使用 SIMD + 多线程）
    start = time.time()
    c = a * b
    par_time = time.time() - start

    print(f"n={n:>10,}  顺序={seq_time:.4f}秒  并行={par_time:.6f}秒  "
          f"加速={seq_time/par_time:.0f}x")
```
