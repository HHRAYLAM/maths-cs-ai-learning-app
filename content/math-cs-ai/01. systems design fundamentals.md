# 系统设计基础

*系统设计是如何构建在大规模下可靠工作的软件。本文件涵盖客户端 - 服务器架构、网络协议、DNS、代理、负载均衡、缓存、数据库、消息队列、一致性模型和弹性模式*

- 生产中的每个 ML 系统都是分布式系统。推荐引擎不仅仅是模型 —— 它是 API 服务器、特征存储、模型注册表、缓存层、消息队列和监控栈，全部通过网络通信。理解系统设计是将"我训练了一个模型"与"我构建了一个产品"区分开来的关键。

- 顶级科技公司（Google、Meta、Amazon、OpenAI）的系统设计面试测试你能否设计这些系统。本章为你提供构建块（本文件）、云基础设施（文件 02）、扩展模式（文件 03）、ML 特定设计（文件 04）和工作示例（文件 05）。

## 客户端 - 服务器架构

- 基本模式：**客户端**发送请求，**服务器**处理它并返回响应。你的浏览器（客户端）向 google.com（服务器）发送 HTTP 请求，返回 HTML。

- **请求 - 响应模型**：同步。客户端等待响应。简单但创建瓶颈：客户端在等待时空闲，服务器必须在继续之前处理请求。

- **无状态服务器**：服务器不记住以前的请求。每个请求包含处理它所需的所有信息。这使扩展变得容易：任何服务器可以处理任何请求，所以你可以在负载均衡器后面添加更多服务器。

- **有状态服务器**：服务器在请求之间维护状态（如用户会话）。更难扩展，因为来自同一用户的请求必须发送到同一服务器（会话亲和性）。现代系统通过将状态存储在数据库或缓存（Redis）中来避免服务器端状态。

## 网络协议

- 我们在第 13 章（TCP/IP 层、套接字）中介绍了网络。这里我们关注系统设计中使用的应用层协议：

- **HTTP/HTTPS**：Web 和大多数 API 的协议。请求方法：GET（读取）、POST（创建/预测）、PUT（更新）、DELETE（删除）。HTTPS 添加 TLS 加密（第 13 章安全）。REST API（第 15 章文件 03）构建在 HTTP 上。

- **WebSocket**：客户端和服务器之间的持久双向连接。与 HTTP（请求 → 响应 → 连接关闭）不同，WebSocket 保持连接开放以进行实时流式传输。用于：LLM token 流式传输（在生成时发送 token）、实时仪表板、聊天应用程序。

- **gRPC**：Google 的 RPC 框架。使用 Protocol Buffers（二进制序列化，比 JSON 小约 10 倍且更快）通过 HTTP/2。支持流式传输（服务器端、客户端端、双向）。用于内部服务间通信（性能重要的地方）。Triton 推理服务器（第 15 章）和 TensorFlow Serving 使用 gRPC。

- **Protocol Buffers**：在 `.proto` 文件中定义消息模式：

```protobuf
message PredictRequest {
    repeated float features = 1;
    string model_version = 2;
}

message PredictResponse {
    float prediction = 1;
    float confidence = 2;
}

service ModelService {
    rpc Predict(PredictRequest) returns (PredictResponse);
}
```

- 模式被编译成任何语言的客户端和服务器代码（Python、C++、Go、Java）。类型安全、向后兼容性和性能免费获得。

## DNS

- **DNS**（域名系统）将人类可读名称转换为 IP 地址（第 13 章）。对于系统设计，DNS 还提供：

- **通过 DNS 负载均衡**：为同一域名返回不同的 IP 地址，在多个服务器之间分配流量。简单但粗粒度（DNS 结果缓存数分钟到数小时，所以流量不会快速重新平衡）。

- **地理路由**：根据客户端位置返回最近数据中心的 IP。东京用户获得日本数据中心；伦敦用户获得欧洲数据中心。

- **故障转移**：如果服务器宕机，DNS 停止返回其 IP。新客户端前往健康服务器。但缓存的 DNS 条目意味着一些客户端继续命中死服务器数分钟（TTL 问题）。

## 代理

- **代理**是客户端和服务器之间的中介：

- **反向代理**（在服务器前面）：客户端连接到代理，代理将请求转发到后端服务器。客户端不知道哪个服务器处理了请求。**Nginx** 和 **HAProxy** 是标准反向代理。它们提供：负载均衡（分配请求）、SSL 终止（在代理解密 HTTPS，发送纯 HTTP 到后端）、缓存、速率限制和压缩。

- **API 网关**：专门用于 API 的反向代理。处理认证、速率限制、请求路由（不同路径 → 不同服务）和 API 版本控制。**Kong**、**AWS API Gateway** 和 **Envoy** 是常见选择。

- 对于 ML 服务：API 网关位于模型服务器前面。它认证 API 密钥、限制免费层用户、路由 `/v1/predict` 到模型服务器 A 和 `/v2/predict` 到模型服务器 B，并收集使用指标。

## 负载均衡

- 当你有多个服务器时，**负载均衡器**将传入请求分配给它们。

![负载均衡器在多个后端服务器之间分配传入请求](../images/load_balancer.svg)

- **算法**：
    - **轮询**：按顺序发送请求（1, 2, 3, 1, 2, 3...）。简单、公平，但不考虑服务器负载。
    - **最少连接**：发送到活动连接最少的服务器。更适合具有可变处理时间的请求（某些 LLM 请求生成 10 个 token，其他生成 1000 个）。
    - **加权轮询**：容量更大的服务器获得更多请求。具有 80 GB GPU 内存的服务器处理 2 倍于 40 GB 的请求。
    - **一致性哈希**：将请求键哈希到特定服务器。相同键总是发送到同一服务器。用于：缓存（对同一用户的请求命中相同缓存）、会话亲和性、前缀缓存（第 17 章：具有相同系统提示的请求前往具有该提示 KV-cache 的服务器）。

- **L4 vs L7 负载均衡**：
    - **L4**（传输层）：基于 IP 和端口路由。快但无法检查请求内容。
    - **L7**（应用层）：基于 HTTP 路径、头部或正文内容路由。可以路由 `/api/chat` 到聊天服务器和 `/api/embed` 到嵌入服务器。较慢但更灵活。

## 缓存

- **缓存**将频繁访问的数据存储在快速存储层（RAM）中，以避免重新计算或重新获取。

![缓存旁路模式：先检查缓存，未命中时从数据库获取并存储到缓存中以备下次使用](../images/cache_aside_pattern.svg)

- **缓存模式**：
    - **缓存旁路**（惰性加载）：应用程序先检查缓存。未命中时，它从数据库获取、存储到缓存并返回。最常见模式。
    - **写透**：每次写入同时进入缓存和数据库。确保缓存始终最新但减慢写入。
    - **回写**：写入仅进入缓存；缓存异步刷新到数据库。最快写入但如果缓存在刷新前崩溃则有数据丢失风险。

- **驱逐策略**（当缓存满时）：
    - **LRU**（最近最少使用）：驱逐最长时间未访问的条目。最常见策略。
    - **LFU**（最不经常使用）：驱逐访问最少的条目。当某些项目始终流行时更好。
    - **TTL**（生存时间）：条目在固定持续时间后过期。用于变得过时的数据（模型预测缓存 5 分钟、特征值缓存 1 小时）。

- **CDN**（内容分发网络）：静态内容（图像、JavaScript、CSS）的全球分布式缓存。全球 100+ 位置的服务器从最近位置向用户提供缓存内容。对于 ML：模型权重可以缓存在 CDN 上以快速下载。

- **Redis**：标准内存缓存/数据库。支持字符串、列表、集合、排序集合、哈希和流。亚毫秒延迟。用于：缓存模型预测、存储会话数据、速率限制（每分钟计数每个用户的请求）和实时特征服务。

- 对于 ML 服务：缓存重复输入的预测。如果许多用户问"法国的首都是什么？"，计算一次答案并提供缓存结果。聊天机器人工作负载的缓存命中率 20-40% 很常见，成比例降低 GPU 成本。

## 数据库

### SQL（关系型）

- **SQL 数据库**（PostgreSQL、MySQL）将数据存储在具有行和列的表中。表之间的关系通过外键表达。查询使用 SQL。**ACID** 保证：

    - **原子性**：事务要么完全完成，要么完全回滚。无部分更新。
    - **一致性**：数据库从一个有效状态移动到另一个。约束（唯一键、外键）始终满足。
    - **隔离**：并发事务不相互干扰。
    - **持久性**：提交的数据在崩溃后存活（写入磁盘后确认）。

- SQL 数据库擅长：具有关系的结构化数据、复杂查询（连接、聚合）、严格一致性要求和数据完整性。

### NoSQL

- **NoSQL 数据库**用一些 ACID 保证换取可扩展性和灵活性：

    - **键值存储**（Redis、DynamoDB）：最简单模型。按键快速查找。用于缓存、会话存储和特征存储。
    - **文档存储**（MongoDB、Firestore）：存储类 JSON 文档。灵活模式（每个文档可以有不同的字段）。用于用户档案、产品目录和配置。
    - **列族存储**（Cassandra、HBase）：针对写重型工作负载和时间序列数据优化。用于事件日志、指标和分析。
    - **图数据库**（Neo4j）：存储节点和边。针对遍历查询优化。用于社交网络、知识图谱和推荐系统。
    - **向量数据库**（Pinecone、Milvus、Weaviate、FAISS）：存储高维嵌入并支持近似最近邻（ANN）搜索。对语义搜索、RAG（检索增强生成）和推荐系统至关重要。

### CAP 定理

- 在分布式数据库中，你最多可以拥有三个属性中的两个：

    - **一致性**：每次读取返回最新写入。
    - **可用性**：每个请求接收响应（即使某些节点宕机）。
    - **分区容错性**：系统在无法通信（节点无法通信）时继续运行。

![CAP 定理：选择 CP（一致）或 AP（可用），因为网络分区是不可避免的](../images/cap_theorem.svg)

- 由于网络分区在分布式系统中是不可避免的，真正的选择是**CP**（一致但在分区期间可能不可用 —— 如 PostgreSQL）vs **AP**（在分区期间可用但可能返回过时数据 —— 如 Cassandra、DynamoDB）。

- 对于 ML：特征存储通常选择 AP（稍微过时的特征值比无预测好）。模型注册表选择 CP（提供错误模型版本是灾难性的）。

### 分片

- **分片**在多台机器之间分割数据库。每个分片保存数据的子集。

- **哈希分片**：哈希键以确定分片。`shard = hash(user_id) % num_shards`。均匀分布但使范围查询不可能。

- **范围分片**：每个分片保存键范围（用户 A-G 在分片 1，H-N 在分片 2）。启用范围查询但可能创建热点（如果许多用户姓名以"S"开头）。

- **重新分片问题**：添加分片使哈希映射无效。**一致性哈希**最小化数据移动：添加第 n 个分片时，只有约 1/n 的键需要移动。

### 数据库索引

- **索引**是加速查询的数据结构，代价是额外存储和更慢写入。没有索引，查询扫描每一行（$O(n)$）。有了索引，它在 $O(\log n)$ 中找到目标。

- **B 树索引**（默认）：平衡树（第 13 章、第 14 章），其中每个节点包含多个键和指针。B 树缓存友好（宽节点适合缓存行）并支持范围查询（`WHERE age BETWEEN 20 AND 30`）。大多数 SQL 数据库使用 B 树。

- **哈希索引**：使用哈希函数将键映射到行位置。$O(1)$ 查找但不支持范围查询。用于精确匹配查找（`WHERE id = 12345`）。

- **复合索引**：多列索引。`CREATE INDEX ON users(country, city)` 加速按国家过滤的查询，或按国家 + 城市，但**不**仅按城市（最左列必须在查询中）。

- **权衡**：每个索引加速读取但减慢写入（索引必须在每次插入/更新/删除时更新）并使用存储（每个索引约表大小的 10-30%）。不要索引所有内容 —— 索引你频繁查询的列。

- **对于 ML 系统**：特征存储的在线数据库需要实体键（user_id、item_id）的索引以快速特征查找。实验跟踪数据库需要（experiment_id、metric_name）的索引用于仪表板查询。

### API 设计

- 系统通过 API 通信。良好的 API 设计使系统可用、可演进和可调试：

- **REST 约定**：对资源使用名词（`/users`、`/models`），对操作使用 HTTP 方法（GET = 读取、POST = 创建、PUT = 更新、DELETE = 删除），对结果使用状态码（200 = 确定、201 = 已创建、400 = 错误请求、404 = 未找到、429 = 速率限制、500 = 服务器错误）。

- **分页**：对于返回列表的端点，永远不要一次返回所有结果。使用基于游标的分页（`GET /items?cursor=abc&limit=50`）或基于偏移量的分页（`GET /items?offset=100&limit=50`）。基于游标对于大数据集更高效（基于偏移量需要跳过行）。

- **版本控制**：在 API 路径前加版本前缀（`/v1/predict`、`/v2/predict`）。这让你演进 API 而不断裂现有客户端。客户端以自己的节奏迁移到 v2；v1 被弃用但直到流量下降才删除。

- **错误响应**：返回结构化错误，具有足够的调试信息：

```json
{
    "error": {
        "code": "INVALID_INPUT",
        "message": "特征'user_age'必须是正整数",
        "details": {"field": "user_age", "value": -5}
    }
}
```

## 消息队列

- **消息队列**将生产者（生成工作的服务）与消费者（处理它的服务）解耦。生产者发送消息到队列；消费者在准备好时拉取它。

- **为什么队列重要**：没有队列，如果消费者慢或宕机，生产者被阻塞。有了队列，生产者触发并忘记；队列缓冲消息直到消费者准备好。

- **Apache Kafka**：分布式、持久、高吞吐量消息队列。消息存储在**主题**中，每个主题分区跨越多个代理。消费者从分区读取，跟踪其位置（**偏移量**）。Kafka 保证分区内的顺序，可以重放消息（日志是持久的）。

- **发布/订阅**：发布者发送消息到主题；该主题的所有订阅者接收副本。用于事件驱动架构："新模型已部署"触发监控服务、A/B 测试服务和日志服务同时。

- 对于 ML：预测请求通过 HTTP 到达，放入 Kafka 队列，由 GPU 工作者处理，结果通过回调或 WebSocket 返回。队列缓冲流量突发，确保如果 GPU 工作者崩溃则不会丢失请求。

## 一致性模型

- 在分布式系统中，不同节点可能对数据有不同视图。**一致性模型**定义系统提供什么保证：

- **强一致性**：写入后，所有后续读取（从任何节点）看到新值。易于推理但慢（需要节点间协调）。

- **最终一致性**：写入后，读取可能在一段时间内看到过时数据，但最终会看到新值。快（无协调）但需要应用程序处理过时读取。

- **因果一致性**：如果操作 A 因果先于 B（例如，"写入 X 然后读取 X"），系统保证 B 看到 A 的结果。但无关操作可能以任何顺序被看到。

- **读己之写**：用户始终立即看到自己的写入，即使其他用户看到过时数据。大多数应用程序需要的最小一致性。

## 弹性模式

- **速率限制**：限制每个用户每个时间窗口的请求数。防止滥用并确保公平访问。使用 Redis 中的令牌桶或滑动窗口计数器实现。

- **断路器**：如果下游服务开始失败（错误率超过阈值），断路器"打开"并停止向它发送请求（立即返回回退响应）。超时后，它"半开"并发送测试请求。如果测试成功，它关闭（正常操作）。这防止级联故障：如果特征存储宕机，模型服务器返回没有特征的预测，而不是在每次请求上超时。

- **背压**：当系统不堪重负时，它向上游发出信号以减慢。与其接受请求并失败，不如早期拒绝多余请求（使用 429 或 503 状态码）。客户端用指数退避重试。

- **带指数退避的重试**：如果请求失败，等待 1 秒并重试。如果再次失败，等待 2 秒。然后 4、8 等。添加抖动（随机延迟）以防止所有客户端同时重试（惊群问题）。

- **幂等性**：如果操作执行两次的效果与执行一次相同，则操作是幂等的。`PUT /user/123 {"name": "Alice"}` 是幂等的（将姓名设置为"Alice"两次没问题）。`POST /payments` 不是（支付两次很糟糕）。使操作幂等确保重试是安全的。
