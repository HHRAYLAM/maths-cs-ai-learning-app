# ML 设计示例

*学习 ML 系统设计最佳方式是通过worked examples。本文件通过七个完整设计进行讲解：推荐系统、搜索排序、广告点击预测、欺诈检测、内容审核、对话式 AI 和大规模图像搜索。*

- 每个示例遵循一致框架：
    1. **问题框架**：我们在构建什么，用户是谁，约束是什么？
    2. **数据**：我们有什么数据，如何收集，如何标注？
    3. **特征**：模型需要什么特征？
    4. **模型**：什么架构和训练方法？
    5. **服务**：如何部署和服务模型？
    6. **评估**：如何衡量成功？
    7. **迭代**：随时间推移我们会做什么改进？

---

## 1. 推荐系统（例如 YouTube、Netflix、Spotify）

### 问题框架

- **目标**：向用户展示他们将喜欢的内容，最大化参与度（观看时长、收听次数、点击）。
- **规模**：10 亿 + 用户，1 亿 + 内容项，每秒 1 万 + 推荐。
- **延迟**：完整推荐管道 <200ms。
- **关键挑战**：候选空间巨大（1 亿内容项）。无法实时为所有用户评分所有内容项。

### 架构：两阶段管道

![推荐管道：1 亿内容项经候选生成缩小到 1000，排序到 100，重排序到 20 个展示项](../images/recommendation_pipeline.svg)

```
1 亿内容项 → 候选生成（快速，粗略）→ 1000 个候选
          → 排序（慢速，精确）→ 100 个排序项
          → 重排序（业务规则）→ 20 个展示给用户
```

### 候选生成

- **目标**：将 1 亿内容项减少到约 1000 个候选。必须快速（<50ms）。
- **双塔模型**：将用户和内容项编码到同一嵌入空间。用户嵌入捕捉偏好；内容项嵌入捕捉内容特征。分数 = 用户和内容项嵌入的点积。
- **训练**：在 (用户，正向内容项，负向内容项) 三元组上进行对比学习。正向 = 用户参与的内容项。负向 = 随机内容项 + 困难负向（用户未参与的热门内容项）。
- **服务**：预计算所有内容项嵌入。请求时：计算用户嵌入，ANN 搜索（向量数据库中的 HNSW）找到 1000 个最近内容项嵌入。

### 排序

- **目标**：精确评分 1000 个候选。可以承受约 100ms。
- **模型**：深度神经网络（MLP 或 transformer），采用丰富特征：用户特征（人口统计、历史、上下文）、内容项特征（内容、热度、新鲜度）和交叉特征（用户 - 内容项交互历史、上下文相关性）。
- **输出**：预测参与概率（点击、观看 50%+、点赞、分享）。可以组合多个目标：$\text{score} = w_1 \cdot P(\text{click}) + w_2 \cdot P(\text{watch}) + w_3 \cdot P(\text{like})$。

### 重排序

- 应用业务规则：多样性（不展示来自同一创作者的 5 个视频）、新鲜度（提升新内容）、安全性（过滤标记内容）和个性化探索（展示一些用户可能发现的较低排名内容项）。

### 粗略数量估算

- **内容项嵌入索引**：1 亿内容项 × 256 维 × float16 = 50 GB。HNSW 索引增加约 2 倍开销 → 约 100 GB。适合单台 128 GB RAM 机器，或分片到 4 台 32 GB 机器。

- **用户嵌入计算**：每用户约 5ms（用户特征的小型 MLP）。在 1 万 QPS 下，需要约 50 个模型副本来处理负载。

- **ANN 搜索**：HNSW 从 1 亿向量中搜索 top-1000 约 2ms。在 1 万 QPS 下，每个索引副本处理约 500 QPS → 需要 20 个副本。

- **排序模型**：1000 个候选 × 每个候选约 0.1ms = 每请求 100ms。在 1 万 QPS 下，每秒需要 1000 GPU 秒 → 仅排序就需要约 10 个 A10G GPU。

- **总基础设施**：约 20 个嵌入索引副本 + 约 50 个用户嵌入服务器 + 约 10 个排序 GPU + 缓存 + 负载均衡器。成本：云价格下约$5 万-$10 万/月。

### 冷启动

- **新用户**（无历史）：使用人口统计特征、设备/位置上下文和基于热度的推荐。5-10 次交互后，切换到个性化模型。

- **新内容项**（无参与数据）：使用基于内容的特征（标题、描述、缩略图嵌入）。分配探索预算：向一小部分用户展示新内容项以快速收集参与数据。提升期后无参与的内容项会被降级。

- **冷启动是系统问题**：特征存储必须优雅处理缺失特征（返回默认值，而非错误）。模型必须用缺失特征训练（训练期间用户历史特征的 dropout 模拟新用户）。

### 评估

- **离线**：NDCG（归一化折损累积增益）、recall@K、precision@K 在保留集上。
- **在线**：A/B 测试衡量观看时长、DAU、留存率。长期 A/B 测试（数周）以捕捉短期测试遗漏的用户留存影响。

## 2. 搜索排序（例如 Google、Bing）

### 问题框架

- **目标**：给定用户查询，从数十亿文档语料库返回最相关结果。
- **延迟**：总共 <500ms（100ms 检索 + 200ms 排序 + 100ms 渲染 + 开销）。

### 架构：查询理解 → 检索 → 排序

### 查询理解

- 检索前处理原始查询以改进结果：

- **拼写纠正**："reccomendation systm" → "recommendation system"。使用编辑距离模型或序列到序列模型在搜索日志的（拼写错误，纠正）对上训练。

- **查询扩展**：添加相关术语以提高召回率。"Python ML" → "Python machine learning scikit-learn pytorch"。使用同义词典、词嵌入或 LLM 生成扩展。

- **意图分类**：确定用户想要什么。"buy Nike shoes" 是**交易型**（展示产品页面）。"How does backpropagation work" 是**信息型**（展示文章）。"facebook.com" 是**导航型**（直接前往网站）。不同意图应触发不同检索策略和结果布局。

- **实体识别**：从查询中提取实体。"best restaurants near Times Square" → 位置："Times Square"，实体类型："restaurants"。路由到位置感知搜索管道。

### 检索

- **BM25**（传统）：使用倒排索引的术语匹配检索。快速，对关键词查询有效。无语义理解（"dog food" 不匹配"canine nutrition"）。

- **稠密检索**：将查询和文档编码为嵌入（使用双编码器如 DPR 或 ColBERT）。通过 ANN 搜索检索。捕捉语义相似性（"dog food" 匹配"canine nutrition"）。比 BM25 慢但对自然语言查询更好。

- **混合检索**：结合 BM25 和稠密检索。BM25 找到精确关键词匹配；稠密检索找到语义匹配。合并并去重。两全其美。

### 排序

- **学习排序**：模型评分每个（查询，文档）对。三种方法：
    - **逐点式**：独立预测每个文档的相关性分数。简单但忽略相对排序。
    - **成对式**：预测两个文档哪个更相关。LambdaMART（梯度提升树）是经典方法。
    - **列表式**：直接优化整个排序列表以优化列表级指标（NDCG）。更复杂但效果最好。

- **交叉编码器**：采用 $[查询，文档]$ 作为输入并输出相关性分数的 transformer。比双编码器（独立编码查询和文档）更准确，因为它捕捉细粒度交互。但对完整语料库太慢 — 仅用于重排序检索后的 top 100-1000 个候选。

### 特征

- **查询特征**：查询长度、语言、意图分类（导航型、信息型、交易型）。
- **文档特征**：PageRank、新鲜度、内容质量分数、域名权威。
- **查询 - 文档特征**：BM25 分数、嵌入相似性、精确匹配计数、历史日志中此（查询，文档）对的点击率。

## 3. 广告点击预测

### 问题框架

- **目标**：预测用户点击广告的概率。这决定实时拍卖中的出价金额。
- **规模**：每秒 10 万 + 拍卖，每次需要 10ms 内预测。
- **收入影响**：点击预测准确率 0.1% 提升转化为数百万额外收入。

### 架构

- **特征工程**是广告系统的核心。特征包括：
    - **用户特征**：人口统计、浏览历史、购买历史、设备、位置、一天中的时间。
    - **广告特征**：创意（图像/文本）、广告主、类别、历史 CTR、出价值。
    - **上下文特征**：页面内容、广告位置、设备类型、连接速度。
    - **交叉特征**：user_category × ad_category 交互、user_region × ad_campaign 交互。

- **模型**：历史上是逻辑回归（简单、快速、可解释）。现代系统使用深度学习：**DLRM**（深度学习推荐模型），具有用于分类特征的嵌入表和用于稠密特征的 MLP。

- **校准**：预测概率必须准确（如果模型说 P(click) = 0.05，那么此类展示的 5% 实际应被点击）。校准至关重要，因为预测概率直接决定出价值。

- **探索 - 利用**：总是展示预测最佳广告长期来看不佳（永远不会发现新广告可能更好）。Thompson 采样或 $\epsilon$-greedy 探索确保一部分展示给不太确定的广告以收集数据。

### 实时竞价

- 用户加载页面时，广告拍卖在 <100ms 内运行：
    1. 发布商发送竞价请求（用户信息、页面上下文）到多个广告交易所。
    2. 每个广告主的竞价服务器预测其广告的 CTR。
    3. 出价 = CTR × 每次点击价值。更高出价赢得拍卖。
    4. 展示获胜广告；如果点击，广告主付费。

## 4. 欺诈检测

### 问题框架

- **目标**：实时检测欺诈性交易（信用卡欺诈、账户盗用、虚假评论）。
- **延迟**：<100ms（支付处理前必须批准或标记交易）。
- **关键挑战**：极端类别不平衡（0.1% 欺诈率）。误报阻止合法用户；漏报损失金钱。

### 架构

![欺诈检测管道：交易 → 实时特征管道 → ML 模型 → 决策引擎 → 允许/审核/阻止，人工审核反馈标签用于重新训练](../images/fraud_detection_pipeline.svg)

### 特征

- **交易特征**：金额、货币、商家类别、一天中的时间、is_international。
- **用户特征**：账户年龄、平均交易金额、最近交易次数、设备指纹。
- **速度特征**（来自流式管道的实时）：过去 5 分钟交易次数、过去 1 小时不同商家次数、与前一次交易的地理距离。
- **图特征**：此商家是否连接到已知欺诈团伙？此设备是否与标记账户共享？

### 模型

- **梯度提升树**（XGBoost、LightGBM）是表格欺诈检测的标准。它们处理混合特征类型，可解释（特征重要性），训练快速。

- **处理不平衡**：欠采样多数类、过采样少数类（SMOTE）或在损失函数中使用类别权重。Focal loss（第 8 章）降低简单负向的权重。

- **成本矩阵**：误报（阻止合法交易）有成本（用户沮丧、损失销售）。漏报（遗漏欺诈）有不同成本（财务损失）。决策阈值应最小化总期望成本，而非最大化准确率。

### 人在回路

- 不确定预测（模型置信度在 0.3 到 0.7 之间）发送给人工审核员。审核员决策成为重新训练的标签。这创建反馈回路：模型随时间推移看到更多标记欺诈案例而改进。

## 5. 内容审核

### 问题框架

- **目标**：自动检测并从平台移除有害内容（仇恨言论、暴力、虚假信息、CSAM）。
- **规模**：每天数十亿帖子（文本、图像、视频）。
- **挑战**：上下文依赖（讽刺、讽刺、文化细微差别）。必须平衡言论自由与安全。

### 架构

- **多模态分类**：文本、图像和视频的独立模型，具有组合信号的融合层。

- **文本审核**：微调语言模型将文本分类到类别（骚扰、仇恨言论、虚假信息、垃圾邮件）。多语言模型处理 100+ 语言。

- **图像审核**：视觉模型检测：明确内容（裸体、暴力）、图像中的文本（OCR + 文本分类器）和已知有害内容（与已知 CSAM 数据库的哈希匹配）。

- **视频审核**：定期采样帧，对每帧运行图像分类器，结合音频转录（ASR → 文本分类器）。

- **策略即代码**：审核策略在结构化规则中定义，将模型输出映射到行动：

```python
if text_model.hate_speech_score > 0.9:
    action = "remove"
elif text_model.hate_speech_score > 0.7:
    action = "human_review"
else:
    action = "allow"
```

- 策略频繁变化（新法规、演变规范）。将策略与模型分离确保无需重新训练即可部署变化。

### 主动与被动审核

- **主动**（发布前）：在内容上线前运行分类器。高置信度违规自动阻止。这防止有害内容可见但增加发布延迟并有误报风险（阻止合法内容）。

- **被动**（发布后）：内容立即上线。用户可以报告违规。报告触发分类器 + 人工审核。发布者延迟更低但有害内容在检测前可见。

- **大多数平台两者都用**：高严重性类别主动（CSAM：零容忍，发布前阻止），细微类别被动（虚假信息：需要人工判断，报告后审核）。

### 哈希匹配

- 对于已知有害内容（CSAM、恐怖主义宣传），使用**感知哈希**：计算对微小修改（裁剪、调整大小、压缩）鲁棒的图像/视频哈希。与已知有害内容数据库（NCMEC 哈希数据库、GIFCT 共享哈希数据库）比较。匹配 → 立即移除，无需分类器。

- **PhotoDNA**（微软）是 CSAM 检测的标准感知哈希。在许多司法管辖区是法律义务，不仅是技术选择。

### 粗略数量估算

- **规模**：10 亿帖子/天 = 约 1.2 万帖子/秒。每帖子需要：文本分类（约 5ms）、图像分类（约 20ms）、哈希匹配（约 1ms）。在 1.2 万 QPS：需要约 60 个文本分类器、约 240 个图像分类器和约 12 个哈希匹配器（加冗余）。

- **人工审核**：如果 2% 帖子标记审核 = 2000 万/天。100 次审核/审核员/天，需要 20 万审核员（这就是为什么自动化准确率重要：误报率每降低 0.1% 每天节省 100 万审核）。

- **延迟预算**：主动审核必须在发布管道内完成（约 500ms）。文本（5ms）+ 图像（20ms）+ 哈希（1ms）+ 开销 = 远低于预算。视频是例外：即使从 10 分钟视频中每秒采样 1 帧也需要 600 次分类器调用 → 异步处理。

### 升级工作流

- 自动移除 → 上诉人工审核 → 专家审核（法律、文化专家）→ 模糊案例策略团队。每层处理更少案例但更细致。

- **反馈到模型**：人工审核决策是重新训练的最高质量标签。模型与审核员之间的分歧优先用于主动学习 — 它们代表模型处理最差的案例。

## 6. 对话式 AI（基于 RAG 的聊天机器人）

### 问题框架

- **目标**：聊天机器人使用公司文档回答关于产品的问题。
- **要求**：准确（不幻觉）、引用来源、处理后续问题、保持在产品领域内。

![RAG 架构：嵌入查询，搜索向量 DB 查找相关块，重排序，提供给 LLM 与原始查询生成有依据的响应](../images/rag_architecture.svg)

### 架构：检索增强生成（RAG）

```
用户查询 → 查询嵌入 → 向量搜索（文档）→ Top-K 块
                                                                      ↓
用户查询 + 检索块 → LLM → 响应（带引用）
```

### 组件

- **文档摄入**：分块文档并嵌入它们。**分块策略**很重要：

    - **固定大小分块**：每 N token 分割（例如 500），M token 重叠（例如 50）。简单，可预测块大小，但可能在句子或段落中间分割，丢失上下文。

    - **语义分块**：在段落或章节边界分割。每块是连贯信息单元。可变大小（有些块 100 token，其他 800），需要检索系统处理可变长度。

    - **递归分块**：尝试在段落边界分割。如果段落太长，在句子边界分割。如果句子太长，在固定大小分割。连贯性和大小一致性的最佳平衡。

    - **嵌入**：用文本编码器（例如 E5、BGE、Cohere embed）嵌入每块。存储在向量数据库中。

- **检索**：嵌入用户查询，在向量数据库中搜索 $k$ 个最相似块（通常 $k = 5$-$10$）。可选用交叉编码器重排序以获得更高精度。

- **生成**：用检索块作为上下文构造提示：

```
System: You are a helpful assistant. Answer based ONLY on the provided context.
If the answer is not in the context, say "I don't know."

Context:
[chunk 1]
[chunk 2]
...

User: {question}
```

- **防护栏**：防止 LLM 回答产品领域外问题、生成有害内容或与检索上下文矛盾。实现为：输入过滤（拒绝离题查询）、输出过滤（检查响应与检索上下文）和宪法提示（指示模型拒绝某些请求）。

- **对话记忆**：维护最后 $n$ 轮对话。将它们包含在提示中以便模型理解后续问题（"定价如何？" → 需要之前关于哪个产品的上下文）。

### 查询重写

- 用户经常问模糊后续问题："定价如何？"（什么的定价？）。**查询重写**使用对话历史生成独立查询：

    - 输入：对话历史 + "定价如何？"
    - 重写后："产品 X 企业计划的定价是什么？"

- 这个重写查询被嵌入并在向量数据库中搜索。没有重写，检索会搜索无上下文的"定价"并返回无关块。

- 查询重写可以用小型 LLM 调用（约 50ms）或微调序列到序列模型（约 5ms）完成。

### 粗略数量估算

- **文档语料库**：1 万页，平均 2000 token 每页 = 2000 万 token。500 token/块，50 重叠 = 约 4.4 万块。
- **嵌入索引**：4.4 万块 × 768 维 × float16 = 约 65 MB。轻松适合内存。即使 1000 万块也约 15 GB。
- **延迟细分**：查询嵌入（5ms）+ 向量搜索（2ms）+ 交叉编码器重排序（top-50 为 20ms）+ LLM 生成（500-2000ms）= 总共约 600-2100ms。LLM 主导。使用流式减少感知延迟。
- **成本**：$3/100 万 token（Claude/GPT-4 API），1000 查询/天，每查询约 2K token = 约$6/天。大规模（100 万查询/天），在 2 个 A10G GPU 上自托管 7B 模型（约$50/天）以 100 倍成本降低。

### 评估

- **检索质量**：Recall@K（top-K 块是否包含答案？）、MRR（平均倒数排名）。
- **生成质量**：事实准确性（响应是否匹配检索上下文？）、有依据性（响应是否引用正确块？）、答案相关性。
- **端到端**：用户满意度（点赞/点踩）、升级到人工代理的比率。

## 7. 大规模图像搜索

### 问题框架

- **目标**：给定图像，从 10 亿 + 图像语料库找到视觉相似图像。
- **应用**：反向图像搜索、产品搜索（照片 → 匹配产品）、重复检测。
- **延迟**：<500ms 包括网络往返。

### 架构

```
查询图像 → 嵌入模型（ViT/CLIP）→ 512 维向量 → ANN 搜索 → Top-K 结果
```

### 嵌入提取

- **模型**：预训练视觉编码器（ViT、CLIP 图像编码器、DINOv2）。如需在特定领域微调（时尚、电商、医学成像）。

- **训练**：对比学习（第 10 章）。正向对 = 同一图像的不同视图（或图像 + 匹配文本）。负向对 = 随机图像。模型学习为相似图像产生相似嵌入，为不同图像产生不同嵌入。

### 索引

- **离线**：嵌入所有 10 亿图像并构建 ANN 索引。对于 HNSW（文件 03），构建索引需要数小时，索引存储在内存中（10 亿 × 512 维 × float16 + 图开销约 128 GB）。

- **分片**：跨多台机器分割索引。每台机器持有分片。查询时，并行搜索所有分片并合并 top-K 结果。

- **增量更新**：新图像（上传、新产品）必须添加到索引。HNSW 支持增量插入，无需重建。向量数据库（Milvus、Pinecone）本地处理此。

### 服务

- **嵌入服务**：运行 ViT 模型的 GPU 服务器。延迟：每图像约 20ms。批量多查询以提高吞吐量。

- **搜索服务**：ANN 索引服务器。延迟：HNSW 从 10 亿向量中 top-100 搜索约 10ms。

- **缓存**：缓存热门查询结果。对于重复检测，缓存最近上传图像的嵌入，在搜索完整索引前将新上传与缓存比较。

### 评估

- **Precision@K**：top-K 结果是否实际相似？
- **Recall@K**：语料库中所有真正相似图像中，多少在 top-K 中？
- **Mean Average Precision (mAP)**：精确率 - 召回率曲线下的面积。
- **人工评估**：对于主观相似性，人工评分员判断检索图像是否相关。

## 面试框架

- 遇到系统设计问题时，遵循此框架：

1. **澄清要求**（2-3 分钟）：询问规模、延迟、一致性要求和边缘情况。"多少用户？可接受什么延迟？故障时会发生什么？"

2. **高级设计**（5-7 分钟）：绘制主要组件及其交互。从 happy path 开始。使用文件 01-03 的模式。

3. **深入**（15-20 分钟）：选择最有趣/最具挑战性的组件并详细设计。这是展示深度的地方。对于 ML 系统，深入通常是：模型架构、特征管道或服务架构。

4. **评估和监控**（3-5 分钟）：如何衡量成功？可能出什么问题？如何检测和响应问题？

5. **迭代**（2-3 分钟）：如果有更多时间/资源会改进什么？这表明理解权衡并能排优先级。

- **面试官寻找**：结构化思维（不急于解决方案）、权衡意识（每个选择都有成本）、实践知识（实际构建过系统）和沟通（能否清晰解释设计）。
