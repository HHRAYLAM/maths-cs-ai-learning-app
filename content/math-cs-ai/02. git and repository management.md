# Git 和版本控制

*Git 是软件团队如何协作而不会相互覆盖工作。本文件涵盖 Git 思维模型、分支策略、合并和变基、冲突解决、pull requests，以及管理 ML 特定挑战如大文件和实验跟踪。*

- 每个严肃软件项目都使用版本控制。Git 是主导系统，被几乎所有开源项目和公司使用。没有 git，协作是发送 zip 文件邮件并祈祷没人覆盖你的更改。有了 git，每个更改被跟踪、可撤销、可归属。

- 对于 ML 工程师：git 跟踪你的代码、配置和实验脚本。结合实验跟踪工具，它给你可重现性："什么确切代码和配置产生这个模型？"

## 思维模型

- Git 跟踪项目的快照。每个提交是那时所有跟踪文件的完整快照，而非差异（内部 git 为效率存储差异，但概念上每个提交是完整状态）。

- 文件的四个"位置"：

    1. 工作目录：磁盘上的实际文件。你编辑这些。
    2. 暂存区 (index)：你已标记为下次提交的文件。`git add` 将更改移到这里。
    3. 本地仓库：你的提交历史，存储在 `.git/` 中。`git commit` 将暂存区保存为新快照。
    4. 远程仓库（如 GitHub）：共享副本。`git push` 上传你的提交，`git pull` 下载他人的。

```
工作目录  →  git add  →  暂存  →  git commit  →  本地仓库  →  git push  →  远程
                                                        ←  git pull  ←
```

- 暂存区是 git 强大的原因。你可以编辑 10 个文件但仅提交其中 3 个，保持其他更改用于单独提交。这实现干净、聚焦的提交。

### 基本命令

```bash
git init                          # 创建新仓库
git clone url                     # 下载远程仓库
git status                        # 什么已更改？（最常用命令）
git add file.py                   # 暂存特定文件
git add .                         # 暂存所有更改（小心使用）
git commit -m "描述性消息"        # 提交暂存更改
git push                          # 上传提交到远程
git pull                          # 下载 + 合并远程更改
git log --oneline                 # 紧凑提交历史
git diff                          # 显示未暂存更改
git diff --staged                 # 显示已暂存更改
```

## 分支

- 分支是指向提交的指针。默认分支是 `main`（或 `master`）。创建分支给你独立开发线：你可以在不影响 `main` 的情况下更改。

```bash
git branch feature-x              # 创建分支
git checkout feature-x            # 切换到它
git checkout -b feature-x         # 一步创建并切换
git branch -d feature-x           # 删除分支（合并后）
git branch -a                     # 列出所有分支（本地 + 远程）
```

- 何时分支：总是。永远不要直接提交到 `main`。每个功能、bug 修复或实验获得自己的分支。这保持 `main` 稳定和可部署。

### 分支策略

- 功能分支（最常见）：每个功能/修复获得 `main` 的分支。完成后，开 pull request (PR) 合并回来。简单，对大多数团队有效。

- 基于主干的开发：开发者频繁提交到 `main`（每天多次），用功能标志隐藏未完成工作。持续部署团队首选（Google、Facebook）。需要优秀 CI/CD。

- Gitflow：为功能、发布和热修复分离分支。更复杂，更好用于带版本发布的软件（移动应用、打包软件）。对大多数 ML 项目过度。

- 对于 ML 团队：功能分支带短命分支（1-3 天内合并）是最佳点。长命分支偏离 `main` 并创建痛苦合并冲突。

## 合并和变基

- 合并创建新"合并提交"组合两个分支：

```bash
git checkout main
git merge feature-x
```

- 这保留完整历史：你可以看到工作在分支上进行以及何时合并。合并提交有两个父节点。

- 变基重放你的分支提交在目标分支顶部：

```bash
git checkout feature-x
git rebase main
```

- 这重写历史：你分支的提交获得新哈希，好像你已从 `main` 当前尖端开始工作。结果是线性历史（无合并提交），更干净易读。

- 何时用哪个：
    - 变基用于用最新 `main` 更新你的功能分支（保持分支干净和最新）。
    - 合并用于集成你的功能分支到 `main`（保留分支历史）。
    - 永远不要变基已推送并与他人共享的提交。变基重写历史；如果其他人基于原始提交进行工作，变基引起混乱。

## 解决冲突

- 冲突发生在两个分支修改同一文件的同一行。Git 无法自动决定保留哪个更改并要求你手动解决。

```
<<<<<<< HEAD
learning_rate = 0.001
=======
learning_rate = 0.0005
>>>>>>> feature-x
```

- 在 `<<<<<<< HEAD` 和 `=======` 之间是当前分支版本。在 `=======` 和 `>>>>>>> feature-x` 之间是进入分支版本。你决定保留哪个（或组合它们），移除标记，保存，并 `git add` 解决的文件。

- 陷阱：不要留冲突标记在提交文件中。它们是破坏代码的字面文本。解决后总是搜索 `<<<<<<<`。

- 减少冲突：保持分支短命，频繁合并 `main` 到你的分支，避免多人同时编辑同一文件。

## 写好提交消息

- 提交消息是给未来自你和队友。"fix bug" 不告诉你任何事。"Fix off-by-one in batch size calculation that caused OOM on 8-GPU training" 告诉你一切。

- 格式：

```
短摘要（50 字符或更少，祈使语气）

更长描述（如需要）。解释为什么，而非什么
（diff 显示什么更改）。72 字符换行。

Fixes #123
```

- 祈使语气："Add feature" 而非 "Added feature" 或 "Adds feature"。读作完成句子："如果应用，这个提交将 add feature。"

- 原子提交：每个提交应做一件事。"Add data loader" 是一个提交。"Add data loader and fix unrelated bug and update README" 应是三个提交。这使 `git bisect`（找哪个提交引入 bug）可能。

## Pull Requests 和代码审查

- pull request (PR) 提议合并分支到 `main`。它是代码审查网关：队友读你的更改，建议改进，合并前批准。

- 好 PR 实践：
    - 保持 PR 小（400 行更改内）。大 PR 被草率审查因为没人想审查 2000 行。
    - 写清晰描述：什么更改、为什么、如何测试。
    - 链接到激发更改的问题或 ticket。
    - 及时响应审查评论。
    - 合并前压缩琐碎提交（所以 `main` 有干净历史）。

- 代码审查不是关于找 bug（测试做那个）。它是关于：知识共享（审查者学习代码库）、设计反馈（这是正确方法吗？）和维持标准（命名、风格、架构）。

## .gitignore

- `.gitignore` 文件告诉 git 哪些文件排除跟踪。对于 ML 项目：

```gitignore
# Python
__pycache__/
*.pyc
*.egg-info/
.venv/
env/

# 数据和模型（对 git 太大）
data/
*.csv
*.parquet
models/
*.pt
*.onnx
*.bin
checkpoints/

# 密钥
.env
*.pem
credentials.json

# IDE
.vscode/
.idea/
*.swp

# OS
.DS_Store
Thumbs.db

# Jupyter
.ipynb_checkpoints/

# 实验输出
wandb/
mlruns/
outputs/
logs/
```

- 陷阱：在文件已提交后添加到 `.gitignore` 不从仓库移除它。你必须 `git rm --cached file` 取消跟踪。文件永久留在历史中除非你重写历史（这混乱）。

## Git for ML

- ML 引入传统软件没有的挑战：

- 大文件：数据集和模型权重是 GB 或更多。Git 为文本文件（源代码）设计，非二进制大对象。解决方案：
    - Git LFS（大文件存储）：在 git 中跟踪指针，存储实际文件在单独服务器。简单但有存储/带宽限制在 GitHub 上。
    - DVC（数据版本控制）：在 git 外管理数据和模型文件，使用远程存储（S3、GCS）。像 git 对数据工作：`dvc add data.csv`、`dvc push`、`dvc pull`。

- 实验跟踪：哪个提交 + 哪个超参数 + 哪个数据产生哪个指标？Git 跟踪代码，而非完整实验上下文。
    - Weights & Biases (W&B)：记录指标、超参数、系统信息，链接到 git 提交。提供仪表板比较运行。
    - MLflow：开源实验跟踪带模型注册表。记录参数、指标和工件。
    - 简单方法：在训练脚本中记录 git 哈希：`git_hash = subprocess.check_output(['git', 'rev-parse', 'HEAD']).strip()`。与结果一起存储。

- 可重现性检查清单（每个实验跟踪什么）：
    - Git 提交哈希（确切代码版本）
    - 配置文件 / 超参数
    - 随机种子
    - Python 和库版本 (`pip freeze`)
    - 数据版本（DVC 哈希或数据集版本标签）
    - 硬件（GPU 类型、GPU 数量）

```bash
# 快速可重现性快照
echo "Commit: $(git rev-parse HEAD)" > experiment_info.txt
echo "Branch: $(git branch --show-current)" >> experiment_info.txt
echo "Dirty: $(git status --porcelain | wc -l) files" >> experiment_info.txt
pip freeze >> experiment_info.txt
nvidia-smi >> experiment_info.txt
```
