## 💾 训练内存的四大组件

**训练时 GPU 内存要装四样东西**：

| 组件 | 说明 | 7B 模型占用 |
|------|------|------------|
| **参数** | 模型权重 | 28 GB (FP32) |
| **梯度** | 反向传播的梯度 | 28 GB (FP32) |
| **优化器状态** | Adam 的一阶二阶矩 | 56 GB (2×FP32) |
| **激活** | 前向传递的中间值 | 可变（通常最大） |

<div class="lesson-two-column">
  <div class="lesson-text-column">

### 7B 模型内存计算

**参数**：7×10⁹ × 4 字节 = 28 GB

**梯度**：同样 28 GB

**优化器状态**：
- Adam 需要 2 个 FP32 缓冲区
- 7×10⁹ × 4 × 2 = 56 GB

**总计**：28 + 28 + 56 = **112 GB**

这还没算激活！单个 80GB GPU 放不下。

### 解决方案

1. **混合精度训练**：用 FP16/BF16 减半激活和梯度
2. **数据并行**：多 GPU 分担
3. **模型并行**：把模型切开

</div>

```

├─────────────────┤
```

## 🔢 混合精度训练 - 用更少的内存

**混合精度**用低精度（FP16/BF16）存储激活和梯度，同时保持 FP32 主权重：

```
混合精度训练流程：

FP32 主权重
    ↓
转 FP16/BF16  → 前向传递 → 计算损失
    ↓
FP16/BF16 梯度  → 反向传递
    ↓
转回 FP32 → 优化器更新主权重

好处：
- 激活和梯度内存减半
- 计算更快（Tensor Core 加速）
- FP32 主权重保证数值稳定
```

<div class="story-card jerry">
  <div class="story-card-title">🐭 Jerry 的理解</div>
  <div class="story-card-content">
    <strong>FP32</strong>：32 位，精确但占内存
    <br><br>
    <strong>FP16</strong>：16 位，省内存但范围小
    <br><br>
    <strong>BF16</strong>：16 位 brain float，范围大精度低
    <br><br>
    <strong>BF16 是现代训练默认选择</strong>！
  </div>
</div>

---

### FP16 vs BF16

| 特性 | FP32 | FP16 | BF16 |
|------|------|------|------|
| **指数位** | 8 | 5 | 8 |
| **尾数位** | 23 | 10 | 7 |
| **数值范围** | 大 | 小 | 大（同 FP32） |
| **精度** | 高 | 中 | 低 |
| **需要损失缩放** | 否 | 是 | 通常否 |

**BF16 优势**：指数范围与 FP32 相同，几乎不会溢出，是现代训练的首选！

## 🔄 数据并行 - 最简单的分布式策略

**数据并行**复制模型到多个 GPU，每个 GPU 处理不同数据：

<div class="lesson-two-column">
  <div class="lesson-text-column">

### 数据并行流程

1. **复制模型**：每个 GPU 有完整模型副本
2. **分割数据**：Batch 分成 N 份
3. **独立计算**：每个 GPU 计算自己的梯度
4. **All-Reduce**：平均所有 GPU 的梯度
5. **更新权重**：每个 GPU 用平均梯度更新

### 有效 Batch Size

$$\text{有效 Batch} = N \times \text{单 GPU Batch}$$

- 4 个 GPU，每 GPU Batch=32 → 有效 Batch=128

</div>
  <div class="lesson-visual-column">

```
数据并行可视化：

         Batch (128 样本)
              ↓
    ┌─────────┴─────────┐
    ↓                   ↓
GPU 1 (32 样本)    GPU 2 (32 样本)
    │                    │
  前向 + 反向          前向 + 反向
    │                    │
  梯度 1              梯度 2
    └─────────┬─────────┘
              ↓
         All-Reduce
         平均梯度
              ↓
       同步更新权重

数学上等价于用 128 Batch 单 GPU训练！
```

<div class="visual-caption">图 2：数据并行工作流程</div>

</div>
</div>

### All-Reduce 通信操作

**All-Reduce** 是分布式训练的核心通信原语：

```
朴素 All-Reduce：
所有 GPU → 主 GPU → 求和 → 广播回所有 GPU
问题：主 GPU 成为瓶颈！

环形 All-Reduce（高效）：
GPU 1 ↔ GPU 2 ↔ GPU 3 ↔ GPU 4
        ↑               ↓
        └───────┘

每 GPU 只与邻居通信
带宽最优，不随 GPU 数量增加！
```

**NCCL**（NVIDIA 集体通信库）实现了优化的 All-Reduce。

## 🧩 模型并行 - 当模型太大放不下

**模型并行**把模型切开，不同部分放不同 GPU：

| 类型 | 分割方式 | 适用场景 |
|------|---------|---------|
| **张量并行** | 层内分割矩阵 | 节点内 8 GPU |
| **流水线并行** | 层间分割 | 跨节点 |
| **混合并行** | 两者结合 | 超大规模训练 |

### 张量并行

**张量并行**在层内分割矩阵运算：

```
大矩阵乘法 Y = XW

单 GPU：
X (B×d) × W (d×d) = Y (B×d)

张量并行（2 GPU）：
W 切成 [W₁, W₂] 每 GPU 一半
GPU1: Y₁ = XW₁
GPU2: Y₂ = XW₂
结果：Y = [Y₁, Y₂] 拼接

需要 GPU 间快速通信（NVLink）
每层结束要同步部分结果
```

**适用**：注意力投影、前馈层的大矩阵

---

### 流水线并行

**流水线并行**把不同层分配给不同 GPU：

<div class="lesson-two-column">
  <div class="lesson-text-column">

### 朴素流水线

```
GPU0: 层 1-4
GPU1: 层 5-8
GPU2: 层 9-12
GPU3: 层 13-16

数据像流水线一样流动：
Batch → GPU0 → GPU1 → GPU2 → GPU3 → 输出
```

### 流水线气泡问题

```
时间 →
GPU0: [计算] [等待] [计算] [等待]
GPU1: [空闲] [计算] [等待] [计算]
GPU2: [空闲] [空闲] [计算] [等待]
GPU3: [空闲] [空闲] [空闲] [计算]

大量空闲时间 = 效率低！
```

</div>
  <div class="lesson-visual-column">

### 微批次解决气泡

```
把 Batch 分成 4 个微批次：

时间 →
GPU0: [μ1] [μ2] [μ3] [μ4] [空]
GPU1: [空] [μ1] [μ2] [μ3] [μ4]
GPU2: [空] [空] [μ1] [μ2] [μ3]
GPU3: [空] [空] [空] [μ1] [μ2]

所有 GPU 大部分时间忙碌！
气泡时间被摊销
```

## 🧠 混合专家（MoE）- 更多参数，相同计算

**MoE** 架构让模型有更多参数但不增加计算量：

<div class="story-card jerry">
  <div class="story-card-title">🐭 Jerry 的理解</div>
  <div class="story-card-content">
    <strong>标准 Transformer</strong>：每个 token 经过所有层
    <br><br>
    <strong>MoE</strong>：每层有多个"专家"，每个 token 只选 top-K 专家
    <br><br>
    <strong>好处</strong>：参数多容量大，但计算量不增加
    <br><br>
    <strong>例子</strong>：Mixtral 8×7B = 47B 参数，但只用 13B 计算
  </div>
</div>

---

### MoE 路由机制

```
MoE 层结构：

输入 token
    ↓
门控网络 G
    ↓
计算每个专家的分数
    ↓
选 Top-K 专家（如 K=2）
    ↓
只运行选中的专家
    ↓
加权组合输出

8 个专家，每 token 只用 2 个：
总参数 8×，但计算量只 2×！
```

### MoE 挑战

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **负载平衡** | token 可能都去同一专家 | 辅助损失鼓励均匀分布 |
| **通信开销** | 专家在不同 GPU 上 | All-to-All 通信优化 |
| **训练稳定** | 路由可能不稳定 | 容量因子、噪声注入 |

## 📈 缩放定律 - 如何分配计算预算

**缩放定律**描述模型性能如何随规模提升：

$$L(N) \propto N^{-\alpha_N}, \quad L(D) \propto D^{-\alpha_D}, \quad L(C) \propto C^{-\alpha_C}$$

- $N$ = 参数量
- $D$ = 数据量
- $C$ = 计算量

<div class="lesson-two-column">
  <div class="lesson-text-column">

### Kaplan 缩放定律（2020）

发现损失随规模呈幂律下降：

- 更大模型 → 更好性能
- 更多数据 → 更好性能
- 更多计算 → 更好性能

### Chinchilla 缩放定律（2022）

**重大发现**：大多数模型训练不足！

对于给定计算预算：
- **最优比例**：每参数 20 token
- 7B 模型应看 140B token
- 而非 Llama 1 的 300B token

</div>
  <div class="lesson-visual-column">

```
Chinchilla 最优训练：

计算固定时：

之前想法：
最大模型 + 较少数据
    ↓
    训练不足！

Chinchilla：
较小模型 + 更多数据
    ↓
    更好性能！

最优比例：~20 token/参数
```

## 🛠️ 其他关键技术

### 梯度累积

**在有限内存上模拟更大 Batch**：

```
正常训练：
每个 Batch 更新一次 → Batch=32

梯度累积（累积 4 步）：
Batch 1: 计算梯度，不更新
Batch 2: 计算梯度，不更新
Batch 3: 计算梯度，不更新
Batch 4: 计算梯度，不更新
然后：平均 4 个梯度，更新一次

等价于 Batch=128，但内存只用 32！
```

---

### 梯度检查点

**用计算换内存**：

<div class="story-card">
  <div class="story-card-title">🐱 Tom 的梯度检查点</div>
  <div class="story-card-content">
    <strong>正常反向传播</strong>：保存所有激活，内存 O(L)
    <br><br>
    <strong>梯度检查点</strong>：只保存部分激活，反向时重新计算
    <br><br>
    <strong>效果</strong>：激活内存减少 √L 倍，计算增加~33%
    <br><br>
    用时间换空间！
  </div>
</div>

---

### 容错与检查点

**训练持续数周时，故障是必然的**：

| 技术 | 作用 | 频率 |
|------|------|------|
| **模型检查点** | 保存权重、优化器状态 | 每 1-8 小时 |
| **训练状态检查点** | 保存学习率、数据位置 | 同步保存 |
| **自动重启** | 故障后从检查点恢复 | 自动化 |

**检查点内容**：
- 模型参数（FP32 主权重）
- 优化器状态（Adam 矩估计）
- 学习率调度器状态
- 数据加载器位置（epoch、sample index）
