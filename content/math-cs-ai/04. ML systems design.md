# ML 系统设计

*ML 系统设计将文件 01-03 的基础设施模式应用于机器学习的特定挑战。本文件涵盖 ML 生命周期、数据管理、训练基础设施、模型评估、服务策略、特征工程、ML 管道和监控。*

- 系统设计面试问题如"为 YouTube 设计推荐系统"不是让你描述推荐算法。它让你设计**整个系统**：数据管道、特征工程、模型训练、评估、服务、监控和迭代。本文件提供框架。

## ML 系统生命周期

![ML 系统生命周期：问题框架到部署和监控，持续迭代](../images/ml_lifecycle.svg)

- 每个 ML 系统遵循相同的生命周期，无论是垃圾邮件分类器还是基础模型：

```
问题框架 → 数据 → 特征 → 训练 → 评估 → 部署 → 监控 → 迭代
       ↑                                                                                    │
       └────────────────────────────────────────────────────────────────────────────────────┘
```

### 问题框架

- 在接触数据或模型之前，定义：
    - **什么**是你预测的？（点击概率、下一个标记、对象边界框）
    - **谁**是用户？（最终用户、内部分析师、其他 ML 模型）
    - **约束是什么？**（延迟 < 100ms、离线批处理可以、必须在设备上运行）
    - **业务指标是什么？**（收入、参与度、准确性）以及 ML 指标如何与之关联？
    - **基线是什么？**（启发式、基于规则的系统、现有模型）—— 你必须击败这个以证明 ML 系统合理。

- **常见错误**：在理解问题之前跳到模型架构。"我们应该使用变压器"不是系统设计答案。"我们需要在 200ms 内预测 10M 候选的点击概率，所以我们需要两阶段系统：快速检索然后小型排名模型"是。

## 数据管理

### 数据收集和标注

- **显式标签**：人类标注数据（点击/未点击、对象边界框、对话质量评级）。昂贵（每个标签约$0.02-$10，取决于复杂度）、慢且主观。

- **隐式标签**：从用户行为推导标签。点击、停留时间、购买、跳过。便宜且丰富但有噪声（点击不意味着满意；跳过不意味着不喜欢）。

- **程序化标注**（Snorkel）：编写标注函数（启发式、正则表达式、现有模型），对每个示例投票。统计聚合投票产生概率标签。扩展到数百万示例，中等准确性。

- **主动学习**：模型识别它最不确定的示例并请求这些的人类标签。这最大化标签效率：1000 个主动选择的标签可以匹配 10,000 个随机标签。

### 数据质量

- **数据验证**：检查每批传入数据的模式违规（缺失字段、错误类型）、分布偏移（平均值显著变化）和体积异常（期望 1M 行，得到 500K）。

- **Great Expectations** 和 **TFX Data Validation** 是定义数据期望并在违规时警报的工具。

- **数据版本控制**：每个训练运行应该是可重现的。**DVC**（第 15 章）跟踪数据文件和代码。每个数据集版本获得哈希；训练配置引用哈希。

### 特征存储

![特征存储：相同特征计算供给离线存储（用于训练）和在线存储（用于服务），防止训练 - 服务偏移](../images/feature_store.svg)

- **特征存储**（第 15 章）为训练和服务提供一致特征。关键概念：

    - **离线特征**：从批处理管道（Spark）计算，存储在数据仓库中。训练期间和批处理推理时使用。示例：用户 30 天平均会话长度、项目总购买数量。

    - **在线特征**：实时计算或预计算并从低延迟存储（Redis、DynamoDB）服务。实时推理时使用。示例：用户最后 5 个动作、当前购物车内容。

    - **训练 - 服务偏移**：如果特征计算在训练和服务之间不同，模型在推理时看到的特征值与训练时不同。特征存储通过对两者使用相同计算消除这个。

## 训练基础设施

- 对于本书读者，分布式训练在第 6 章深入涵盖（数据并行、模型并行、混合精度、缩放定律）。这里我们关注**系统**方面：

- **实验跟踪**（W&B、MLflow — 第 15 章）：每个训练运行记录超参数、指标、git 提交、数据版本和硬件。这是 ML 等效的模型版本控制。

- **超参数调优**：超参数自动搜索。方法：网格搜索（穷举、昂贵）、随机搜索（惊人地有效）、贝叶斯优化（对目标建模、在可能改进的地方采样）和 **ASHA**（异步连续减半：开始许多试验，提前终止表现不佳的）。

- **训练管道编排**（Airflow、Kubeflow — 第 15 章）：自动化数据准备 → 训练 → 评估 → 注册序列。安排每日重新训练。失败时警报。

## 模型评估

### 离线评估

- **保留测试集**：在模型训练期间从未见过的数据上评估。标准但如果测试集不代表生产数据可能误导。

- **基于切片的评估**：在子组上评估（按用户人口统计、内容类型、语言、时间段）。总体准确性 95% 的模型对特定少数群体可能只有 70% 准确性 — 不可接受。

- **回测**：对于时间序列或序列预测，按时间顺序在历史数据上评估。在时间$t$ 之前训练数据，在时间$t$ 到$t + \Delta t$ 数据上评估。避免使用未来数据训练的泄漏。

### 在线评估

![A/B 测试：随机分割用户到对照组（旧模型）和处理组（新模型），带统计显著性比较指标](../images/ab_testing.svg)

- **A/B 测试**：随机分割实时流量到对照组（旧模型）和处理组（新模型）。带统计显著性比较业务指标（收入、参与度、保留）。评估 ML 变更的黄金标准。

    - **样本大小**：你需要足够数据检测预期效应大小。点击率 0.1% 改进需要数百万展示才能显著检测。

    - **持续时间**：运行至少一个完整周期（大多数产品 1-2 周）以捕获星期几效应。

    - **护栏指标**：监控不应变化的指标（页面加载时间、错误率、崩溃率）与目标指标一起。增加收入但也增加崩溃的模型是净负面。

- **影子部署**：在生产中与新模型一起运行旧模型。两者接收相同请求，但只有旧模型的预测服务给用户。比较输出。这在无用户风险情况下捕获 bug 和质量问题。

- **交错**：对于排名问题，在单个列表中交错旧和新模型的结果。用户与交错列表交互，你测量哪个模型的结果获得更多参与。比 A/B 测试需要更少用户达到显著性。

## 模型服务

### 批处理与实时

- **批处理推理**：预计算所有可能输入的预测。存储在数据库/缓存中。从缓存服务。适用于：输入空间有限（每晚为所有用户推荐）、新鲜度不关键（每日预测可以）、延迟容忍度高。

- **实时推理**：按需为每个请求计算预测。适用于：输入空间无限（任何用户查询）、新鲜度重要（立即为此特定查询预测）、延迟必须低。

- 许多系统使用**两者**：批处理预计算一组候选（便宜、覆盖 80% 流量），实时处理其余（昂贵、覆盖长尾查询和新用户）。

### 模型版本控制和注册表

- **模型注册表**（MLflow、W&B、SageMaker）存储训练模型带元数据：
    - 版本号和训练日期。
    - 训练配置和数据版本。
    - 评估指标（准确性、延迟、内存使用）。
    - 阶段：开发 → 暂存 → 生产 → 归档。

- **回滚**：如果新模型在生产中降低指标，立即恢复到一个先前版本。注册表使这个成为一键操作。

## 特征工程

- **特征工程**将原始数据转换为模型需要的输入。它通常是 ML 中最高杠杆活动：更好特征改进每个模型，而更好模型受它们接收的特征限制。

### 在线与离线特征

- **离线特征**预计算且变化慢（用户人口统计、30 天聚合）。由批处理管道（Spark）计算，存储在特征存储中。

- **在线特征**反映当前状态且变化快（购物车中的项目、最后动作、当前位置）。从事件流实时计算或从快速存储查找。

- **特征新鲜度**：一些特征需要秒级新鲜（欺诈检测：给定最后 5 个交易，此交易是否异常？）。其他可以是小时级陈旧（推荐：基于历史，此用户偏好什么类型？）。更新鲜特征计算和服务更昂贵。

### 常见特征模式

- **计数特征**：时间窗口内事件计数（过去 7 天购买、过去 24 小时登录）。
- **嵌入特征**：分类变量的学习嵌入（用户嵌入、项目嵌入、查询嵌入）。这些是双塔模型和类似架构的输入。
- **交叉特征**：两个或更多特征的组合（user_age × item_category）。捕获单个特征错过的交互。
- **时间特征**：自最后动作以来的时间、星期几、一天中的小时。捕获时间模式。
- **聚合特征**：组上数值特征的平均值、中位数、最小值、最大值、标准差（此卖家项目的平均评级）。

## ML 管道

- ML 管道编排从数据到部署模型的整个工作流：

```
数据摄入 → 验证 → 特征工程 → 训练 → 评估 → 注册 → 部署 → 监控
```

- 每个步骤是编排器中的任务（Airflow、Kubeflow、Metaflow — 第 15 章）。管道：
    - 按计划运行（每日重新训练）或触发器（新数据可用）。
    - 是幂等的（重新运行产生相同结果）。
    - 有重试逻辑（如果特征计算失败，重试 3 次带退避）。
    - 产生工件（训练模型、评估报告、特征统计），版本化并存储。

- **Metaflow**（Netflix/Outerbounds）特别适合 ML：它将代码、数据和模型一起版本化，支持本地开发和云执行使用相同代码，并与 K8s 和 AWS 集成。

## 监控

- 我们在第 15 章涵盖监控基础（Prometheus、Grafana、警报）。这里我们关注**ML 特定监控**：

### 数据漂移

- **数据漂移**发生在传入数据分布相对于训练数据变化时。在夏季数据上训练的模型在冬季数据上可能表现差（不同用户行为、不同产品可用性）。

- **检测**：使用统计测试将传入特征分布与训练分布比较：
    - **KS 检验**（Kolmogorov-Smirnov）：比较两个经验分布。测试它们是否来自相同底层分布。
    - **PSI**（群体稳定性指数）：测量分布偏移多少。PSI < 0.1 稳定，0.1-0.25 中等偏移，> 0.25 显著。
    - **嵌入漂移**：使用质心距离或 MMD（最大均值差异）将传入查询的嵌入分布与训练集比较。

### 概念漂移

- **概念漂移**发生在输入和输出之间关系变化时。特征看起来相同，但正确预测不同。示例：用户偏好在文化事件、大流行或产品变化后转移。

- 概念漂移比数据漂移更难检测，因为它需要标注数据。监控代理指标：点击率、转化率、用户满意度分数。持续下降表明概念漂移。

### 模型退化

- 模型随时间退化因多种原因：数据漂移、概念漂移、特征管道 bug（特征开始返回 null）和上游数据变化（第三方 API 更改其响应格式）。

- **响应**：检测到退化时，行动取决于严重性：
    - 轻度：在近期数据上重新训练（计划重新训练处理这个）。
    - 中度：调查根本原因（哪个特征变化？哪个用户段受影响？）。
    - 重度：立即回滚到先前模型版本，然后调查。

### 反馈循环

- ML 系统创建**反馈循环**：模型的预测影响用户行为，成为下一模型版本的训练数据。这些循环可以是良性或恶性。

- **正反馈循环**（危险）：推荐模型显示大多数热门项目 → 用户点击热门项目（因为他们只看到这些） → 模型学习热门项目甚至更受欢迎 → 多样性崩溃。模型创建确认其偏倚的数据。

- **负反馈循环**（也危险）：欺诈检测模型捕获所有 A 类欺诈 → 无 A 类欺诈到达训练数据 → 下一模型不学习检测 A 类 → A 类欺诈重新出现。

- **缓解**：
    - **探索**：显示一些模型不确定的项目（epsilon-greedy、Thompson 采样）。这生成多样化训练数据。
    - **反事实日志记录**：记录模型*本会*预测什么，不仅用户看到什么。在反事实数据上训练以去偏。
    - **保留集**：随机服务一部分流量无模型过滤。未过滤数据提供评估模型质量的真实情况。
    - **延迟标签**：在使用数据训练前等待真实结果。今天点击的推荐可能明天后悔。欺诈预测必须等待拒付窗口（30-90 天）。

### 嵌入表管理

- 大规模 ML 系统通常有 1 亿 + 条目的嵌入表（每个用户、项目、广告或实体一个嵌入）。大规模管理这些是系统挑战：

- **存储**：1 亿实体 × 256 维 × float16 = 50 GB。不适合 GPU 内存。解决方案：存储在 CPU 内存带 GPU 端缓存、跨多台机器分片或使用**哈希嵌入**（哈希实体到固定大小表，接受冲突）。

- **更新**：嵌入随模型重新训练变化。部署新嵌入表到服务需要：加载 50 GB 到内存不中断实时流量、验证正确性、如果指标退化则回滚。对嵌入表使用蓝绿部署。

- **陈旧**：新创建用户无嵌入（冷启动问题）。解决方案：使用默认嵌入、通过特征到嵌入模型从用户特征推导嵌入、或回退到非个性化模型。

### 公平性和偏倚

- ML 系统可以系统地对不同组区别对待，通常反映训练数据中的偏倚。**公平性监控**是责任，不是可选特性。

- **监控指标**：
    - **人口统计 parity**：正预测率在不同组之间是否不同（性别、种族、年龄）？
    - **平等机会**：真正率在不同组之间是否不同？（招聘模型应该同样擅长识别来自所有组的合格候选人。）
    - **校准**：如果模型说组 A 的 P(qualified) = 0.7，组 A 实际 70% 合格吗？组 B 相同吗？

- **实际步骤**：
    - 在切片（子组）上评估模型性能，不仅总体指标。
    - 在模型评估管道中包含公平性指标（改进总体准确性但降低特定组的模型不应在无审查情况下部署）。
    - 记录已知限制和故障模式。
    - 为部署在敏感域（招聘、贷款、刑事司法、医疗保健）的模型建立审查流程。
