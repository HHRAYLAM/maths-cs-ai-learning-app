# 边缘推理

*边缘推理在用户设备（手机、笔记本电脑、IoT 传感器）上运行模型，无需将数据发送到云端。本文件涵盖边缘约束、模型压缩管道、设备端运行时、编译器栈、硬件目标（NPU、Neural Engine）、设备端 LLM、联邦学习和延迟优化*

- 云推理需要网络连接，增加延迟（50-200 毫秒往返），每次请求花费金钱，并将用户数据发送到第三方服务器。**边缘推理**消除所有四个：模型本地运行，即时响应，每次推理零成本，并保持数据私密。

- 权衡：边缘设备的计算和内存比数据中心 GPU 少 100-1000 倍。使模型在这些约束内运行需要在每个级别进行激进优化。

- **Cactus**（[github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)）是为移动和可穿戴设备构建的低延迟 AI 引擎。它在生产中展示本文件涵盖的许多技术：用于注意力和矩阵运算的定制 ARM SIMD 内核（第 16 章）、KV-cache 量化（第 17 章文件 01）、分块预填充、Apple 和 Qualcomm 芯片上的 NPU 加速推理、用于 10 倍降低 RAM 使用的零拷贝内存映射，以及当设备端计算不足时自动云回退。Cactus 支持 iOS、Android、macOS 和嵌入式 Linux 上的多模态推理（LLM、视觉、语音），带 Swift、Kotlin、Python、Flutter、React Native 和 Rust 的 SDK。其基准测试显示 M4 Pro 上 1.2B 模型 INT4 解码 100 tokens/s，iPhone 17 Pro 上 48 tokens/s，这是优化边缘推理的具体示例。

## 边缘约束

| 资源 | 云 GPU (H100) | 笔记本电脑 (M4) | 手机 (Snapdragon 8 Gen 3) | IoT (ESP32) |
|----------|-----------------|-------------|---------------------------|-------------|
| RAM | 80 GB HBM3 | 16-36 GB 统一 | 8-12 GB LPDDR5 | 520 KB |
| 计算 | 989 TFLOPS (FP8) | 38 TOPS (Neural Engine) | 45 TOPS (NPU) | 0.001 TOPS |
| 功耗 | 700 W | 15-30 W | 5-10 W | 0.1 W |
| 存储 | TB | 256 GB-2 TB | 128-512 GB | 4 MB |

- 云 GPU 和手机 NPU 之间的计算差距约 20 倍。GPU 和微控制器之间是约 1,000,000 倍。不同设备需要不同级别的压缩和不同模型架构。

## 模型压缩管道

- 对于边缘部署，压缩不是单一技术——它是顺序应用的互补技术**管道**：

```
完整模型（FP32，70B 参数）
    ↓ 知识蒸馏 → 更小模型（7B 参数）
    ↓ 结构化剪枝 → 移除冗余头/层（4B 有效）
    ↓ 量化（INT4）→ 4 倍更小（2 GB）
    ↓ 编译器优化 → 融合内核、优化内存布局
    ↓ 运行时 → 设备端执行
```

- 每个步骤减少大小和延迟。顺序很重要：先蒸馏（减少架构），然后剪枝（移除结构），然后量化（减少精度），然后编译（针对目标硬件优化）。量化后蒸馏会尝试压缩已经损耗的模型。

## 设备端运行时

- **运行时**加载模型、分配内存并在目标硬件上执行推理。每个平台有其首选运行时：

- **ONNX Runtime**：跨平台（Windows、Linux、macOS、iOS、Android）。支持 CPU、GPU（CUDA、DirectML、CoreML、NNAPI）和许多加速器后端。最便携选项。模型从 PyTorch/TensorFlow 导出到 ONNX 格式。

- **TensorFlow Lite (TFLite)**：Google 的边缘运行时。针对 ARM CPU 和 Android NPU 优化。小二进制（约 1 MB）。支持 INT8 和 float16。Android 部署的标准。

- **Core ML**：Apple 用于 iOS/macOS 的运行时。根据模型特征自动使用 Neural Engine、GPU 或 CPU。模型使用 `coremltools` 从 PyTorch/TensorFlow 转换。与 Apple 硬件紧密集成（统一内存、Neural Engine）。

- **ExecuTorch**：Meta 用于设备端 PyTorch 的新运行时。设计用于边缘部署，带预先编译和运算符级委托到硬件加速器。PyTorch Mobile 的继任者。

- **TensorRT**：NVIDIA 用于 GPU 推理优化的运行时（第 15 章）。融合层、选择最佳内核并自动量化。NVIDIA GPU 上比 PyTorch 急切模式快 2-5 倍。

- **llama.cpp**：LLM 的单文件 C++ 推理引擎。支持 GGUF 量化（Q4、Q5、Q8）、CPU（AVX/NEON）、Metal（Apple GPU）、CUDA 和 Vulkan。在消费硬件上运行 LLM 的首选。

## 编译器栈

- 在高级模型（PyTorch 图）和硬件（NPU 指令）之间是**编译器栈**，它针对特定目标优化模型：

```
PyTorch 模型
    ↓ 导出（torch.export、ONNX、TorchScript）
图 IR（中间表示）
    ↓ 图优化
        - 常量折叠（编译时计算常量表达式）
        - 死代码消除（移除未使用运算）
        - 运算符融合（conv + bn + relu → 单个融合运算）
        - 布局转换（NCHW → NHWC 用于 ARM，通道优先）
    ↓ 降低
硬件特定 IR
    ↓ 后端优化
        - 分块和循环排序（缓存友好访问模式）
        - 向量化（SIMD，第 16 章）
        - 内存规划（重用缓冲区以最小化峰值内存）
        - 内核选择（为每个运算选择最佳实现）
    ↓ 代码生成
机器代码 / NPU 指令
```

- **运算符融合**是最有影响力的优化。变压器块有约 20 个运算（matmul、add、layernorm、softmax 等）。无融合时，每个将其输出写入内存，下一个读回。融合时，多个运算组合成单个内核，将数据保留在寄存器/缓存中。这可以快 2-5 倍（第 16 章，屋顶线模型）。

- **内存规划**：编译器分析模型图以确定哪些张量生命周期重叠并可以共享相同内存缓冲区。有 100 个中间张量的模型可能只需要 10 个内存，因为大多数在其他创建之前被消耗和释放。这在 RAM 有限的设备上至关重要。

## 硬件目标

### 移动 GPU

- **Qualcomm Adreno**（Android）：支持 OpenCL、Vulkan 计算（第 16 章）和 Qualcomm 专有的 SNPE（Snapdragon Neural Processing Engine）。Adreno GPU 有 256-1024 个 ALU，带 FP16 和 INT8 支持。

- **ARM Mali**（Android）：支持 OpenCL 和 Vulkan。Mali GPU 使用基于图块的架构（与桌面 GPU 不同），这影响最佳内存访问模式。

- **Apple GPU**（iOS/macOS）：通过 Metal（Apple 的 GPU API）访问。统一内存架构意味着无 CPU↔GPU 复制开销。Metal Performance Shaders（MPS）提供优化的 ML 原语。

### 神经处理单元（NPU）

- NPU 是专为 ML 推理设计的固定功能加速器。对于标准 ML 运算（matmul、conv、激活），它们比 GPU 更节能。

- **Apple Neural Engine**：16 核，约 38 TOPS（INT8）。通过 Core ML 访问。非常适合视觉模型和设备端扩散。不能运行任意代码——只有 Core ML 支持的运算。

- **Qualcomm Hexagon NPU**：集成到 Snapdragon SoC 中。支持 INT8 和 INT4 推理。通过 SNPE 或带 QNN 后端的 ONNX Runtime 访问。支持设备端功能如背景模糊、语音识别和实时翻译。

- **Google Edge TPU**：云 TPU 的小型、低功耗版本。4 TOPS，2W。用于 Coral 设备上的设备端推理。仅支持 INT8 量化 TFLite 模型。

- **委托模式**：运行时在 NPU（用于支持的运算）和 CPU（用于不支持的运算）之间分割模型图。最大化在 NPU 上运行的比例是性能和能效的关键。

## 设备端 LLM

- 使用小模型和激进量化，在手机和笔记本电脑上运行 LLM 变得可行：

| 模型 | 参数 | 量化大小 | 目标设备 | 性能 |
|-------|--------|---------------|---------------|-------------|
| Phi-3 Mini | 3.8B | ~2 GB (Q4) | 手机/笔记本电脑 | ~15 tokens/s on iPhone 15 |
| Gemma 2B | 2B | ~1.5 GB (Q4) | 手机 | ~20 tokens/s on Pixel 8 |
| Llama 3.2 1B | 1B | ~700 MB (Q4) | 手机 | ~30 tokens/s |
| Llama 3.2 3B | 3B | ~2 GB (Q4) | 手机/笔记本电脑 | ~15 tokens/s |
| Llama 3.1 8B | 8B | ~4.5 GB (Q4) | 笔记本电脑 | ~20 tokens/s on M2 |

- **挑战**：
    - **内存**：3B Q4 模型适合 2 GB，但长对话的 KV-cache 显著增加。手机上上下文长度通常限制为 2-4K 标记。
    - **热节流**：持续推理加热手机。连续生成 30 秒后，SoC 节流时钟速度以防止过热，性能降低 30-50%。
    - **电池**：以 15 tokens/s 运行 3B 模型消耗约 3-5W。30 分钟对话消耗典型手机电池约 5%。偶尔使用可以接受，始终在线应用有问题。

- **llama.cpp** 是设备端 LLM 的标准。它运行在 CPU（AVX2、NEON、I8MM）、Apple GPU（Metal）、NVIDIA GPU（CUDA）、AMD GPU（ROCm/Vulkan）甚至手机（通过 Android 上的 Termux）。

## 联邦学习

- **联邦学习**在不集中数据的情况下跨多台设备训练模型。每个设备在本地数据上训练，计算梯度更新，并仅将更新（不是数据）发送到中央服务器聚合更新。

- **算法**（FedAvg）：
    1. 服务器将当前模型发送到$K$个选定设备。
    2. 每个设备在本地数据上微调模型几步。
    3. 每个设备将其更新模型（或差异）发送回服务器。
    4. 服务器平均更新：$W_{\text{new}} = \frac{1}{K} \sum_{k=1}^{K} W_k$。
    5. 重复。

- **隐私**：原始数据从不离开设备。服务器只看到聚合模型更新。**差分隐私**向更新添加噪声，使单个数据点不能从梯度逆向工程。

- **通信效率**：模型更新很大（与模型大小相同）。压缩技术减少这个：**梯度量化**（发送 INT8 梯度而不是 FP32）、**稀疏化**（仅发送最大梯度）和**梯度累积**（做更多本地步骤，少发送）。

- **应用**：Google 的键盘预测（Gboard）、Apple 的语音识别、健康监测（在敏感健康数据上训练而不集中）。

## 延迟优化

- 除压缩外，几种技术减少端到端推理延迟：

- **提前退出**：在中间层添加分类头。如果模型在第 6 层（共 24 层）自信，返回预测而不运行第 7-24 层。简单输入提前退出，困难输入使用完整模型。对于混合简单和困难输入的任务，平均延迟显著下降。

- **模型分区**：在 NPU（matmul 高效）、GPU（不规则运算高效）和 CPU（处理其他所有）之间分割模型。编译器根据分析决定哪些运算去哪里。

- **缓存**：对于重复查询的应用（自动完成、代码完成），缓存最近计算。如果用户输入"How do I"且模型最近生成了"How do I"的完成，可以重用缓存的 KV-cache，完全跳过预填充阶段。

- **推测预取**：预测用户下一步做什么并在他们请求前开始推理。聊天应用可能在用户阅读当前答案时开始生成可能后续问题的响应。

## 编码任务（使用 CoLab 或 notebook）

1. 模拟模型压缩管道。从 float32 模型开始，应用蒸馏（模拟）、剪枝和量化，并跟踪每个步骤的大小。
```python
def compression_pipeline(original_params_M, original_bits=32):
    size_mb = original_params_M * 1e6 * original_bits / 8 / 1e6

    print(f"Original: {original_params_M}M params, {original_bits}-bit → {size_mb:.0f} MB")

    # 步骤 1：知识蒸馏（减少参数）
    distilled_params = original_params_M * 0.15  # 70B → ~10B equivalent
    size_mb = distilled_params * 1e6 * original_bits / 8 / 1e6
    print(f"After distillation ({distilled_params:.0f}M params): {size_mb:.0f} MB")

    # 步骤 2：结构化剪枝（移除剩余 30%）
    pruned_params = distilled_params * 0.7
    size_mb = pruned_params * 1e6 * original_bits / 8 / 1e6
    print(f"After pruning ({pruned_params:.0f}M params): {size_mb:.0f} MB")

    # 步骤 3：INT4 量化
    size_mb = pruned_params * 1e6 * 4 / 8 / 1e6
    print(f"After INT4 quantisation: {size_mb:.0f} MB")

    print(f"Total compression: {original_params_M * 1e6 * original_bits / 8 / 1e6 / size_mb:.0f}x")

print("=== Starting from 70B model ===")
compression_pipeline(70000)

print("\n=== Starting from 7B model ===")
compression_pipeline(7000)
```

2. 估计设备端推理延迟。给定模型运算计数和硬件规格，计算是否满足延迟目标。
```python
def estimate_latency(model_name, params_M, bits, compute_tops, mem_bw_gbs, seq_len=256):
    """估计内存带宽受限模型的标记生成延迟。"""
    # 模型大小（字节）
    model_bytes = params_M * 1e6 * bits / 8

    # 解码是内存受限：每标记必须加载整个模型
    time_per_token_ms = model_bytes / (mem_bw_gbs * 1e9) * 1000

    # 每秒标记数
    tokens_per_sec = 1000 / time_per_token_ms

    print(f"{model_name}: {params_M/1000:.1f}B params @ {bits}-bit = {model_bytes/1e9:.1f} GB")
    print(f"  Memory bandwidth: {mem_bw_gbs} GB/s")
    print(f"  Time per token: {time_per_token_ms:.1f} ms")
    print(f"  Tokens/sec: {tokens_per_sec:.0f}")
    print()

# Apple M2 Pro：200 GB/s 统一内存带宽
print("=== Apple M2 Pro (200 GB/s) ===")
estimate_latency("Llama-7B Q4", 7000, 4, 15.8, 200)
estimate_latency("Llama-7B Q8", 7000, 8, 15.8, 200)
estimate_latency("Llama-70B Q4", 70000, 4, 15.8, 200)

# 手机（Snapdragon 8 Gen 3）：约 50 GB/s LPDDR5
print("=== Snapdragon 8 Gen 3 (50 GB/s) ===")
estimate_latency("Phi-3 Mini Q4", 3800, 4, 45, 50)
estimate_latency("Llama-3B Q4", 3000, 4, 45, 50)
```
