# 代码库设计和模式

*好的代码库设计将研究原型与生产软件区分开来。本文件涵盖项目结构、干净代码原则、与 ML 相关的设计模式、配置管理、日志记录、API 设计和打包。*

- 大多数 ML 代码从 Jupyter notebook 开始。notebook 不断增长，被复制、修改、共享，最终变成充满全局变量、死单元格和魔术数字的不可维护的纠缠。**代码库设计**是组织代码的学科，以便在项目增长时保持可理解和可修改。

- 这不是为了遵循规则而遵循规则。这是关于减少"我想更改 X"和"X 已更改并工作"之间的时间。在设计良好的代码库中，这个时间是几分钟。在设计糟糕的代码库中，它是数天通过无文档的意大利面条式代码的考古。

## 项目结构

- 一致的项目布局让任何人（包括未来的你）立即导航代码库。

```
my_project/
├── src/my_project/       # 源代码（可导入包）
│   ├── __init__.py
│   ├── data/             # 数据加载和预处理
│   │   ├── __init__.py
│   │   ├── dataset.py
│   │   └── transforms.py
│   ├── models/           # 模型架构
│   │   ├── __init__.py
│   │   ├── transformer.py
│   │   └── layers.py
│   ├── training/         # 训练循环、优化器
│   │   ├── __init__.py
│   │   ├── trainer.py
│   │   └── losses.py
│   └── utils/            # 共享工具
│       ├── __init__.py
│       └── logging.py
├── configs/              # 配置文件
│   ├── base.yaml
│   └── experiment_1.yaml
├── scripts/              # 入口点（训练、评估、服务）
│   ├── train.py
│   ├── evaluate.py
│   └── serve.py
├── tests/                # 测试文件（镜像 src/ 结构）
│   ├── test_dataset.py
│   ├── test_model.py
│   └── test_trainer.py
├── notebooks/            # 仅探索（非生产代码）
├── pyproject.toml        # 项目元数据和依赖
├── README.md
├── .gitignore
└── Dockerfile
```

- **`src/` 布局**：将源代码放在 `src/my_project/` 下防止从当前目录意外导入（这会掩盖在生产环境中出现的导入错误）。用 `pip install -e .` 开发安装。

- **单体仓库 vs 多仓库**：**单体仓库**将所有相关项目保持在一个仓库中（更容易跨项目更改、共享 CI）。**多仓库**给每个项目自己的仓库（更清晰的边界、独立版本控制）。大多数 ML 团队从单体仓库开始，如果需要再拆分。

- **脚本 vs 库**：将入口点（`train.py`、`evaluate.py`）保留在 `scripts/` 中。将可重用逻辑保留在 `src/` 中。训练脚本应该约 50 行：解析配置、构建数据集、构建模型、构建训练器、训练。所有复杂性都在库中。

## 干净代码原则

- **命名**：你可以做的最有影响力的事情。名为 `x` 的变量需要你阅读周围代码来理解它。名为 `learning_rate` 的变量是自文档化的。

```python
# 错误
def proc(d, n, lr):
    for i in range(n):
        for k, v in d.items():
            v -= lr * g[k]

# 正确
def update_parameters(parameters, num_steps, learning_rate):
    for step in range(num_steps):
        for name, param in parameters.items():
            param -= learning_rate * gradients[name]
```

- **单一职责原则**：每个函数/类做一件事。名为 `load_data_and_train_model` 的函数在做两件事，应该拆分。这使每个部分可独立测试、可重用和可理解。

- **DRY（不要重复自己）**——但不要过早。如果你复制粘贴代码三次，将其提取为函数。但不要为你只使用过一次的代码创建抽象。过早抽象比重复更糟：它增加复杂性而没有证明的好处。

```python
# 过早抽象（一个用例，过度工程）
class AbstractDataTransformPipelineFactory:
    ...

# 正好（直接、清晰、在三个地方使用）
def normalise_image(image, mean, std):
    return (image - mean) / std
```

- **魔术数字**：永远不要使用无法解释的字面值。

```python
# 错误
if len(batch) > 32:
    split_batch(batch, 32)

# 正确
MAX_BATCH_SIZE = 32
if len(batch) > MAX_BATCH_SIZE:
    split_batch(batch, MAX_BATCH_SIZE)
```

- **函数应该简短**：如果函数不适合一个屏幕（~30 行），它可能做得太多了。将逻辑块提取到带描述性名称的辅助函数中。函数体读起来像高级摘要。

## ML 的设计模式

- 设计模式是常见问题的可重用解决方案。这些是与 ML 代码库最相关的：

- **工厂模式**：创建对象而无需指定确切的类。当你的配置说 `model: "transformer"` 且你需要实例化正确的类时有用：

```python
MODEL_REGISTRY = {
    "transformer": TransformerModel,
    "cnn": CNNModel,
    "mlp": MLPModel,
}

def build_model(config):
    model_cls = MODEL_REGISTRY[config["model"]]
    return model_cls(**config["model_params"])
```

- 这将训练脚本与特定模型实现解耦。添加新模型意味着在注册表中添加一行，而非修改训练循环。

- **策略模式**：在运行时交换算法。对损失、优化器、调度器有用：

```python
LOSS_FUNCTIONS = {
    "mse": nn.MSELoss,
    "cross_entropy": nn.CrossEntropyLoss,
    "focal": FocalLoss,
}

loss_fn = LOSS_FUNCTIONS[config["loss"]]()
```

- **观察者模式**（回调/钩子）：让模块对事件反应而无需紧密耦合。训练框架（PyTorch Lightning、Keras）广泛使用回调：

```python
class EarlyStopping:
    def __init__(self, patience=5):
        self.patience = patience
        self.best_loss = float('inf')
        self.counter = 0

    def on_epoch_end(self, epoch, val_loss):
        if val_loss < self.best_loss:
            self.best_loss = val_loss
            self.counter = 0
        else:
            self.counter += 1
            if self.counter >= self.patience:
                return "stop"
```

- **依赖注入**：将依赖传递给函数/类而非在内部创建它们。这使测试容易（注入模拟）和配置灵活：

```python
# 错误：硬编码依赖
class Trainer:
    def __init__(self):
        self.logger = WandbLogger()  # 没有 W&B 无法测试

# 正确：注入依赖
class Trainer:
    def __init__(self, logger):
        self.logger = logger  # 可以注入任何日志器，包括模拟
```

## 配置管理

- 硬编码超参数、文件路径和模型设置使实验不可重现且修改痛苦。**外部化配置**到文件中。

- **YAML** 是 ML 配置最常用的格式：

```yaml
# configs/experiment_1.yaml
model:
  name: transformer
  d_model: 512
  n_heads: 8
  n_layers: 6

training:
  batch_size: 64
  learning_rate: 3e-4
  max_epochs: 100
  early_stopping_patience: 10

data:
  train_path: /data/train.parquet
  val_path: /data/val.parquet
  max_seq_length: 512
```

- **Hydra** (Facebook) 是支持组合的配置框架（合并基础配置与实验特定覆盖）、命令行覆盖（`python train.py training.lr=1e-3`）和多运行（超参数扫描）。

- **argparse** 对带几个参数的脚本更简单：

```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--lr", type=float, default=3e-4)
parser.add_argument("--batch-size", type=int, default=64)
parser.add_argument("--config", type=str, default="configs/base.yaml")
args = parser.parse_args()
```

- **最佳实践**：有带所有默认值的基础配置，以及每个实验的配置仅覆盖变化的部分。跟踪每个实验的配置及其结果。

## 日志记录和可观察性

- `print` 语句用于调试。**日志记录**用于生产：

```python
import logging

logger = logging.getLogger(__name__)
logger.setLevel(logging.INFO)

logger.debug("批次加载：%d 个样本", len(batch))     # 嘈杂，用于调试
logger.info("第 %d 轮：损失=%.4f, 学习率=%.6f", epoch, loss, lr)  # 正常操作
logger.warning("GPU 内存 >90%%，考虑减少批次大小")
logger.error("加载检查点失败：%s", path)       # 可恢复错误
logger.critical("CUDA 内存不足，中止")            # 致命
```

- **为什么不用 print**：日志记录支持级别（在生产中过滤调试消息）、格式化（时间戳、模块名）和处理器（写入文件、发送到监控系统）而无需更改日志调用。

- **结构化日志记录**输出机器可解析格式（JSON）以及人类可读消息。这支持搜索和对特定字段告警：

```python
logger.info("training_step", extra={
    "epoch": 5, "step": 1200, "loss": 0.0342, "lr": 2.1e-4
})
```

## API 设计

- 如果你的模型将被其他服务使用（web 应用、移动应用、另一个 ML 流水线），它需要**API**（应用程序编程接口）。

- **REST API** 使用 HTTP 方法：`GET` 读取、`POST` 创建/预测、`PUT` 更新、`DELETE` 删除。端点遵循基于资源的命名：

```
POST /api/v1/predict          # 发送输入，获取预测
GET  /api/v1/models           # 列出可用模型
GET  /api/v1/models/{id}      # 获取模型详情
POST /api/v1/models/{id}/predict  # 用特定模型预测
```

- **FastAPI** 是 ML 服务的首选 Python 框架：

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class PredictRequest(BaseModel):
    text: str

class PredictResponse(BaseModel):
    label: str
    confidence: float

@app.post("/predict", response_model=PredictResponse)
async def predict(request: PredictRequest):
    result = model.predict(request.text)
    return PredictResponse(label=result.label, confidence=result.score)
```

- FastAPI 自动生成 API 文档（Swagger UI 在 `/docs`）、用 Pydantic 模型验证输入/输出，并支持 async 高吞吐量。

- **gRPC** 对内部服务间服务通信比 REST 快。它使用 Protocol Buffers（二进制序列化，比 JSON 更小更快）并支持流式。被 TensorFlow Serving、Triton Inference Server 和许多微服务架构使用。

## 打包和分发

- 使你的代码可作为包安装让其他人（和你自己的脚本）干净地导入它：

```toml
# pyproject.toml
[project]
name = "my-ml-project"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.0",
    "jax>=0.4",
    "pydantic>=2.0",
]

[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

[build-system]
requires = ["setuptools>=64"]
build-backend = "setuptools.backends._legacy:_Backend"
```

```bash
pip install -e ".[dev]"    # 可编辑模式安装带开发依赖
```

- **可编辑安装**（`-e`）：对源代码的更改立即反映而无需重新安装。开发期间必不可少。

- **固定依赖**：带确切版本的 `requirements.txt`（`torch==2.2.1`，而非 `torch>=2.0`）确保可重现性。用 `pip freeze > requirements.txt` 捕获当前环境。对更复杂的依赖管理，使用 `uv`、`poetry` 或 `pip-tools`。

## 与 AI 编码代理协作

- AI 编码代理（Claude Code、GitHub Copilot、Cursor 等）现在是专业工程工作流的一部分。用好，它们显著加速开发。用不好，它们引入细微 bug、侵蚀你对自己代码库的理解，并创造生产力的虚假感觉。

- 正确的心智模型：**代理是快但经验不足结对编程伙伴**。它可以快速编写代码、了解语法和标准模式，并读过比你永远能读的更多的文档。但它不理解你的特定系统、你的业务约束、你的边界情况或你设计决策背后的*为什么*。你是高级工程师；代理是初级工程师。你指导、审查并承担责任。

### 代理擅长时

- **样板和脚手架**：生成 Dockerfiles、CI 配置、测试夹具、数据类定义、argparse 设置。这些遵循众所周知的模式且手工编写繁琐。让代理生成它们，然后审查正确性。

- **编写测试**：描述函数的行为，代理生成测试用例。它经常捕获你会错过的边界情况（空输入、负值、Unicode）。始终阅读生成的测试——它们验证你的假设，而不仅是你的代码。

- **重构**："将此块提取为函数"、"将此类转换为使用 dataclasses"、"为此模块添加类型提示"。意图清晰且细微错误风险低的机械转换。

- **探索和原型**："编写快速脚本基准测试推理延迟"或"向我展示如何使用 HuggingFace 分词器 API"。代理让你比阅读文档更快获得工作起点。

- **文档和 docstrings**：代理可以从你的代码结构生成文档。审查准确性，但苦活已自动化。

- **调试辅助**：粘贴错误回溯并请求诊断。代理经常可以识别根本原因并建议修复，尤其对常见问题（形状不匹配、导入错误、CUDA 内存不足）。

### 何时不依赖代理

- **新颖架构决策**：如果你在设计新训练流水线，代理会给出通用答案。它不知道你的数据约束、延迟要求或团队专业知识。使用代理实现你已经深思熟虑的设计。

- **安全关键代码**：认证、加密、输入净化。代理可能生成看起来正确但有细微漏洞的代码（SQL 注入、不安全默认值、时序攻击）。安全代码应由理解威胁模型的人编写，并由另一个人审查。

- **性能关键内部循环**：代理会编写正确但天真的代码。对 GPU 内核、内存关键数据结构或延迟敏感服务路径，你需要了解硬件约束（第 13 章、第 16 章）并故意优化。

- **你不理解的代码**：如果代理生成 200 行且你不能解释每行做什么，不要提交它。你现在维护你不理解的代码，当它坏时（它会），你无法调试它。这是最常见和最危险的故障模式。

### 审查纪律

- **始终阅读每一行**生成的代码再提交。这不是可选的。代理的代码是草稿，而非成品。完全像对待同事的 pull request 一样：严格审查它。

- **检查什么**：
    - **正确性**：它真的做了你要求的吗？代理经常解决与你的意图略有不同的问题。
    - **边界情况**：它处理空输入、None 值、负数、非常大输入吗？代理经常省略边界情况处理。
    - **幻觉 API**：代理可能调用不存在或使用不存在的参数的函数，尤其对更新或较不常见的库。验证每个 API 调用是真实的。
    - **过度工程**：代理倾向于产生比必要更多的代码。10 行问题的 50 行解决方案增加复杂性而无益。无情简化。
    - **安全**：硬编码密钥、未净化用户输入、不安全默认值。代理不对抗思考。
    - **风格一致性**：生成代码匹配你项目的约定（命名、模式、错误处理）吗？

### 如何写好提示

- 代理输出的质量与你的指令质量成正比。模糊提示得到模糊代码。

- **坏**："编写数据加载器"
- **好**："为带'text'和'label'列的 CSV 文件编写 PyTorch DataLoader。用 HuggingFace 分词器'bert-base-uncased'分词文本，max_length=512。返回 input_ids、attention_mask 和 label 为张量。处理 CSV 中标志列有缺失值的情况，跳过那些行。"

- **提供上下文**：告诉代理关于你的项目结构、现有代码、约束和约定。上下文越多，输出越好。

- **指定约束**："仅使用标准库"、"必须与 Python 3.10 工作"、"不要使用全局变量"、"遵循 `src/models/transformer.py` 中的现有模式"。

- **请求解释**："实现 X 并解释关键设计决策"。这迫使代理阐明其推理，使你更容易发现缺陷假设。

### 使用质量门捕获代理错误

- 你现有的质量基础设施（文件 04）捕获代理错误与人类错误一样好：

    - **类型检查（mypy）**：捕获幻觉 API 签名和类型不匹配。
    - **Linting（ruff）**：捕获未使用导入、未定义变量和风格违规。
    - **测试（pytest）**：如果代理的代码通过你的测试套件，它更可能正确。如果你没有测试，在请求代理实现功能*之前*编写它们（测试驱动开发与代理特别有效）。
    - **CI 流水线**：在每次提交自动运行所有上述。

- "代理编写代码" + "质量门验证它"的组合比单独任何一个都更高效。代理快但马虎；门彻底但不编写代码。一起，你获得速度和正确性。

### 生产力陷阱

- 编码代理的最大风险是**生产力的幻觉**。你可以在 10 分钟内生成 500 行代码。但如果你花 2 小时调试那 500 行因为你不理解它们，你比自己在 30 分钟内编写 200 行更慢。

- 与代理的真正生产力来自：
    1. **保持控制**：你决定架构，代理填充实现。
    2. **理解生成的内容**：如果你不能解释它，重写它或要求代理简化。
    3. **投资质量门**：测试、类型和 linting 将它们的成本摊销到每次代理交互。
    4. **用代理弥补你的弱点**：如果你擅长算法但写测试慢，让代理写测试。如果你擅长 UI 代码但不熟悉数据库查询，让代理起草 SQL。发挥你的优势，委托你的差距。

- 从编码代理获得最大收益的工程师是那些已经知道如何编码好的人。代理放大你现有的技能；它不替代它。理解数据结构、算法、系统设计、软件工程（整个这一章）让你有效指导代理并批判性评估其输出。
