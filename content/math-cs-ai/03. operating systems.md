# 操作系统

*操作系统是硬件和应用程序之间的软件层，管理资源、提供抽象和执行隔离。本文件涵盖操作系统做什么、进程、线程、CPU 调度、内存管理、文件系统和系统调用*

- 没有操作系统的计算机就像没有厨师的厨房：食材（硬件）在那里，但没有人协调谁使用炉灶、哪里放盘子、或如何防止两个人抓同一把刀。操作系统就是那个协调者。

- 对于 ML 从业者，操作系统概念解释：为什么 `nvidia-smi` 显示每个进程的 GPU 内存使用、为什么训练因"内存不足"崩溃、为什么 `fork()` 复制你的 Python 进程、为什么 Docker 容器提供隔离环境。

## 操作系统做什么

- 操作系统有三个核心职责：

    - 抽象：在干净接口后隐藏硬件复杂性。程序读写"文件"，无需知道底层存储是 SSD、HDD 还是网络驱动器。它们分配"内存"，无需管理物理 RAM 芯片。它们在"CPU"上运行，无需担心中断和缓存一致性。

    - 资源管理：多个程序共享 CPU、内存、磁盘和网络。操作系统决定谁获得什么、何时、多久。公平、高效的分配策略保持系统响应。

    - 隔离和保护：程序不能互相干扰。Web 浏览器中的 bug 不应该使内核崩溃。恶意程序不应该读取另一个程序的密码。操作系统使用硬件支持（特权级别、虚拟内存）执行边界。

## 进程

- 进程是运行的程序。它是操作系统的基本工作单元。每个进程有：

    - 代码（程序指令，只读）。
    - 数据（全局变量、堆分配）。
    - 栈（函数调用帧、局部变量）。
    - 状态（寄存器值、程序计数器、打开的文件等）。

- 进程控制块（PCB）是操作系统跟踪进程的数据结构。它存储进程 ID（PID）、状态、程序计数器、寄存器内容、内存映射、打开的文件描述符和调度优先级。当操作系统从一个进程切换到另一个进程时，它将当前进程的状态保存到其 PCB，加载下一个进程的状态。这是上下文切换。

- 上下文切换昂贵：保存和恢复寄存器、刷新缓存、使 TLB 条目无效需要微秒。在运行数千个进程的系统上，开销可能显著。这就是为什么每进程服务器架构（如旧的 Apache）被基于线程或事件驱动的架构取代。

- Unix 中的进程创建使用 `fork()` 和 `exec()`：

    - `fork()` 创建当前进程的副本。子进程获得父进程内存、文件描述符和状态的副本。两个进程从同一点继续执行，但 `fork()` 在子进程中返回 0，在父进程中返回子进程的 PID。

    - `exec()` 用新程序替换当前进程的代码。`fork()` 后，子进程通常调用 `exec()` 运行不同的程序。

    - 这个 fork-then-exec 模型优雅：创建新进程（fork）和加载新程序（exec）是独立的操作，可以独立定制。在 fork 和 exec 之间，子进程可以重定向 I/O、更改环境变量或放弃特权。

![进程状态转换：new → ready → running → blocked/terminated，带抢占和 I/O 等待](../images/process_states.svg)

- 进程状态：进程处于几个状态之一：
    - Running（运行）：当前在 CPU 核心上执行。
    - Ready（就绪）：等待 CPU 核心（可运行但尚未调度）。
    - Blocked（阻塞/等待）：不能继续，直到某些事件发生（I/O 完成、锁获取、定时器到期）。
    - Terminated（终止）：完成执行，等待父进程收集其退出状态。

## 线程

- 线程是进程内的轻量级执行单元。进程中的所有线程共享相同的代码、数据和堆，但每个有自己的栈和寄存器状态。

- 相对于多个进程的优势：线程共享内存，所以它们之间的通信快（只需读/写共享变量）。进程需要进程间通信（管道、套接字、共享内存映射），这更慢更复杂。

- 劣势：共享内存是危险的。两个线程同时写入同一变量导致竞争条件（结果取决于哪个线程先运行）。这引向同步，在文件 4 中涵盖。

- 内核线程由操作系统调度器管理。每个线程独立调度到 CPU 核心上。创建和切换内核线程涉及系统调用，开销类似于（但小于）进程上下文切换。

- 用户线程（绿线程）由用户空间的运行时库管理，操作系统不可见。它们创建和切换更便宜（不需要系统调用），但一个用户线程的阻塞操作阻塞进程中的所有线程（因为操作系统只看到一个内核线程）。

- 现代系统使用混合模型：许多用户线程映射到较少数量的内核线程（M:N 线程）。Go 的 goroutine 和 Erlang 的进程是用户级线程，由语言运行时调度到 OS 线程上。

- 线程池预先创建固定数量的线程，等待任务。当任务到达时，它分配给空闲线程。这避免为每个任务创建和销毁线程的开销。Web 服务器、数据库引擎和 ML 推理服务器都使用线程池。

## CPU 调度

- 调度器决定哪个进程/线程在每个时刻在哪个 CPU 核心上运行。目标是：最大化 CPU 利用率、最小化响应时间（对于交互式任务）、最大化吞吐量（对于批处理任务）、确保公平。

- 先来先服务（FCFS）：进程按到达顺序运行。简单但 suffers from  convoy effect：长运行进程阻塞所有后面的短进程。

- 短作业优先（SJF）：先运行最短进程。可证明最小化平均等待时间，但需要预先知道作业长度（通常不可能）。抢占版本，最短剩余时间优先（SRTF），如果更短的作业到达，中断运行中的作业。

- 轮转（RR）：每个进程获得固定时间量子（例如 10 ms），然后被抢占并移到队列后面。公平且响应快，但时间量子重要：太小意味着过度上下文切换，太大退化为 FCFS。

- 优先级调度：每个进程有优先级。更高优先级的进程先运行。危险是饥饿：如果高优先级进程不断到达，低优先级进程可能永远不运行。老化解决这个问题：进程等待越久，其优先级越高。

- 多级反馈队列（MLFQ）：多个队列，不同优先级和时间量子。新进程从最高优先级队列开始（短量子）。如果进程使用完整量子（CPU 密集型），它被降级到较低优先级队列（更长量子）。交互式进程自然保持在高优先级队列（它们在用量子前阻塞 I/O）。这适应工作负载，无需预先了解作业类型。

- 完全公平调度器（CFS）：Linux 调度器。它维护进程的红黑树（平衡二叉搜索树），按"虚拟运行时间"排序——它们消耗了多少 CPU 时间。虚拟运行时间最小的进程下一个运行。这确保随时间每个进程获得其公平份额。CFS 每个调度决策以 $O(\log n)$ 运行。

## 内存管理

- 操作系统管理物理 RAM，分配给进程并在不再需要时回收。

- 分页（来自文件 2）将虚拟内存划分为固定大小页面，物理内存划分为帧。页表将页面映射到帧。分页消除外部碎片（分配之间的浪费空间），因为所有页面大小相同。

- 请求分页仅在首次访问时将页面加载到 RAM（不是在进程启动时）。这节省内存：有 1 GB 代码的程序在典型运行期间可能只使用 50 MB。其余从不加载。

- 当 RAM 满且需要新页面时，操作系统必须驱逐现有页面。页面替换算法（LRU、FIFO、时钟，来自文件 2）决定驱逐哪个页面。好的替换最小化页面错误；坏的替换导致抖动。

- 分段将内存划分为可变大小段（代码、数据、栈、堆），每个有自己的基地址和长度。段提供逻辑组织，而分页提供物理管理。现代系统最小化使用分段（主要用于保护），依赖分页进行内存管理。

- 堆是动态分配内存所在（C 中的 `malloc`/`free`、Java 中的 `new`、Python 中隐式）。操作系统将大块内存提供给进程，内存分配器（如 `glibc malloc`、`jemalloc`、`tcmalloc`）将这些块细分为更小的分配。分配器设计影响性能：碎片浪费空间，线程之间的争用浪费时间。

## 文件系统

- 文件系统将持久存储（SSD、HDD）上的数据组织为命名文件和目录的层次结构。

- inode（索引节点）存储文件的元数据：大小、所有权、权限、时间戳和指向磁盘上数据块的指针。文件名存储在目录中，目录将名称映射到 inode 号。这个分离意味着文件可以有多个名称（硬链接）指向同一个 inode。

- FAT（文件分配表）：简单文件系统，用于 USB 驱动器和 SD 卡。表将每个簇（块）映射到文件中的下一个簇，形成链表。简单但不支持权限、日志或大文件。

- ext4：默认 Linux 文件系统。使用 inode，带直接、间接、双重间接和三重间接块指针，处理任何大小的文件。支持扩展（连续块范围），大文件更高效。最大文件大小：16 TB，最大分区：1 EB。

- 日志防止崩溃导致的损坏。在修改文件系统结构之前，更改写入日志（日志）。如果系统在操作中途崩溃，日志在重启时重放，完成或撤销操作。没有日志，写入期间的崩溃可能使文件系统处于不一致状态（文件的數據块更新但其 inode 未更新，或反之）。

- 基于 B 树的文件系统（Btrfs、ZFS）使用 B 树（平衡搜索树）组织数据和元数据，实现高效搜索、写时复制快照和数据完整性的内置校验和。这与数据库索引中使用的 B 树相同。

## 系统调用和内核模式

- 系统调用是用户程序和操作系统内核之间的接口。当程序需要做特权事情（读取文件、分配内存、创建进程、发送网络包）时，它进行系统调用。

- CPU 在两种模式下操作：
    - 用户模式：受限。程序可以执行自己的代码和访问自己的内存，但不能直接访问硬件、其他进程的内存或操作系统数据结构。
    - 内核模式：无限制。操作系统内核可以访问所有硬件和内存。系统调用是从用户模式到内核模式的受控网关。

- 当程序调用 `read()` 时，发生以下情况：
    1. 程序将参数放入寄存器并触发陷阱（软件中断）。
    2. CPU 切换到内核模式并跳转到系统调用处理程序。
    3. 内核验证参数，执行 I/O 操作，将数据复制到用户的缓冲区。
    4. 内核切换回用户模式并返回结果。

- 常见系统调用：`open`、`read`、`write`、`close`（文件），`fork`、`exec`、`wait`、`exit`（进程），`mmap`、`brk`（内存），`socket`、`bind`、`listen`、`accept`（网络）。

- 中断是硬件信号，强制 CPU 暂时停止正在做的事情，运行中断处理程序（在内核中）。键盘按下、网络包到达或定时器滴答都生成中断。定时器中断特别重要：它允许操作系统抢占运行中的进程并切换到另一个（抢占式多任务）。

## 网络基础

- 网络栈是操作系统子系统，实现机器之间的通信。理解它解释分布式训练如何同步梯度、模型服务如何处理请求、为什么延迟重要。

![TCP/IP 栈：应用、传输、网络和链路层，每层添加头部](../images/tcp_ip_layers.svg)

- TCP/IP 模型将网络组织为层，每层为上层提供抽象：

    - 链路层：处理单个物理链路上的通信（以太网、Wi-Fi）。处理 MAC 地址和帧。
    - 网络层（IP）：跨多个网络从源到目的地路由包。每个机器有IP 地址（例如 192.168.1.1 用于 IPv4，或 128 位 IPv6 地址）。路由器根据目标 IP 逐跳转发包。
    - 传输层（TCP/UDP）：提供应用程序之间的端到端通信。
    - 应用层：应用程序直接使用的协议，如 HTTP、DNS、gRPC。

- TCP（传输控制协议）提供可靠、有序交付。它建立连接（三次握手：SYN、SYN-ACK、ACK），保证所有数据按顺序到达（使用序列号和确认），重传丢失的包，控制发送速率，避免压垮网络（拥塞控制）。成本是延迟：握手添加往返，重传添加延迟。

- UDP（用户数据报协议）提供不可靠、无序交付。没有握手、没有重传、没有顺序保证。比 TCP 延迟低得多。用于速度比可靠性更重要的地方：视频流、在线游戏、DNS 查找。在 ML 中，一些梯度同步协议使用基于 UDP 的 RDMA 实现更低延迟。

- 套接字是网络通信的操作系统 API。套接字是端点，由（IP 地址、端口号）标识。服务器创建套接字，将其绑定到端口（例如 80 用于 HTTP），监听连接并接受它们。客户端创建套接字并连接到服务器的地址：端口。然后数据通过套接字读和写，就像文件一样。

- DNS（域名系统）将人类可读名称（google.com）转换为 IP 地址（142.250.80.46）。它是分布式、分层数据库：你的机器询问本地解析器，解析器询问根服务器，根服务器委托给每个域的权威服务器。

- HTTP（超文本传输协议）是 Web 的请求 - 响应协议。客户端发送请求（方法 + URL + 头部 + 可选正文），服务器发送响应（状态码 + 头部 + 正文）。ML 模型服务（如 TensorFlow Serving、Triton）将模型公开为 HTTP 或 gRPC 端点。

- 延迟 vs 带宽：延迟是一个包从源到目的地传输的时间（由物理距离和网络跳数决定）。带宽是数据速率（每秒多少字节）。高带宽、高延迟连接（卫星互联网）可以传输大量数据，但每个字节需要很长时间到达。对于分布式训练，延迟对同步屏障重要（所有 GPU 必须等待最慢的），而带宽对传输大梯度张量重要（第 6 章）。

## 虚拟化和容器

- 虚拟化在单个物理机器上运行多个操作系统。管理程序（VMware、KVM、Xen）创建虚拟机（VM），每个有自己的虚拟 CPU、内存、磁盘和网络接口。每个 VM 运行完整操作系统（客户操作系统），认为它有专用硬件。

- VM 提供强隔离（一个 VM 崩溃不影响其他）和灵活性（在同一机器上运行 Linux 和 Windows，在物理主机之间迁移 VM）。成本是开销：每个 VM 运行完整操作系统内核，消耗内存和 CPU 用于与主机操作系统冗余的操作系统操作。

![VM 在虚拟硬件上运行独立的客户操作系统；容器共享主机内核，更轻量](../images/container_vs_vm.svg)

- 容器（Docker、Podman）提供更轻量的替代方案。容器不是虚拟化整个硬件，而是共享主机操作系统内核，使用内核功能隔离进程：

    - 命名空间隔离进程可以看到的：每个容器获得自己的进程树视图（PID 命名空间）、网络接口（网络命名空间）、文件系统挂载点（挂载命名空间）和主机名（UTS 命名空间）。容器内的进程不能看到其他容器中的进程。

    - Cgroups（控制组）限制进程可以使用的：CPU 时间、内存、磁盘 I/O、网络带宽。容器不能消耗超过其 cgroup 允许的资源，防止一个容器饿死其他。

- 容器在几毫秒内启动（没有操作系统启动），使用最小开销（共享内核），由Dockerfile定义，指定基础镜像、依赖项和命令。这使它们可重现：`docker build` 在任何地方产生相同的环境。

- 对于 ML，容器解决"在我的机器上工作"问题。带特定版本 CUDA、cuDNN、PyTorch 和 Python 的训练环境打包为容器镜像。任何人都可以在任何机器上重现确切环境。云训练平台（AWS SageMaker、GCP Vertex AI）在容器中运行训练作业。

- Kubernetes（K8s）大规模编排容器：它将容器调度到机器集群上、重新启动失败的容器、根据负载扩展/缩小、管理容器之间的网络。大规模 ML 服务（数千个模型副本处理数百万请求）在 Kubernetes 上运行。

## 安全基础

- 操作系统通过多种机制执行安全：

- 权限：每个文件有所有者、组和权限位（所有者、组和其他人的读/写/执行）。进程以启动它的用户的身份（UID）运行，只能访问权限位允许的文件。root用户（UID 0）绕过所有权限检查，这就是为什么以 root 身份运行是危险的。

- 特权分离：进程以所需的最小权限运行。Web 服务器不需要 root 访问；它应该作为受限用户运行，只能读取 Web 文件并绑定到端口 80。如果服务器被攻破，攻击者的访问限于受限用户可以做的。

- 沙箱：限制进程可以做的，超出文件权限。seccomp（Linux）限制进程可以进行哪些系统调用。AppArmor和SELinux定义强制访问控制策略。容器结合命名空间、cgroups 和 seccomp，实现多层隔离。

- 地址空间布局随机化（ASLR）：每次程序运行时随机化栈、堆和库的内存位置。这使攻击者更难利用内存损坏 bug（缓冲区溢出），因为他们无法预测代码或数据在内存中的位置。

- 安全是系统范围的关注点：链条只和最弱的链接一样强。模型服务系统需要安全网络通信（TLS/HTTPS）、经过身份验证的 API 访问（API 密钥、OAuth）、输入验证（防止对抗性输入）和隔离执行（带最小权限的容器）。

1. 探索进程创建。使用 Python 的 `os.fork()`（仅 Unix）创建子进程，观察父进程和子进程从同一点继续。

```python
import os

pid = os.fork()

if pid == 0:
    # 子进程
    print(f"子进程：我的 PID 是 {os.getpid()}，父进程 PID 是 {os.getppid()}")
else:
    # 父进程
    print(f"父进程：我的 PID 是 {os.getpid()}，子进程 PID 是 {pid}")
    os.wait()  # 等待子进程完成
```

2. 模拟轮转调度。给定带运行时间的进程列表，模拟调度并计算平均等待时间。
```python
def round_robin(processes, quantum=3):
    """模拟轮转调度。
    processes: (name, burst_time) 元组列表。
    """
    queue = [(name, burst, 0) for name, burst in processes]  # (name, remaining, wait)
    time = 0
    log = []

    while queue:
        name, remaining, waited = queue.pop(0)
        waited += (time - waited - (processes[[p[0] for p in processes].index(name)][1] - remaining))
        run_time = min(quantum, remaining)
        log.append(f"  t={time:3d}: {name} 运行 {run_time} (剩余：{remaining - run_time})")
        time += run_time
        remaining -= run_time

        if remaining > 0:
            queue.append((name, remaining, time))
        else:
            log.append(f"  t={time:3d}: {name} 完成 (周转：{time})")

    for line in log:
        print(line)

print("轮转 (quantum=3)：")
round_robin([("P1", 10), ("P2", 4), ("P3", 6)], quantum=3)
```

3. 模拟带 LRU 的页面替换。给定页面访问序列和固定数量的帧，计算页面错误。
```python
def lru_page_replacement(pages, n_frames):
    """模拟 LRU 页面替换。"""
    frames = []
    faults = 0

    for page in pages:
        if page in frames:
            frames.remove(page)
            frames.append(page)  # 移到最近使用
            status = "HIT "
        else:
            faults += 1
            if len(frames) >= n_frames:
                evicted = frames.pop(0)  # 移除最近最少使用
                status = f"MISS (驱逐 {evicted})"
            else:
                status = "MISS (冷)"
            frames.append(page)
        print(f"  页面 {page}: {status}  frames={frames}")

    print(f"\n总错误：{faults}/{len(pages)} ({faults/len(pages):.0%})")

print("带 3 帧的 LRU：")
lru_page_replacement([1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5], n_frames=3)
```
