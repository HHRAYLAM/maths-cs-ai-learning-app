# 部署和 DevOps

*部署是你的模型停止成为研究工件并开始成为产品的地方。本文件涵盖 ML 的 Docker、模型服务、实验跟踪、可重复性、生产监控、特征存储和管道编排，这些是将训练好的模型从 notebook 带到数百万用户的基础设施。*

- 仅在你的笔记本电脑上运行的模型是原型。可靠地在大规模运行、在毫秒内提供预测、从故障恢复并可以在无停机情况下更新的模型是产品。两者之间的差距是**部署和 DevOps**。

- 大多数 ML 工程师花在部署、监控和调试生产问题上的时间比训练模型多。对于任何构建真实 ML 系统的人来说，理解这个基础设施不是可选的。

## ML 的 Docker

- 我们在第 13 章（操作系统）中概念性地涵盖了容器。这里我们关注实践方面：为 ML 工作负载编写 Dockerfile。

- **Dockerfile** 是构建容器镜像的配方：

```dockerfile
# 从官方 CUDA 基础镜像开始
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04

# 系统依赖
RUN apt-get update && apt-get install -y \
    python3.11 python3-pip git \
    && rm -rf /var/lib/apt/lists/*

# Python 依赖（单独安装以利用缓存）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制源代码（频繁变化，所以这层最后）
COPY src/ /app/src/
COPY configs/ /app/configs/
WORKDIR /app

# 入口点
CMD ["python3", "src/scripts/serve.py", "--config", "configs/serve.yaml"]
```

- **层缓存**：Docker 缓存每一层。如果 `requirements.txt` 未变化，重建时跳过 `pip install`。将不常变化的层（系统包、pip 安装）放在频繁变化的层（源代码）之前。这将 10 分钟的构建变成 10 秒的重建。

- **GPU 访问**：使用 `nvidia/cuda` 基础镜像并用 `docker run --gpus all` 运行。`nvidia-container-toolkit` 提供从主机到容器的 GPU 直通。

- **多阶段构建**通过分离构建环境和运行时减少镜像大小：

```dockerfile
# 构建阶段：安装构建工具，编译依赖
FROM python:3.11 AS builder
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# 运行时阶段：仅运行时依赖
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04
COPY --from=builder /root/.local /root/.local
COPY src/ /app/src/
ENV PATH=/root/.local/bin:$PATH
```

- 最终镜像仅包含运行时库，不包含编译器、头文件或构建工具。5 GB 构建镜像变成 2 GB 运行时镜像。

- **Docker Compose** 运行多容器设置（模型服务器 + 负载均衡器 + 监控）：

```yaml
# docker-compose.yml
services:
  model:
    build: .
    ports:
      - "8080:8080"
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
```

## 模型服务

- **模型服务**是将推理作为服务运行：接收请求，运行模型，返回预测。

- **FastAPI**（文件 03 涵盖）对于低到中等吞吐量是最简单的方法。对于高吞吐量和 GPU 优化服务，使用专用工具：

- **Triton 推理服务器**（NVIDIA）：以 TensorRT、ONNX、PyTorch 和 TensorFlow 格式服务模型。功能：
    - **动态批处理**：收集单个请求并将它们批处理以提高 GPU 效率。单个请求流被批处理成 32 个一组，显著提高吞吐量。
    - **模型集成**：在单个请求中链接多个模型（预处理器 → 模型 → 后处理器）。
    - **多模型服务**：在同一 GPU 上服务多个模型，共享资源。
    - **并发模型执行**：在同一 GPU 上并行运行多个推理请求。

- **TorchServe**（PyTorch）：用 REST/gRPC API 服务 PyTorch 模型。支持模型版本控制、A/B 测试和自定义处理器。

- **vLLM**：专为 LLM 服务设计。实现 PagedAttention（高效 KV 缓存管理）、连续批处理和跨 GPU 张量并行。对于大语言模型实现比朴素服务高 10-20 倍的吞吐量。

- **Cactus**（[github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)）：用于移动和边缘设备上设备服务的低延迟 AI 引擎。Cactus 提供**OpenAI 兼容 API**（聊天补全、流式传输、工具调用、转录、嵌入、RAG、视觉）完全在设备上运行，当本地模型无法处理请求时具有自动**云回退**。这个混合架构意味着你的应用程序代码使用相同的 API，无论推理是在本地还是云端运行 — 引擎根据模型置信度和设备能力决定。Python、Swift、Kotlin、Flutter、React Native 和 Rust 可用 SDK，HuggingFace 上有预转换的模型权重。支持多模态推理（LLM、视觉、语音），具有自定义 ARM SIMD 内核用于 ARM CPU 上最快推理和零复制内存映射用于 10 倍更低 RAM 使用（第 16 章，第 17 章）。

- **模型格式优化**：
    - **ONNX**：用于互操作性的开放格式。从 PyTorch/TensorFlow 导出，在任何地方运行。
    - **TensorRT**：NVIDIA 的优化器。融合层、选择最优内核、量化权重。通常比 NVIDIA GPU 上的 PyTorch 快 2-5 倍。
    - **GGUF/GGML**：用于 CPU 高效推理的格式，常用于在消费级硬件上运行 LLM。

## 实验跟踪

- 没有实验跟踪，ML 研究退化为："我想上周二那个模型是我改动过某些配置后最好的，但我不记得我改了什么。"

- **Weights & Biases (W&B)**：最流行的实验跟踪器。从训练脚本记录任何内容：

```python
import wandb

wandb.init(project="my-project", config={
    "model": "transformer",
    "lr": 3e-4,
    "batch_size": 64,
})

for epoch in range(num_epochs):
    train_loss = train_one_epoch()
    val_loss = validate()

    wandb.log({
        "train/loss": train_loss,
        "val/loss": val_loss,
        "epoch": epoch,
    })

    # 将模型记录为 artifact
    if val_loss < best_loss:
        wandb.save("best_model.pt")

wandb.finish()
```

- W&B 提供：用于比较运行的仪表盘、超参数扫描工具、模型注册表、数据集版本控制和团队协作。

- **MLflow**：开源替代方案。本地运行或在服务器上运行：

```python
import mlflow

mlflow.set_experiment("my-experiment")

with mlflow.start_run():
    mlflow.log_params({"lr": 3e-4, "batch_size": 64})
    mlflow.log_metric("val_loss", 0.042, step=epoch)
    mlflow.pytorch.log_model(model, "model")
```

- **模型注册表**：具有版本控制、分期（dev → staging → production）和元数据的训练模型中央存储。W&B 和 MLflow 都提供注册表。注册表回答："当前生产中是什么模型，谁训练的，它的验证准确率是什么，什么代码/数据产生的？"

## 可重复性

- 可重复性意味着：给定相同的代码、数据和配置，产生相同的模型。这在 ML 中令人惊讶地困难，因为 GPU 操作、数据洗牌和浮点累加中的非确定性。

- **可重复性检查清单**：

| 什么 | 如何 |
|------|-----|
| 代码版本 | Git 提交哈希 |
| 配置/超参数 | 配置文件（在 git 中版本控制或记录到 W&B） |
| 随机种子 | 设置并记录所有种子（Python、NumPy、PyTorch、CUDA） |
| 数据版本 | DVC 哈希、数据集版本标签或 S3 对象版本 |
| 依赖 | `pip freeze`、Docker 镜像哈希或锁定文件 |
| 硬件 | GPU 类型、GPU 数量、CUDA 版本 |
| 非确定性 | `torch.backends.cudnn.deterministic = True`（更慢但可重复） |

- **固定一切**：`pip install torch==2.2.1` 而非 `torch>=2.0`。小版本 bump 可以改变数值行为、优化器实现或默认超参数。

- **Docker 用于可重复性**：Docker 镜像固定操作系统、系统库、Python 版本和 pip 包。镜像哈希是完整环境指纹。如果你能重现 Docker 镜像，你就能重现训练。

## 生产监控

- 部署模型不是结束 — 它是一组新问题的开始。随着现实世界变化（**概念漂移**）和输入数据分布变化（**数据漂移**），模型随时间退化。

- **监控什么**：

    - **延迟**：推理需要多长时间？跟踪 p50（中位数）、p95 和 p99。p99 为 500ms 意味着 100 个用户中有 1 个等待半秒，这可能不可接受。

    - **吞吐量**：每秒多少请求？系统是否跟上需求？

    - **错误率**：多少比例的请求失败（异常、超时、无效输入）？

    - **模型指标**：保留集上的准确率、精确率、召回率。如果生产中可用标记数据（例如用户纠正），跟踪在线指标。

    - **数据漂移**：传入数据的分布是否发生变化？在白天照片上训练的模型可能在夜间照片上失败。统计测试（KS 检验、PSI）比较训练分布和实时分布。

    - **特征漂移**：单个特征分布是否发生变化？训练时正态分布但现在双峰的特征信号数据管道问题。

- **工具**：
    - **Prometheus** + **Grafana**：基础设施监控的标准。Prometheus 收集指标，Grafana 在带警报的仪表盘中可视化它们。
    - **Evidently AI**：开源 ML 监控。生成数据漂移、模型性能和数据质量的报告。

- **警报**：不要仅仪表盘 — 设置自动警报。"如果 p99 延迟超过 200ms 持续 5 分钟，发送 Slack 通知。" "如果数据漂移分数超过阈值，呼叫值班工程师。"

## 特征存储

- **特征存储**是预计算特征的集中存储库，在训练和服务之间共享。它解决两个问题：

    - **训练 - 服务偏差**：训练期间使用的特征必须与服务期间使用的特征相同。如果训练使用一种方式计算的 `user_age_at_signup` 而服务用不同方式计算它，模型的预测会静默出错。

    - **特征复用**：多个模型通常使用相同特征（用户人口统计、项目嵌入、聚合统计）。计算一次并共享避免重复和不一致。

- **Feast** 是最流行的开源特征存储。它管理在线特征（低延迟，从 Redis 或 DynamoDB 服务）和离线特征（批处理，存储在数据仓库中用于训练）。

- 特征存储对于推荐系统、欺诈检测以及任何特征从原始数据管道计算的应用至关重要。

## 管道编排

- 生产 ML 系统不只是模型。它是**管道**：数据摄入 → 预处理 → 特征计算 → 训练 → 评估 → 部署 → 监控。每个步骤依赖于前一个，可以独立失败，可能需要在不同计划上运行。

- **编排器**管理这些管道：

- **Apache Airflow**：数据管道编排的标准。DAG（有向无环图）定义任务依赖。每个任务独立运行，失败时可以重试，并通过 Web UI 监控。

```python
# airflow DAG 示例（简化）
from airflow import DAG
from airflow.operators.python import PythonOperator

dag = DAG("training_pipeline", schedule="@daily")

preprocess = PythonOperator(task_id="preprocess", python_callable=preprocess_data, dag=dag)
train = PythonOperator(task_id="train", python_callable=train_model, dag=dag)
evaluate = PythonOperator(task_id="evaluate", python_callable=evaluate_model, dag=dag)
deploy = PythonOperator(task_id="deploy", python_callable=deploy_model, dag=dag)

preprocess >> train >> evaluate >> deploy
```

- **Kubeflow 管道**：Kubernetes 上特定于 ML 的编排。每个步骤在容器中运行，GPU 资源按需分配，实验自动跟踪。

- **Prefect** 和 **Dagster**：Airflow 的现代替代方案，具有更好的开发人员体验、原生 Python API 和内置数据血统。

- **何时编排**：当管道有超过 2-3 个步骤、按计划运行、涉及多个团队或服务、或需要自动从故障恢复时。单脚本训练作业不需要编排器。每天从 5 个来源摄入数据、训练 3 个模型、评估它们并部署最佳模型的每日再训练管道绝对需要。
